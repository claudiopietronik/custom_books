Capitolo 16 · Progetto finale

Un progetto completo

Ci siamo. In quindici capitoli hai raccolto tutti i pezzi — HTML, CSS, JavaScript, DOM, eventi, dati — e ora li montiamo in un'unica applicazione vera, costruita da zero davanti ai tuoi occhi: una lista di cose da fare (una «to-do list») che aggiungi, spunti, cancelli, e che ricorda i tuoi dati anche dopo aver chiuso il browser. È il progetto che chiunque abbia imparato lo sviluppo web ha costruito almeno una volta, perché racchiude in piccolo tutto ciò che serve: struttura, stile, interazione e persistenza. Lo costruiamo a strati — prima l'HTML (le ossa), poi il CSS (l'aspetto), poi il JS (la vita), infine il salvataggio — e lo vedi funzionare dal vivo qui nella pagina. Alla fine avrai un'app completa, che capisci riga per riga, e le basi per costruire la prossima da solo.

🗺 Dove siamo

L'idea (§16.1), la struttura HTML (§16.2), lo stile CSS (§16.3), il comportamento JS (§16.4), il salvataggio con localStorage (§16.5), e dove andare da qui per continuare (§16.6). L'app completa è qui, giocabile, in §16.4.

🔑 Perché ci tocca: dal sapere al saper fare

Conoscere i mattoni non basta: il salto è combinarli in qualcosa che funziona. Questo progetto ti fa vedere come i pezzi si incastrano in un'app reale, e ti lascia uno scheletro riutilizzabile: quasi ogni piccola app (una lista della spesa, un blocco note, un tracker) ha questa stessa forma. Da qui non parti più da zero: parti da un modello che hai capito.

Sezione 16.1L'idea: una lista di cose da fare

Prima di scrivere codice, definiamo cosa deve fare l'app. Un buon progetto parte sempre da una lista chiara di funzionalità — piccola, ma completa.

📋 Le funzionalità (piccole ma complete)

La nostra to-do list deve permettere di:

  • Aggiungere una cosa da fare, scrivendola e premendo un pulsante (o Invio);
  • Spuntare una voce come completata (con un clic, che la barra);
  • Cancellare una voce che non serve più;
  • Ricordare tutto: se chiudi e riapri la pagina, le tue voci sono ancora lì.

Riconosci gli strumenti dietro ognuna? Aggiungere = createElement+appendChild (Cap. 13) su un evento click (Cap. 14); spuntare/cancellare = classList/remove su altri eventi; ricordare = localStorage (lo vediamo in §16.5). Nessuno strumento nuovo: solo combinazione.

Sezione 16.2La struttura HTML

Partiamo dalle ossa: il minimo HTML necessario. Un campo per scrivere, un pulsante per aggiungere, e una lista vuota che riempiremo con JavaScript.

index.html (il corpo)
<body>
    <h1>Le mie cose da fare</h1>

    <div class="aggiungi">
        <input type="text" id="campo" placeholder="Cosa devi fare?">
        <button id="aggiungi">Aggiungi</button>
    </div>

    <ul id="lista"></ul>   <!-- vuota: la riempie il JS -->

    <script src="app.js"></script>   <!-- in fondo! -->
</body>
💡 Gli id sono i «ganci» per JavaScript

Nota i tre id: campo, aggiungi, lista. Sono i punti a cui JavaScript «si aggancerà» col querySelector. La <ul> è volutamente vuota: non scriviamo le voci a mano nell'HTML, le genera il JS a partire dai dati. E lo <script> è in fondo, così quando parte, campo/pulsante/lista esistono già (ricordi l'errore «null» del Cap. 12?).

Sezione 16.3Lo stile CSS

Le ossa ci sono; diamogli un aspetto pulito. Poche regole, tutte tecniche che conosci dai Cap. 6–10: centratura, box model, Flexbox, una classe per le voci «fatte».

style.css (estratto)
body { font-family: system-ui, sans-serif; max-width: 460px;
       margin: 2rem auto; padding: 0 1rem; }

.aggiungi { display: flex; gap: .5rem; }        /* campo + bottone affiancati */
.aggiungi input { flex: 1; padding: .6rem; }         /* il campo si allarga */

li { display: flex; justify-content: space-between;
     padding: .6rem; border-bottom: 1px solid #eee; cursor: pointer; }

li.fatto { text-decoration: line-through; color: #999; }  /* voce completata */
🎨 Il trucco della classe .fatto

La riga chiave è li.fatto: definisce l'aspetto di una voce completata (barrata e grigia). Il CSS descrive com'è fatta una voce completata; sarà poi il JavaScript, con classList.toggle("fatto") al clic, a decidere quando una voce lo diventa. È di nuovo la divisione dei ruoli: CSS l'aspetto, JS il momento. Esattamente la lezione del Cap. 13.

Sezione 16.4Il comportamento JS

Ora la vita. Il JavaScript aggancia gli elementi, ascolta i click, e crea/spunta/rimuove le voci. Prova subito l'app completa qui sotto — funziona davvero, e ricorda i dati (§16.5) — poi leggi il codice che la anima.

La tua to-do list — funzionante e persistente clic sul testo = fatto · ✕ = elimina
    Niente da fare! Aggiungi qualcosa. ✨

    Ecco il cuore del codice — riconoscerai ogni pezzo dai capitoli precedenti:

    app.js (struttura essenziale)
    // 1) aggancia gli elementi (Cap. 12)
const campo  = document.querySelector("#campo");
const lista  = document.querySelector("#lista");

// 2) funzione che crea UNA voce nel DOM (Cap. 13)
function creaVoce(testo) {
    const li = document.createElement("li");
    li.textContent = testo;

    // clic sul testo → segna "fatto" (Cap. 13 + 14)
    li.addEventListener("click", () => {
        li.classList.toggle("fatto");
        salva();
    });

    // bottoncino ✕ → rimuove la voce
    const x = document.createElement("button");
    x.textContent = "✕";
    x.addEventListener("click", (e) => {
        e.stopPropagation();   // non far scattare anche il "fatto"
        li.remove();
        salva();
    });
    li.appendChild(x);
    lista.appendChild(li);   // attacca la voce alla pagina
}

// 3) aggiungere: al click sul bottone (Cap. 14)
document.querySelector("#aggiungi").addEventListener("click", () => {
    if (campo.value.trim()) {
        creaVoce(campo.value.trim());
        campo.value = "";   // svuota il campo
        salva();
    }
});
    🔍 Una novità utile: e.stopPropagation()

    L'unico pezzo nuovo. Il pulsante ✕ è dentro il <li>: senza precauzioni, cliccando la ✕ scatterebbe anche il click del li (che segna «fatto») — l'evento «risale» dal figlio al genitore. e.stopPropagation() ferma questa risalita, così la ✕ fa solo il suo lavoro (rimuovere). È un dettaglio che incontrerai spesso quando annidi elementi cliccabili.

    Sezione 16.5Salvare con localStorage

    Manca l'ultima magia: la memoria. Finora, ricaricando la pagina, la lista si svuoterebbe. Con localStorage — un piccolo archivio che il browser conserva sul dispositivo — le voci sopravvivono alla chiusura.

    salvare e ricaricare
    // SALVARE: raccogli le voci e mettile in localStorage (come testo JSON)
function salva() {
    const voci = [];
    for (const li of lista.children) {
        voci.push({
            testo: li.firstChild.textContent,
            fatto: li.classList.contains("fatto")
        });
    }
    localStorage.setItem("cose", JSON.stringify(voci));
}

// RICARICARE: all'avvio, leggi da localStorage e ricrea le voci
function carica() {
    const salvato = localStorage.getItem("cose");
    if (!salvato) return;
    for (const v of JSON.parse(salvato)) {
        creaVoce(v.testo, v.fatto);
    }
}
carica();   // esegui all'avvio della pagina
    localStorage setItem · getItem · JSON.stringify/parse
    localStorage.setItem("chiave", testo)
    salva un testo sotto un nome («chiave»). Resta sul dispositivo anche dopo aver chiuso il browser. Salva solo stringhe, per questo si usa JSON.
    localStorage.getItem("chiave")
    rilegge ciò che avevi salvato (o null se non c'è nulla).
    JSON.stringify(oggetto) / JSON.parse(testo)
    la coppia di traduttori: stringify trasforma un array/oggetto JS in testo JSON (per salvarlo); parse fa l'inverso, dal testo all'oggetto (per rileggerlo). Stesso JSON del Cap. 15, qui usato per la memoria locale invece che per la rete.
    È esattamente il meccanismo del segnalibro 🔖 di questi libri, e della funzione «riprendi la lettura» della home. Chiamando salva() dopo ogni modifica (aggiungi, spunta, rimuovi) e carica() all'avvio, l'app ha la memoria. La to-do list qui sopra lo fa davvero: prova ad aggiungere voci e ricaricare la pagina — sono ancora lì.

    Fermati un attimo a guardare cosa hai costruito: un'applicazione con struttura (HTML), stile (CSS), interazione (eventi), manipolazione dinamica del contenuto (DOM) e persistenza (localStorage). Sono gli stessi ingredienti — solo in scala maggiore — di qualsiasi applicazione web che userai o costruirai. Non hai imparato «dei trucchi»: hai imparato l'architettura del web lato browser.

    Sezione 16.6Dove andare da qui

    Hai le fondamenta solide. Ecco le direzioni naturali per crescere — non serve tutto subito, ma sapere dove guardare ti dà la mappa.

    La mappa dei prossimi passi dal front-end in poi
    Approfondire il front-end
    CSS più avanzato (Grid, animazioni, variabili CSS), JavaScript più a fondo (array methods come map/filter, moduli), e la accessibilità. Poi i framework (React, Vue): strumenti per costruire interfacce grandi senza ripeterti — ma si basano tutti su ciò che hai imparato qui. Nessun framework ha senso senza queste basi.
    Andare verso il back-end
    finora sei stato nel browser (il «front-end»). L'altra metà del web è il server: dove vivono i dati veri, i database, la logica, l'autenticazione. È ciò che sta dietro le API che hai imparato a chiamare col fetch. È il naturale capitolo successivo del tuo percorso — e in questa stessa collana c'è un libro dedicato al Backend Development (server, database, FastAPI, Docker…).
    Costruire, costruire, costruire
    il consiglio più importante: fai progetti tuoi. Una lista della spesa, un timer, una galleria di foto, un piccolo gioco. Si impara a programmare programmando, sbagliando e correggendo — proprio con la console (Cap. 11) come alleata.
    Hai attraversato i tre linguaggi del web e costruito un'app vera. Da qui, ogni sito che visiti non è più magia: è HTML che riconosci, CSS che sapresti scrivere, JavaScript che capisci. Il web ha smesso di essere una scatola nera. Buon codice. 🌐

    Esercizio 16.3

    Perché per salvare in localStorage usiamo JSON.stringify, e per rileggere JSON.parse?

    Esercizio 16.4

    (Progetto) Aggiungi all'app un contatore che mostra «N cose ancora da fare» (le voci non spuntate). Quali pezzi useresti? (Suggerimento: querySelectorAll e la classe fatto.)

    Sezione 16.8Soluzioni

    Soluzione 16.1

    Perché le voci non sono fisse: cambiano di continuo (l'utente le aggiunge e rimuove) e devono essere ricaricate da localStorage a ogni avvio. È JavaScript a generarle dinamicamente con creaVoce. Scriverle a mano nell'HTML sarebbe inutile (sparirebbero al primo salvataggio) e sbagliato: i dati stanno nell'archivio, non nel markup. L'HTML fornisce solo il contenitore vuoto.

    Soluzione 16.2

    La ✕ è dentro il <li>, che a sua volta ha un gestore di click (segna «fatto»). Gli eventi «risalgono» dal figlio al genitore, quindi un clic sulla ✕ farebbe scattare anche il click del li: la voce verrebbe rimossa e nel frattempo segnata «fatto» — comportamento confuso. e.stopPropagation() ferma la risalita, isolando l'azione della ✕ (solo rimozione).

    Soluzione 16.3

    Perché localStorage sa memorizzare solo testo (stringhe), mentre le nostre voci sono un array di oggetti. JSON.stringify converte quell'array in una stringa JSON (salvabile); JSON.parse fa il percorso inverso quando rileggiamo, ricostruendo l'array di oggetti su cui poter ciclare. Sono il ponte tra i dati «vivi» in memoria e la loro forma «testuale» salvabile.

    Soluzione 16.4
    JS
    function aggiornaContatore() {
    const tutte  = document.querySelectorAll("#lista li").length;
    const fatte  = document.querySelectorAll("#lista li.fatto").length;
    const restano = tutte - fatte;
    document.querySelector("#contatore").textContent = restano + " cose ancora da fare";
}

    Chiami aggiornaContatore() dopo ogni modifica (dentro salva(), per esempio). Usa querySelectorAll per contare tutte le voci e quelle .fatto, e la differenza sono quelle rimaste. È proprio ciò che fa la to-do list qui sopra sotto la lista.
