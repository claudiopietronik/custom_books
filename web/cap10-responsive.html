Capitolo 10

Responsive: ogni schermo

Oggi la maggior parte delle persone naviga dal telefono, non dal computer. Una pagina disegnata per un grande monitor, vista su uno schermo piccolo, diventa illeggibile: testo minuscolo, colonne strette, scroll orizzontale. Il responsive design è l'arte di fare una pagina che si adatta a qualsiasi dimensione — bella sul monitor e nel palmo della mano. Gli strumenti sono tre: le unità relative (%, rem, vw) che scalano invece di essere fisse, il meta viewport (una riga essenziale per il mobile), e le media query — la capacità di applicare stili diversi in base alla larghezza dello schermo, per esempio impilare in colonna ciò che sul desktop è affiancato. È ciò che separa un sito moderno da uno che «non si vede» sul cellulare.

🗺 Dove siamo

Perché il responsive (§10.1), le unità relative %, rem, vw (§10.2), il meta viewport (§10.3), e le media query (§10.4). Nel laboratorio (§10.5) costruisci una pagina che si adatta.

🔑 Perché ci tocca: il web è mobile

Ignorare il mobile oggi significa escludere più della metà dei visitatori. Un sito che non funziona sul telefono è, per molti, un sito rotto — e Google stesso penalizza i siti non responsive. Ma c'è di più: pensare «responsive» ti fa progettare interfacce flessibili e robuste, che funzionano in condizioni che non avevi previsto. È una competenza indispensabile per qualsiasi cosa tu costruisca per il web.

Sezione 10.1Perché responsive

Il problema è semplice: gli schermi hanno dimensioni diversissime, da un telefono stretto a un monitor ultrawide. Una pagina «rigida», pensata per una sola dimensione, sugli altri schermi fa una brutta figura.

📐 Un solo sito, mille schermi

L'obiettivo del responsive: un solo HTML e CSS che si adatta a tutti. Non fai un sito «per il desktop» e uno «per il mobile» (raddoppierebbe il lavoro): fai una pagina che si riorganizza. Su schermo largo, tre card affiancate; su schermo stretto, impilate in colonna. Testo che scala, immagini che non sforano, menu che si trasforma. La filosofia moderna è «mobile-first»: progetta prima per il piccolo (il caso più vincolato), poi «espandi» per gli schermi grandi. Ma il concetto chiave è uno: la pagina reagisce alle condizioni, invece di ignorarle. Vediamo gli strumenti che lo rendono possibile.

Sezione 10.2Unità relative (%, rem, vw)

Il primo passo verso il responsive è smettere di usare misure fisse (i pixel per tutto) e usare unità relative, che scalano in base al contesto.

Fisse vs relative px, %, rem, vw
px — pixel, fisso
una misura assoluta: width: 300px è sempre 300px, su ogni schermo. Utile per bordi o piccoli dettagli, ma rigido per le dimensioni principali: 300px sono metà di un telefono ma un decimo di un monitor.
% — percentuale del genitore
width: 50% = «metà del contenitore». Scala col genitore: su schermo grande è larga, su piccolo è stretta, ma sempre «metà». Ottima per larghezze fluide.
rem — relativa al font base
1rem = la dimensione del font radice (di solito 16px). font-size: 1.5rem, padding: 2rem. Il vantaggio: se l'utente ingrandisce il font (accessibilità), tutto scala insieme. È l'unità consigliata per font e spaziature.
vw / vh — % dello schermo
100vw = tutta la larghezza della finestra («viewport width»), 100vh = tutta l'altezza. Utile per elementi che devono riempire lo schermo.
La regola pratica: usa rem per font e spazi, % (o Flexbox) per le larghezze fluide, px solo per piccoli dettagli fissi (bordi). Aggiungi max-width per non far diventare il contenuto troppo largo sui monitor giganti (max-width: 800px; margin: 0 auto = larghezza fluida ma con un tetto, centrata — la ricetta di innumerevoli siti, e di questi libri). Le unità relative sono la base: un layout fatto di misure che scalano è già a metà strada verso il responsive.

Sezione 10.3Il meta viewport

Una singola riga nell'<head>, senza la quale nulla del responsive funziona sul mobile. È così importante — e così spesso dimenticata — da meritare la sua sezione.

Il meta viewport la riga essenziale per il mobile
nell'head, sempre
<meta name="viewport" content="width=device-width, initial-scale=1">
⚠ Senza questa riga, il mobile «bara»

Per compatibilità con vecchi siti, i telefoni fingono di avere uno schermo largo (es. 980px) e poi rimpiccioliscono tutto per farcelo stare: risultato, un sito minuscolo e illeggibile, da ingrandire con le dita. Il meta viewport dice: «usa la vera larghezza del dispositivo (width=device-width) e non zoomare (initial-scale=1)». Senza questa riga, il tuo CSS responsive non serve a nulla sul telefono: le media query non scattano correttamente. È letteralmente il prerequisito. Mettila in ogni pagina (l'hai già vista in cima a tutti questi libri).

Sezione 10.4Le media query

Lo strumento centrale del responsive: le media query permettono di applicare stili CSS solo a certe condizioni — tipicamente, quando lo schermo è più stretto (o più largo) di una certa soglia.

Una media query @media (condizione) { ... }
stile.css
/* stile normale (per schermi larghi): tre card affiancate */
.riga { display: flex; gap: 1rem; }

/* SOLO quando lo schermo è largo al massimo 600px (telefoni): */
@media (max-width: 600px) {
    .riga { flex-direction: column; }   /* impila in colonna! */
}
@media (max-width: 600px) { … }
un «blocco condizionale»: le regole dentro si applicano solo se la condizione è vera — qui, se lo schermo è largo al massimo 600px (un telefono). Su schermo largo, la regola dentro è ignorata; su schermo stretto, entra in gioco e sovrascrive.
l'esempio classico: da riga a colonna
lo schema più comune: sul desktop gli elementi sono affiancati (Flexbox row); sotto una certa larghezza, una media query li fa impilare in colonna (flex-direction: column). Così le tre card affiancate diventano tre card una sotto l'altra sul telefono — leggibili.
i «breakpoint»
le soglie (600px, 768px, 1024px…) si chiamano breakpoint: i punti in cui il layout «cambia». Ne scegli pochi, dove il design «si rompe», e riorganizzi lì.
Prova il widget qui sotto: trascina lo slider per «restringere lo schermo» e guarda il layout passare da riga a colonna quando scende sotto la soglia — è esattamente ciò che fa una media query quando ruoti il telefono o cambi dispositivo.

Sezione 10.5Laboratorio: una pagina adattiva

Costruiamo una pagina con tre card che si adatta: affiancate sul desktop, impilate sul telefono. È il pattern responsive più comune, e ora sai tutti i pezzi.

Il file adattiva.html viewport + unità relative + media query
adattiva.html (estratto)
<head>
    <meta charset="utf-8">
    <meta name="viewport" content="width=device-width, initial-scale=1">  <!-- essenziale! -->
    <style>
        * { box-sizing: border-box; }
        body { font-family: system-ui, sans-serif; max-width: 900px; margin: 0 auto; padding: 1rem; }
        .riga { display: flex; gap: 1rem; }
        .card { flex: 1; padding: 1.5rem; background: #f4f2ff; border-radius: 12px; }

        @media (max-width: 600px) {   /* sui telefoni: */
            .riga { flex-direction: column; }   /* card impilate */
        }
    </style>
</head>
<body>
    <h1>Pagina adattiva</h1>
    <div class="riga">
        <div class="card">Uno</div>
        <div class="card">Due</div>
        <div class="card">Tre</div>
    </div>
</body>
Apri e ridimensiona la finestra del browser (o usa F12 → l'icona «dispositivo mobile» per simulare un telefono): sopra i 600px le tre card sono affiancate, sotto si impilano in colonna. Una sola pagina, due layout, gestiti da una media query. Nota tutti i pezzi al lavoro: il meta viewport (senza cui sul telefono non funzionerebbe), max-width + margin: 0 auto (fluido ma con tetto, centrato), Flexbox coi flex: 1 (colonne uguali), e la media query che riorganizza. Questo è, in miniatura, il responsive design. Sperimenta: cambia il breakpoint, aggiungi una media query che sotto una certa larghezza rimpicciolisce il titolo. Restringi e allarga: la pagina «respira».
🔑 Quello che devi portarti dietro dal Capitolo 10
  1. Il responsive: un solo sito che si adatta a ogni schermo (spesso «mobile-first») (§10.1).
  2. Unità relative: % (del genitore), rem (font/spazi, scala con l'utente), vw/vh (dello schermo); px solo per dettagli. max-width + margin:0 auto = fluido con tetto (§10.2).
  3. Il meta viewport (width=device-width, initial-scale=1) è essenziale: senza, il responsive non funziona sul mobile (§10.3).
  4. Le media query @media (max-width: …) { } applicano stili a certe larghezze; schema classico: da riga a colonna sui telefoni (§10.4).

Con questo si chiude il CSS: sai stilizzare, selezionare, disporre col box model e Flexbox, e adattare a ogni schermo. Le tue pagine sono ora strutturate e belle e responsive — ma ancora statiche: reagiscono al mouse (:hover) e alla dimensione, ma non «fanno» nulla. Per dare vita vera alle pagine — reagire ai click, aggiornare i contenuti, calcolare, comunicare — serve il terzo linguaggio. Nel Capitolo 11 entriamo in JavaScript: le sue basi (variabili, funzioni, condizioni, cicli) e la console del browser. Se hai fatto il libro di Python, molte idee ti saranno familiari — qui imparerai il «dialetto» del web, quello che rende le pagine interattive.

Sezione 10.7Soluzioni

Soluzione 10.1

Perché px è fisso: la stessa misura appare enorme su uno schermo piccolo e minuscola su uno grande. % e rem scalano col contesto (il genitore o il font base), adattandosi alle diverse dimensioni. In più, rem rispetta le impostazioni di font dell'utente (accessibilità).

Soluzione 10.2

Il meta viewport dice al telefono di usare la vera larghezza del dispositivo e di non zoomare (width=device-width, initial-scale=1). Senza, il telefono finge uno schermo largo e rimpicciolisce tutto: la pagina appare minuscola e le media query non scattano come dovrebbero — il responsive non funziona.

Soluzione 10.3
CSS
@media (max-width: 500px) {
    body { font-size: 14px; }
}
Soluzione 10.4
CSS
@media (max-width: 600px) {
    .navbar { flex-direction: column; gap: .5rem; }
}

Sotto i 600px la navbar passa in colonna: il logo sopra, il menu sotto — una soluzione tipica per il mobile.
