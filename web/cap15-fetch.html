Capitolo 15

Fetch: dati dal web

Le tue pagine ora reagiscono all'utente, ma vivono chiuse in se stesse: tutto ciò che mostrano è scritto dentro di loro. Le vere applicazioni, invece, chiedono dati al mondo: il meteo di oggi, l'elenco dei tuoi messaggi, i prezzi aggiornati, i risultati di una ricerca. Questi dati vivono su un server, e JavaScript li va a prendere — mentre la pagina è aperta, senza ricaricarla — con una funzione chiamata fetch. C'è però una novità concettuale: chiedere dati a un server richiede tempo (viaggiano per la rete), e JavaScript non si ferma ad aspettare. Devi imparare a lavorare con operazioni che finiranno più tardi: le promise, e la sintassi async/await che le rende leggibili. È l'ultimo tassello: con fetch, le tue pagine si collegano a tutto l'ecosistema del web.

🗺 Dove siamo

Parlare con un server da JS e cos'è un'API (§15.1), fetch e le promise (§15.2), la sintassi async/await (§15.3), mostrare i dati nel DOM (§15.4). Nel laboratorio (§15.5) «chiami» un'API e mostri i risultati.

🔑 Perché ci tocca: qui la pagina si apre al mondo

Ogni app che conosci — social, meteo, mappe, e-commerce, chat — è per lo più una pagina che chiede dati a dei server e li mostra bene. Imparare fetch è imparare a costruire applicazioni «vere», non solo pagine isolate. E le promise / async-await che incontri qui non servono solo per la rete: sono il modo in cui JavaScript gestisce qualsiasi operazione che richiede tempo. È un concetto che ti seguirà per tutta la programmazione web, front-end e (col libro di Backend) anche lato server.

Sezione 15.1Parlare con un server da JS

Nel libro «Reti» hai visto che il browser chiede pagine a un server via HTTP. Qui succede lo stesso, ma è JavaScript a fare la richiesta, in background, per ottenere dati (non un'intera pagina) e usarli subito. Il punto di contatto con il server si chiama API.

📖 API e JSON: chiedere dati, non pagine

Un'API (Application Programming Interface) è un indirizzo, esposto da un server, pensato per essere chiamato da un programma invece che da un umano. Tu chiami un URL tipo https://api.esempio.com/meteo?citta=Roma e il server, invece di una pagina HTML, ti risponde con dati puri, di solito in formato JSON.

Il JSON è un modo testuale di rappresentare dati, praticamente identico agli oggetti e agli array di JavaScript: { "citta": "Roma", "gradi": 28, "cielo": "sereno" }. È la lingua franca degli scambi di dati sul web: leggibile da umani, facilissimo da usare in JS. Quando fetch ti riporta la risposta, la converti da JSON a un normale oggetto JavaScript e la usi come già sai.

Sezione 15.2fetch e le promise

fetch(url) lancia una richiesta a un server. Ma la risposta non arriva all'istante: viaggia per la rete. Quindi fetch non ti restituisce i dati — ti restituisce una promise: una «promessa» che i dati arriveranno (o che qualcosa andrà storto).

⏳ Cos'è una promise

Una promise è un oggetto che rappresenta un risultato non ancora pronto. È come lo scontrino del guardaroba: non è il cappotto, ma è la garanzia che potrai ritirarlo. JavaScript non si blocca ad aspettare: va avanti col resto, e quando la risposta arriva, esegue il codice che hai «prenotato». Una promise finisce in due modi: mantenuta (i dati sono arrivati) o respinta (c'è stato un errore, es. niente rete). Questo modo di lavorare — «parti ora, finisci dopo» — si chiama asincrono.

fetch con le promise (stile .then)
fetch("https://api.esempio.com/dati")
    .then(risposta => risposta.json())   // converti la risposta da JSON
    .then(dati => {
        console.log(dati);              // qui i dati sono pronti!
    })
    .catch(errore => {
        console.log("Qualcosa è andato storto:", errore);
    });
Il vocabolario delle promise .then · .json() · .catch
.then(funzione)
«quando la promise è pronta, esegui questa funzione col risultato». È l'equivalente asincrono di «poi fai questo». Puoi concatenarne più d'uno.
.json()
la prima cosa che fai con la risposta: convertire il testo JSON in un oggetto JavaScript usabile. Anche questa restituisce una promise (la lettura richiede un attimo), per questo c'è un secondo .then.
.catch(funzione)
«se qualcosa va storto (rete assente, server in errore), esegui questo». La gestione degli errori: fondamentale, perché la rete può sempre fallire.
Questo stile «.then concatenati» funziona, ma con più passaggi diventa difficile da leggere. Per questo JavaScript offre una sintassi più pulita per la stessa cosa: async/await. È quella che userai d'ora in poi — vediamola.

Sezione 15.3async / await

async/await è «zucchero sintattico»: fa esattamente ciò che fanno le promise, ma ti fa scrivere codice asincrono come se fosse lineare, dall'alto in basso — molto più facile da leggere.

lo stesso fetch, con async/await
async function caricaDati() {
    try {
        const risposta = await fetch("https://api.esempio.com/dati");
        const dati = await risposta.json();
        console.log(dati);          // i dati, pronti, su una riga pulita
    } catch (errore) {
        console.log("Errore:", errore);
    }
}
caricaDati();
async / await / try-catch codice asincrono che si legge dall'alto in basso
async function …
marchia una funzione come asincrona: dentro di essa potrai usare await. Praticamente, ogni funzione che fa fetch sarà async.
await
«aspetta che questa promise sia pronta, poi dammi il risultato». Fa sembrare l'asincrono sincrono: la riga dopo un await parte solo quando il dato è arrivato — ma senza bloccare il resto della pagina. Niente più .then annidati: scrivi const dati = await … e prosegui.
try { … } catch (errore) { … }
il modo di gestire gli errori in stile async/await (rimpiazza .catch): prova il blocco try; se qualcosa fallisce, salta al catch. Avvolgi sempre i tuoi fetch in un try/catch: la rete è inaffidabile per natura.
Confronta i due esempi (§15.2 e §15.3): fanno la stessa identica cosa. Ma la versione async/await si legge come una lista di passi ordinati, ed è quella che vedrai in quasi tutto il codice moderno. Le promise restano «sotto il cofano»; await è solo il modo comodo di aspettarle.

Sezione 15.4Mostrare i dati nella pagina

Ottenere i dati è metà dell'opera; l'altra metà è mostrarli — e qui torna tutto il Cap. 13. Ricevi un array di oggetti JSON, lo scorri, e per ciascuno crei un elemento nel DOM.

dai dati alla pagina
async function mostraUtenti() {
    const risposta = await fetch("https://api.esempio.com/utenti");
    const utenti = await risposta.json();   // es. [{nome:"Anna"}, {nome:"Luca"}]

    const lista = document.querySelector("#lista");
    for (const u of utenti) {        // scorri i dati (Cap. 13)
        const li = document.createElement("li");
        li.textContent = u.nome;      // usa i campi dell'oggetto
        lista.appendChild(li);        // attaccalo alla pagina
    }
}
🔗 Tutto il libro converge qui

Guarda quanti pezzi si uniscono in questa funzioncina: fetch + await (questo capitolo) per prendere i dati, for…of (Cap. 13) per scorrerli, createElement + appendChild (Cap. 13) per costruire la pagina, l'accesso ai campi dell'oggetto (u.nome, Cap. 11). Chiedere dati e trasformarli in interfaccia: è, in sostanza, ciò che fa il 90% delle applicazioni web. Il resto sono dettagli e raffinatezze.

🔑 Quello che devi portarti dietro dal Capitolo 15
  1. Un'API è un indirizzo che restituisce dati (non pagine), di solito in JSON — quasi identico agli oggetti JS (§15.1).
  2. fetch(url) avvia una richiesta e restituisce una promise: un risultato che arriverà più tardi (lavoro asincrono) (§15.2).
  3. async/await: const r = await fetch(…), const dati = await r.json() — codice asincrono che si legge dall'alto in basso. Avvolgilo in try/catch (§15.3).
  4. Mostrare: scorri i dati con for…of e costruisci il DOM con createElement/appendChild (§15.4).
  5. Mostra sempre uno stato di caricamento e gestisci gli errori: la rete può fallire (§15.5).

E con questo hai in mano tutti e tre i linguaggi e le loro tecniche fondamentali: struttura (HTML), aspetto (CSS), comportamento e dati (JavaScript, DOM, eventi, fetch). Sai fare una pagina, renderla bella e responsive, farla reagire all'utente e collegarla a dei dati. È il momento di metterlo tutto insieme in qualcosa di completo e tuo. Nel Capitolo 16, il progetto finale: costruiamo da zero una piccola applicazione web vera — una lista di cose da fare che ricorda i dati anche quando chiudi la pagina. Tutto ciò che hai imparato, in un unico progetto.

Esercizio 15.4

Perché è importante avvolgere un fetch in un try/catch (o usare .catch)?

Sezione 15.7Soluzioni

Soluzione 15.1

Un'API è un indirizzo (URL) esposto da un server e pensato per essere chiamato da un programma, che restituisce dati anziché una pagina. Il formato usuale è JSON. È comodo per JavaScript perché la sua sintassi è quasi identica a quella di oggetti e array JS: dopo .json() i dati diventano un normale oggetto/array che usi con .campo e for…of, senza conversioni faticose.

Soluzione 15.2

Perché i dati devono viaggiare per la rete, e questo richiede tempo. JavaScript non si ferma ad aspettare (bloccherebbe tutta la pagina): fetch ritorna subito una promise, cioè la «promessa» che il risultato arriverà. «Asincrono» significa proprio questo: l'operazione parte ora ma finisce più tardi, e nel frattempo il resto del programma continua. Col risultato reagisci quando è pronto (via .then o await).

Soluzione 15.3
JS
async function prendi() {
    try {
        const r = await fetch("https://api.x.com/cose");
        const dati = await r.json();
        console.log(dati);
    } catch (err) {
        console.log("Errore:", err);
    }
}
Soluzione 15.4

Perché la rete può sempre fallire: connessione assente, server giù, risposta malformata. Senza gestione, l'errore «rompe» il codice e l'utente resta magari su un eterno «caricamento…» senza spiegazioni. Con try/catch (o .catch) intercetti il problema e puoi reagire con eleganza — per esempio mostrando un messaggio «Impossibile caricare i dati, riprova». È la differenza tra un'app robusta e una fragile.
