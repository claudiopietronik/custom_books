Un progetto completo

Hai imparato tutti i mattoni — variabili, collezioni, decisioni, cicli, funzioni, file, classi, gestione errori, stile. È il momento di costruire qualcosa di vero con le tue mani: un piccolo programma completo, dall'idea al codice funzionante. 🔑 Perché ci tocca: il metodo, non solo il codice

Questo capitolo non insegna nuove funzioni: insegna il metodo con cui si affronta un progetto reale — scomporre, pianificare, costruire un pezzo alla volta, provare, migliorare. È la competenza che trasforma un principiante in un programmatore: non «conoscere la sintassi», ma saper portare un'idea dal nulla a un programma funzionante. È lo stesso metodo che userai per qualsiasi cosa costruirai — nei prossimi libri e oltre.

Sezione 16.1Dall'idea al piano

Ogni progetto parte da un'idea e da qualche domanda. Prima di scrivere una riga, chiarisci cosa deve fare: è metà del lavoro.

📐 L'idea: un gestore di attività

Vogliamo un programma che permetta di: vedere le attività, aggiungerne, segnarne come fatte, rimuoverle, e — cruciale — ricordarle tra un avvio e l'altro. Domande da porsi subito: come rappresento un'attività? (un dizionario con testo e stato «fatto»). Dove salvo i dati? (un file JSON, Cap. 12). Come interagisce l'utente? (un menu che si ripete, Cap. 8). Rispondere a queste domande prima di programmare evita di riscrivere tutto a metà strada. Un piano anche approssimativo vale più di mille righe scritte a caso.

Sezione 16.2Strutturare il progetto

Con il piano in mano, decidiamo la struttura: quali funzioni servono, come sono organizzate. Pensare per funzioni piccole, ognuna con un compito, rende tutto più semplice.

📐 Scomporre in funzioni

Il segreto è scomporre il problema grande in pezzi piccoli, uno per compito:

  • carica() — legge le attività dal file JSON (o lista vuota se non c'è);
  • salva(attivita) — scrive le attività nel file;
  • mostra(attivita) — stampa l'elenco numerato;
  • aggiungi(attivita, testo) — aggiunge una nuova attività;
  • completa(attivita, n) — segna la n-esima come fatta;
  • un ciclo menu che chiama queste funzioni in base alla scelta dell'utente.

Ogni funzione fa una cosa sola e bene (Cap. 9). Così puoi scriverle e provarle una alla volta, e se qualcosa non va sai subito dove guardare. È il «dividi e conquista» in pratica: un programma grande è solo tante funzioncine che collaborano.

Sezione 16.3Scrivere il codice, pezzo per pezzo

Ora costruiamo, una funzione alla volta. Ecco il programma completo — leggilo notando come ogni concetto del libro trova il suo posto.

Il progetto attivita.py tutto il libro, in un file
attivita.py
import json
from pathlib import Path

FILE = Path("attivita.json")

def carica():
    """Legge le attività dal file, o lista vuota se non esiste."""
    if FILE.exists():
        return json.loads(FILE.read_text())
    return []

def salva(attivita):
    FILE.write_text(json.dumps(attivita, indent=2))

def mostra(attivita):
    if not attivita:                     # lista vuota è "falsy" (Cap. 7)
        print("  (nessuna attività)")
    for i, a in enumerate(attivita, 1):   # enumerate (Cap. 14)
        segno = "✓" if a["fatto"] else " "
        print(f"  {i}. [{segno}] {a['testo']}")

def menu():
    attivita = carica()               # carica all'avvio
    while True:                         # ciclo menu (Cap. 8)
        print("\n--- LE MIE ATTIVITÀ ---")
        mostra(attivita)
        scelta = input("[a]ggiungi [f]atto [q]esci: ").strip().lower()

        if scelta == "a":
            testo = input("Cosa? ").strip()
            if testo:
                attivita.append({"testo": testo, "fatto": False})
                salva(attivita)          # salva dopo ogni modifica
        elif scelta == "f":
            try:                          # gestione errori (Cap. 11)
                n = int(input("Numero: ")) - 1
                attivita[n]["fatto"] = True
                salva(attivita)
            except (ValueError, IndexError):
                print("  Numero non valido.")
        elif scelta == "q":
            print("Ciao!")
            break

menu()   # avvia il programma
Fai il conto di quanti concetti sono al lavoro: funzioni (Cap. 9), liste di dizionari (Cap. 5-6), file JSON (Cap. 12), ciclo while e break (Cap. 8), if/elif (Cap. 7), try/except (Cap. 11), enumerate e valori falsy (Cap. 14, 7), f-string (Cap. 4). Nessuno di questi pezzi, da solo, è difficile — ma insieme formano un'applicazione vera. È così che si costruisce il software: componendo tante idee semplici.

Sezione 16.4Provare e correggere

Il codice scritto non è codice finito: va provato. E provare non significa solo «vedere se funziona coi dati giusti», ma soprattutto cosa succede quando le cose vanno storte.

📐 I casi da provare (specie quelli «cattivi»)
  • Il caso normale: aggiungi un'attività, segnala fatta, esci e riapri — c'è ancora? (la persistenza funziona).
  • I casi limite: la lista vuota all'inizio; aggiungere testo vuoto; segnare fatto un numero inesistente (5 quando ci sono 3 attività) o non-numerico («abc»).
  • Sono proprio i casi «cattivi» a rivelare i bug: il nostro try/except gestisce numeri sbagliati, e l'if testo: evita attività vuote. Se non li avessimo previsti, il programma sarebbe crollato al primo errore dell'utente.

Quando trovi un bug, torna amico del traceback (Cap. 11): ti dice la riga esatta. Provare pensando «come potrebbe rompersi?» è ciò che distingue il codice fragile da quello robusto.

Sezione 16.5Migliorare e rifinire

Un programma che funziona è l'inizio, non la fine. Un occhio critico trova sempre modi per renderlo più utile, più chiaro, più robusto. È la fase — piacevolissima — del miglioramento.

💡 Idee per far crescere il progetto
  • Funzionalità: aggiungere la rimozione di un'attività; una data di scadenza (con datetime, Cap. 15); priorità; il conteggio di quante sono fatte / da fare.
  • Qualità del codice: trasformare l'attività da dizionario a una classe Attivita (Cap. 13), con metodi completa() e __str__(); spostare le funzioni in un modulo separato (Cap. 10).
  • Robustezza: gestire il caso di un file JSON corrotto; confermare prima di uscire.

Il codice migliore nasce dalla riscrittura: la prima versione fa funzionare le cose, le successive le fanno funzionare bene. Non aver paura di rimettere mano a ciò che hai scritto — è lì che si impara di più, e dove un progetto diventa davvero tuo.

🧭 Il cammino continua
  • Costruisci. La cosa più importante: fai progetti tuoi. Automatizza una noia, organizza dei file, crea un giochino, analizza dei dati. Programmare si impara programmando: le idee di questo libro diventano tue solo usandole.

  • Un altro linguaggio. Se vuoi capire il computer più a fondo, C++ (in arrivo) ti mostra il basso livello — memoria, puntatori, prestazioni. Ma ora hai le fondamenta: cambiare linguaggio è questione di dialetto.
🔑 Quello che devi portarti dietro da tutto il libro

  2. Sai controllare il flusso: decisioni (if/elif/else) e cicli (for/while), il cuore della logica.
  3. Sai organizzare: funzioni, moduli, classi; e gestire gli imprevisti (try/except) e i dati persistenti (file, JSON).
  4. Sai scrivere Python con stile (comprehension, idiomi pythonici) e riusare l'ecosistema (libreria standard, pip). E soprattutto: sai portare un'idea dal nulla a un programma funzionante. Il resto è pratica.

Sezione 16.8Soluzioni

Soluzione 16.1

Soluzione 16.4
attivita_classe.py
class Attivita:
    def __init__(self, testo, fatto=False):
        self.testo = testo
        self.fatto = fatto
    def completa(self):
        self.fatto = True
    def __str__(self):
        return f"[{'✓' if self.fatto else '☐'}] {self.testo}"
    def to_dict(self):
        return {"testo": self.testo, "fatto": self.fatto}   # per il JSON

Con la classe, print(attivita) mostra direttamente «[✓] Comprare pane», e per salvare converti ogni oggetto con .to_dict(). È il progetto «cresciuto» in stile OOP.
