Capitolo 1

Cos'è il controllo di versione

Hai presente le cartelle piene di file chiamati tesi_definitiva.doc, tesi_definitiva_v2.doc, tesi_definitiva_v2_QUESTA.doc? È il modo in cui, senza gli strumenti giusti, cerchiamo di «tenere traccia» delle versioni — ed è un disastro: caotico, pieno di copie, e quando qualcosa si rompe non sai più tornare indietro. Per il codice (e non solo) esiste una soluzione infinitamente migliore: il controllo di versione, e il suo strumento dominante si chiama Git. Git registra ogni modifica come un'istantanea nel tempo, ti lascia tornare a qualsiasi versione passata, lavorare su più idee in parallelo senza confusione, e — insieme a GitHub — collaborare con altri in tutto il mondo. In questo primo capitolo capiamo il problema che risolve (§1.1), cosa fa un sistema di versioning (§1.2), la differenza tra Git e GitHub (§1.3) e come Git ragiona (§1.4). È la mappa mentale su cui costruiremo, comando dopo comando, una competenza che userai per tutta la tua carriera.

🗺 Dove siamo

Il problema dei cambiamenti (§1.1), cosa fa il versioning (§1.2), Git ≠ GitHub (§1.3), come lavora Git (§1.4). Nel laboratorio (§1.5) fissi la mappa mentale prima di installare Git nel prossimo capitolo.

🔑 Perché ci tocca: è la competenza che tutti danno per scontata

Git è ovunque: ogni azienda, ogni progetto open source, ogni team di sviluppo lo usa. Saperlo usare non è «un plus»: è un prerequisito, spesso richiesto ancor prima di saper programmare bene. Eppure viene raramente insegnato con calma. Impararlo davvero — capendo cosa succede, non solo memorizzando comandi — ti dà sicurezza, ti salva da disastri, e ti apre la porta della collaborazione. Vale ogni minuto.

Sezione 1.1Il problema: tenere traccia dei cambiamenti

Ogni progetto — un programma, un sito, un documento — cambia nel tempo. Aggiungi cose, correggi errori, provi idee, a volte torni sui tuoi passi. Senza uno strumento apposito, gestire questi cambiamenti diventa presto ingestibile.

⚠ Il metodo «copie con nomi diversi» non funziona

Provare a versionare a mano — salvando copie come progetto_v1, progetto_v2, progetto_FINALE — porta a una lista di problemi concreti: (1) non ricordi mai cosa è cambiato tra una copia e l'altra; (2) le copie occupano spazio e generano confusione; (3) se lavori in due sullo stesso file, come fondete le vostre modifiche? A mano, copincollando? (4) se una modifica rompe tutto, ritrovare l'ultima versione «buona» è un incubo. Manca uno strumento che registri la storia in modo ordinato, sicuro e navigabile. Quello strumento è il controllo di versione.

Sezione 1.2Cosa fa un sistema di versioning


  

    Un sistema di controllo di versione (in inglese version control system, VCS) è uno strumento che
    registra ogni cambiamento ai tuoi file nel tempo, così puoi rivederli, confrontarli e tornare a qualsiasi
    versione precedente. Git è il VCS più usato al mondo.
  


  

    
Cosa ti dà un VCS le superpotenze del versioning

    

      

        
una storia completa

        
ogni modifica è registrata come un'istantanea (snapshot) con data, autore e una descrizione.
            Puoi rivedere tutta l'evoluzione del progetto, riga per riga, per sempre.

        
tornare indietro

        
hai rotto qualcosa? Torni a una versione precedente che funzionava, in un istante. Sperimentare non
            fa più paura: c'è sempre una via di ritorno.

        
lavorare in parallelo

        
con i branch (Cap. 7) provi un'idea «di lato», senza toccare la versione principale, e la unisci
            solo quando è pronta.

        
collaborare senza pestarsi i piedi

        
più persone lavorano sugli stessi file; il VCS fonde le modifiche e segnala i (rari) conflitti da
            risolvere (Cap. 9). Fine del copincolla manuale.

        
chi ha fatto cosa

        
ogni cambiamento è attribuito a un autore. Utile per capire, dare credito, e — quando serve — trovare
            quando è entrato un bug.

      

    

    

      Il concetto chiave è lo snapshot (istantanea): ogni volta che «salvi» in Git (una commit, Cap. 4),
      registri una fotografia completa dello stato del progetto in quel momento, con un'etichetta. La storia del
      progetto diventa una sequenza di snapshot navigabili — proprio la «macchina del tempo» del widget qui
      sopra. Tutto il resto del libro è imparare a creare, leggere e gestire questi snapshot.
    

  



  
  
Sezione 1.3Git e GitHub: non sono la stessa cosa


  

    Due nomi che si assomigliano e che i principianti confondono di continuo — ma sono cose diverse e
    complementari. Chiarirlo subito ti eviterà tanta confusione.
  


  

    
Git vs GitHub lo strumento vs il servizio

    

      

        
Git — lo strumento

        
un programma che gira sul tuo computer e fa il controllo di versione: registra le commit, gestisce
            i branch, tiene la storia. Funziona anche completamente offline, da solo. È gratuito e open
            source, creato nel 2005.

        
GitHub — il servizio

        
un sito web (un'azienda) che ospita i repository Git online, per condividerli e collaborare.
            Aggiunge alla base di Git funzioni sociali: pull request, code review, issue, pagine di
            progetto (Cap. 10-12). Non è l'unico: esistono anche GitLab, Bitbucket…

        
l'analogia

        
Git sta a GitHub come la fotografia sta a un social di foto: Git «scatta» e organizza le
            versioni (lo fai sul tuo PC); GitHub è il luogo online dove le carichi per condividerle e lavorarci in
            gruppo. Puoi usare Git senza GitHub; GitHub ha bisogno di Git.

      

    

    

      In questo libro impariamo prima Git (Cap. 2-9), lavorando sul nostro computer: le fondamenta. Poi
      GitHub (Cap. 10-14), per portare i progetti online e collaborare. Questo ordine non è casuale:
      capire Git da solo rende GitHub facilissimo; partire da GitHub senza capire Git genera solo confusione (è
      l'errore più comune di chi «impara Git cliccando su GitHub»).
    

  



  
  
Sezione 1.4Come lavora Git (panoramica)


  

    Prima di installarlo, fissiamo il modello mentale di come Git ragiona. Non serve capire tutto ora — solo
    familiarizzare con tre parole che torneranno di continuo: repository, commit, branch.
  


  

    
I tre concetti fondamentali repository · commit · branch

    

      

        
il repository (o «repo»)

        
la cartella del tuo progetto più tutta la sua storia, gestita da Git. È il contenitore di
            tutto: i file attuali e ogni loro versione passata (Cap. 3).

        
la commit

        
uno snapshot salvato: una fotografia del progetto in un dato momento, con un messaggio che descrive
            cosa hai cambiato. La storia è una sequenza di commit (Cap. 4). È l'unità base di Git.

        
il branch (ramo)

        
una linea di sviluppo indipendente. Il principale si chiama di solito main. Puoi
            creare altri branch per lavorare su idee separate e poi unirle (Cap. 7-8). È ciò che rende Git
            potente per sperimentare e collaborare.

      

    

    

      Il flusso base che imparerai è semplice e ripetitivo: modifichi dei file, ne fai una commit (uno
      snapshot con descrizione), e ripeti. Ogni tanto crei un branch per un'idea, ci lavori, e lo
      unisci. Con GitHub, spingi (push) le commit online per condividerle. Tutto qui: il resto
      sono dettagli e comodità attorno a questo ciclo. Nei prossimi capitoli lo costruiamo, un pezzo alla
      volta, partendo dall'installazione.

Sezione 1.6Esercizi


  

    
Esercizio 1.1

    

      Elenca almeno tre problemi del «versionare a mano» salvando copie con nomi diversi
      (progetto_v1, progetto_FINALE…).
    

  


  

    
Esercizio 1.2

    

      Con parole tue, cos'è uno snapshot (una commit) in Git? Cosa contiene oltre ai file?
    

  


  

    
Esercizio 1.3

    

      Qual è la differenza tra Git e GitHub? È possibile usare Git senza GitHub? E GitHub senza Git?
    

  


  

    
Esercizio 1.4

    

      Abbina ogni termine alla sua definizione: (a) repository, (b) commit, (c) branch —
      (1) uno snapshot salvato con un messaggio, (2) il progetto più la sua storia, (3) una linea
      di sviluppo parallela.
    

  



  
  
Sezione 1.7Soluzioni


  

    
Soluzione 1.1

    

      Tra i problemi: (1) non sai cosa è cambiato tra una copia e l'altra (nessuna descrizione);
      (2) le copie occupano spazio e creano confusione (quale è davvero l'ultima?); (3) è
      quasi impossibile fondere le modifiche di due persone; (4) tornare a una versione funzionante dopo
      un errore è difficile e rischioso. In sintesi: manca una storia ordinata, sicura e navigabile.
    

  


  

    
Soluzione 1.2

    

      Uno snapshot (commit) è una fotografia completa dello stato del progetto in un dato momento,
      salvata da Git. Oltre ai file (nella loro versione di quel momento), contiene dei metadati: l'autore,
      la data, un messaggio che descrive cosa è cambiato, e un identificatore univoco. La storia del
      progetto è la sequenza di questi snapshot.
    

  


  

    
Soluzione 1.3

    

      Git è lo strumento di controllo di versione che gira sul tuo computer (anche offline); GitHub
      è un servizio online che ospita i repository Git per condividerli e collaborare. Sì, puoi usare Git
      senza GitHub (versioni i tuoi progetti in locale, da solo). GitHub senza Git invece no: GitHub serve
      proprio a ospitare repository Git, quindi si appoggia a Git.
    

  


  

    
Soluzione 1.4

    

      (a) repository → (2) il progetto più la sua storia; (b) commit → (1) uno snapshot salvato con un
      messaggio; (c) branch → (3) una linea di sviluppo parallela. Sono i tre concetti fondamentali su cui si
      regge tutto Git.
