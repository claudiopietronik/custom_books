Capitolo 3

Il primo repository

È il momento di trasformare una cartella qualsiasi in un repository Git — il contenitore dove Git registra il tuo progetto e tutta la sua storia. Basta un comando, git init, e da quel momento Git «osserva» quella cartella. Ma il vero cuore di questo capitolo è un concetto che, una volta capito, rende tutto il resto di Git limpido: le tre aree. In Git, un file attraversa tre «zone» — la working directory (dove lavori), la staging area (dove prepari cosa salvare) e il repository (dove lo snapshot viene registrato per sempre). Capire questo viaggio spiega perché servono due comandi (git add e git commit) per salvare, e cosa significano i messaggi che Git ti mostra. In questo capitolo: cos'è un repository (§3.1), git init (§3.2), le tre aree (§3.3) e git status, la tua bussola (§3.4). Il modello delle tre aree è la chiave che apre tutto Git: dedicagli attenzione.

🗺 Dove siamo

Cos'è un repository (§3.1), crearlo con git init (§3.2), le tre aree — working, staging, repository (§3.3), git status per orientarti (§3.4). Nel laboratorio (§3.5) crei il tuo primo repo.

🔑 Perché ci tocca: capire le tre aree = capire Git

La maggior parte della confusione dei principianti con Git nasce dal non aver capito le tre aree: si chiedono perché serva add e commit, cosa vuol dire «staged», perché git status dice certe cose. Chiarito questo modello, i comandi smettono di sembrare magia e diventano ovvi. È l'investimento più redditizio di tutto il libro.

Sezione 3.1Cos'è un repository

Un repository (o «repo») è una cartella di progetto che Git sta tracciando: contiene i tuoi file attuali più tutta la loro storia (ogni commit passata). Tutta la «magia» sta in una sottocartella nascosta che Git crea al suo interno.

📖 La cartella .git: il cervello del repository

Quando trasformi una cartella in repository, Git crea al suo interno una cartella nascosta chiamata .git. Lì dentro c'è tutto: ogni commit, ogni branch, la configurazione, l'intera storia. I tuoi file di progetto restano dove sono, normali; .git è il «registro» invisibile accanto a loro. Non devi mai toccarla a mano (rischieresti di rompere il repository): ci parli sempre attraverso i comandi git. Cancellare la cartella .git significa smettere di tracciare il progetto (i file restano, ma la storia sparisce). In pratica: cartella normale + .git = repository.

Sezione 3.2git init

Creare un repository è un solo comando: git init, eseguito dentro la cartella che vuoi tracciare.

terminale — inizializzare un repository
$ mkdir mio-progetto       # crea la cartella del progetto
$ cd mio-progetto          # entraci
$ git init                  # trasformala in un repository Git
Initialized empty Git repository in /Users/anna/mio-progetto/.git/
git init smontato un comando, una volta per progetto
git init
crea la cartella nascosta .git nella posizione corrente, trasformandola in un repository. Da questo momento Git «osserva» questa cartella e i suoi file.
«empty Git repository»
il messaggio conferma: repository creato, ma vuoto (nessuna commit ancora). Contiene la struttura, non ancora la storia.
una volta sola
git init si esegue una sola volta per progetto, all'inizio. Rifarlo su un repo esistente non serve (e non fa danni, ma è inutile).
l'alternativa: git clone
quando un progetto esiste già online (su GitHub), invece di init lo scarichi con git clone (Cap. 10) — che crea il repository già pieno di storia.
Attenzione a dove lanci git init: crea il repository nella cartella corrente (controlla con pwd, Cap. 2). Un errore comune è lanciarlo per sbaglio nella home o in una cartella sbagliata, iniziando a tracciare troppa roba. Assicurati di essere dentro la cartella del progetto prima di eseguirlo.

Sezione 3.3Le tre aree: working, staging, repository

Ecco il concetto centrale. In Git, quando salvi il lavoro, un file compie un viaggio in tre tappe. Capirlo spiega tutto il resto.

Il viaggio di una modifica working → staging → repository
1 · la working directory (cartella di lavoro)
dove modifichi davvero i file: è la cartella del progetto come la vedi. Qui le modifiche sono «libere», non ancora registrate da Git.
2 · la staging area (area di sosta / «index»)
una zona di preparazione: ci metti (con git add) le modifiche che vuoi includere nel prossimo snapshot. È come una «scatola» dove selezioni cosa salvare, prima di salvare davvero.
3 · il repository (la storia)
quando fai git commit, ciò che è nella staging area diventa uno snapshot permanente nella storia. Da qui non si perde più.
i due comandi che spostano
git add sposta le modifiche da working → staging; git commit le sposta da staging → repository. Due passi, due comandi.
Perché due passi e non uno? Perché la staging area ti dà controllo: puoi scegliere quali modifiche includere in una commit e quali lasciar fuori, componendo snapshot puliti e sensati. Immagina di aver cambiato tre file ma di voler salvare solo due modifiche insieme (perché riguardano la stessa cosa): metti quei due in staging e fai commit; il terzo resta in attesa. È una comodità che apprezzerai col tempo. Prova il widget qui sotto: sposta un file lungo le tre aree e osserva git status.

Sezione 3.4git status: la tua bussola


  

    Il comando che userai più di ogni altro è git status: ti dice, in ogni momento, in che stato
    sono i tuoi file rispetto alle tre aree. È la tua bussola: quando sei incerto, git status.
  


  

    
terminale — git status racconta la situazione

    
$ git status
On branch main

No commits yet

Untracked files:
  (use "git add <file>..." to include in what will be committed)
        note.txt

nothing added to commit but untracked files present

  


  

    
Leggere git status gli stati di un file

    

      

        
untracked (non tracciato)

        
un file nuovo che Git non sta ancora seguendo (mai stato aggiunto). Git lo vede ma non lo
            registra finché non fai git add.

        
modified (modificato)

        
un file già tracciato che hai cambiato nella working directory, ma la modifica non è ancora in staging.

        
staged (in staging, «to be committed»)

        
una modifica che hai messo in staging con git add: pronta per la prossima commit.
            git status la elenca sotto «Changes to be committed».

        
i suggerimenti di Git

        
nota come git status ti dica anche cosa fare («use git add…»): Git è un
            assistente, non solo un esecutore. Leggere questi suggerimenti è il modo più veloce per imparare.

      

    

    

      Abitudine d'oro: esegui git status spessissimo — prima e dopo ogni comando, quando sei
      in dubbio, quando riprendi il lavoro. Ti dice sempre «dove sei» nel viaggio delle tre aree, e ti suggerisce il
      passo successivo. È impossibile «perdersi» in Git se controlli lo status di frequente. Nessun comando fa danni:
      git status è solo lettura.
    

  



  
  
Sezione 3.5Laboratorio: crea un repository


  

    Crea il tuo primo repository e osserva git status in azione. Non facciamo ancora la commit (è il
    Capitolo 4): qui il focus è vedere il repository nascere e imparare a leggere lo stato.
  


  

    
terminale — passo per passo

    
# 1) crea la cartella del progetto ed entra
$ mkdir diario && cd diario

# 2) trasformala in repository
$ git init
Initialized empty Git repository in .../diario/.git/

# 3) controlla lo stato: repository vuoto, nessun file
$ git status
On branch main
No commits yet
nothing to commit

# 4) crea un file e ricontrolla
$ echo "Oggi ho imparato Git." > giorno1.txt
$ git status
Untracked files:
        giorno1.txt
nothing added to commit but untracked files present

  


  

    
✅ Cosa osservare

    

      Dopo git init, il repository esiste ma è vuoto («No commits yet»). Creando
      giorno1.txt, git status lo segnala come untracked: Git lo vede, ma non lo
      sta ancora tracciando — aspetta un tuo git add. Questo è il file nella working directory, non
      ancora nelle altre due aree. Nota che Git ti suggerisce già il comando da usare. Nel prossimo capitolo
      faremo proprio quello: git add e poi git commit, completando il viaggio del file fino
      alla storia. (Curiosità: prova ls -a per vedere la cartella nascosta .git creata da
      git init.)

Sezione 3.7Soluzioni


  

    
Soluzione 3.1

    

      La cartella .git contiene tutto ciò che rende la cartella un repository: ogni commit
      (l'intera storia), i branch, la configurazione locale, i riferimenti. Se la cancellassi, la cartella tornerebbe
      a essere una cartella normale: i file di progetto attuali resterebbero, ma tutta la storia
      versionata andrebbe persa (commit, branch, ecc.), e Git smetterebbe di tracciare il progetto.
    

  


  

    
Soluzione 3.2

    

      Le tre aree, nell'ordine: 1) working directory (dove modifichi i file) → 2) staging area (dove
      prepari cosa salvare) → 3) repository (dove lo snapshot è registrato). git add sposta le
      modifiche da working a staging; git commit le sposta da staging a repository.
    

  


  

    
Soluzione 3.3

    

      Lo segnala come untracked (non tracciato). Significa che Git vede il file nella cartella ma non lo
      sta ancora seguendo: non fa parte di nessuna commit e le sue modifiche non vengono registrate. Diventerà
      tracciato solo dopo un git add, che lo porta nella staging area.
    

  


  

    
Soluzione 3.4

    

      La staging area dà controllo su cosa finisce in ciascuna commit: puoi scegliere quali modifiche
      includere e quali lasciar fuori, invece di salvare tutto in blocco. Questo permette di comporre commit
      pulite e coerenti — per esempio raggruppando solo le modifiche che riguardano la stessa correzione, e
      rimandando le altre a una commit successiva. È un passo in più, ma ripaga in ordine e chiarezza della storia.
