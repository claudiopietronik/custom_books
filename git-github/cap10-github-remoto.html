Capitolo 10

GitHub: il repository remoto

Hai imparato a versionare sul tuo computer: creare commit, ramificare, unire. Tutto questo vive però in un solo posto — il tuo disco. Se il computer si rompe, se vuoi lavorare da un'altra macchina, o se vuoi collaborare con qualcuno, hai bisogno di una copia condivisa e online del repository. È il repository remoto, e il luogo più diffuso dove ospitarlo è GitHub. Da qui in poi il libro cambia scenario: dal Git «in solitaria» alla collaborazione. In questo capitolo capiamo cos'è un repository remoto (§10.1), come creare un account e un repository su GitHub (§10.2), come collegare il tuo repository locale a quello remoto (§10.3), e come clonare un progetto già esistente (§10.4). È il ponte tra il tuo computer e il resto del mondo: backup, accesso da ovunque, e collaborazione.

🗺 Dove siamo

Cos'è un repository remoto (§10.1), creare account e repo su GitHub (§10.2), collegare locale e remoto con git remote (§10.3), git clone per scaricare un progetto (§10.4). Nel laboratorio (§10.5) pubblichi il tuo primo repo.

🔑 Perché ci tocca: senza remoto non c'è collaborazione (né backup)

Un repository solo locale è fragile e solitario: un guasto e sparisce, e nessuno può lavorarci con te. Il remoto risolve entrambe le cose: è un backup sempre aggiornato e il punto d'incontro del team. GitHub è, di fatto, dove vive il codice del mondo — dalle piccole app ai grandi progetti open source. Saperci pubblicare e collaborare è indispensabile per lavorare con altri e per costruirti un portfolio.

Sezione 10.1Cos'è un repository remoto

Un repository remoto è una copia del tuo repository che vive su un server, accessibile via rete (come su GitHub). Tu continui a lavorare sul tuo repo locale, e periodicamente lo sincronizzi con quello remoto.

Locale e remoto: due copie che si sincronizzano il modello di GitHub
il repository locale
quello sul tuo computer, dove fai commit, branch, merge (tutto il libro finora). Qui lavori offline, come sempre.
il repository remoto
una copia su un server (GitHub), condivisa. Non ci «lavori» direttamente: è un punto di raccolta e scambio. Ci mandi le tue commit e da lì scarichi quelle degli altri.
la sincronizzazione
due direzioni: push (manda le tue commit al remoto) e pull (scarica quelle nuove dal remoto). Sono il tema del Cap. 11. Il remoto è il «terreno comune» su cui tutti convergono.
origin
il nome convenzionale del remoto principale (quello da cui hai clonato o a cui ti sei collegato). Lo vedrai spesso: git push origin main = «manda a origin il branch main».
Un punto importante: ogni collaboratore ha una copia locale completa del repository (con tutta la storia!), e il remoto è la copia «ufficiale» condivisa. Non è come un file su un disco condiviso che tutti modificano insieme: ognuno lavora sulla sua copia e sincronizza col remoto quando è pronto. Questo modello «distribuito» è ciò che rende Git robusto e adatto al lavoro di gruppo.

Sezione 10.2Creare un account e un repository su GitHub

GitHub è un sito web: per usarlo crei un account gratuito, poi puoi creare repository (pubblici o privati). Non servono comandi qui — si fa dal browser.

I passi su GitHub dal browser, una volta
1 · crea un account
su github.com, gratuito. Usa la stessa email che hai messo in git config (Cap. 2), così GitHub collega le tue commit al profilo.
2 · crea un nuovo repository
col pulsante «New repository»: gli dai un nome, scegli se pubblico (visibile a tutti) o privato (solo tu e chi inviti), e crei. Ottieni un repository remoto vuoto, con un URL.
3 · l'URL del repository
qualcosa come https://github.com/tuo-nome/mio-progetto.git. È l'indirizzo che userai per collegare il tuo repo locale (§10.3) o per clonarlo (§10.4).
l'autenticazione
per mandare dati a GitHub dovrai autenticarti. Oggi si usa di solito un token di accesso (Personal Access Token) o SSH, non la password. GitHub ti guida la prima volta; è un passaggio una tantum.
Pubblico o privato? Un repo pubblico è visibile a tutti (ideale per progetti open source e per il tuo portfolio); uno privato solo a te e a chi inviti (per lavori personali o riservati). Puoi cambiarlo in seguito. Per imparare, un repo pubblico va benissimo — ma ricorda la regola d'oro (Cap. 13): non mettere mai password o segreti in un repository, tanto meno pubblico.

Sezione 10.3Collegare locale e remoto

Hai un repository locale (creato con git init) e ne hai creato uno vuoto su GitHub. Ora li colleghi con git remote add, poi mandi su GitHub la tua storia con il primo push.

terminale — collegare il locale a GitHub
# 1) collega il tuo repo locale al remoto, chiamandolo "origin"
$ git remote add origin https://github.com/anna/mio-progetto.git

# 2) verifica il collegamento
$ git remote -v
origin  https://github.com/anna/mio-progetto.git (fetch)
origin  https://github.com/anna/mio-progetto.git (push)

# 3) manda la storia locale al remoto (prima volta: -u imposta il collegamento)
$ git push -u origin main
... il tuo codice è ora su GitHub!
git remote e il primo push remote add · remote -v · push -u
git remote add origin <url>
collega il repo locale a un remoto, dandogli il nome origin (la convenzione). Non copia nulla ancora: dice solo «il mio remoto ufficiale è a questo URL».
git remote -v
elenca i remoti collegati (con i loro URL). Un modo per controllare che il collegamento sia giusto.
git push -u origin main
il primo push: manda il branch main a origin. Il -u («upstream») imposta il collegamento, così dalla volta dopo ti basterà git push (senza specificare origin e main). Il push lo vediamo a fondo al Cap. 11.
Dopo questo, ricaricando la pagina del repository su GitHub, vedrai i tuoi file e tutta la storia delle commit, online. Il tuo lavoro locale ha ora una copia remota: sei protetto da guasti e pronto a collaborare. Nota che questa procedura serve quando parti dal locale; se invece il progetto esiste già su GitHub, non colleghi: lo cloni (§10.4).

Sezione 10.4Clonare un repository esistente


  

    L'altro scenario: il progetto esiste già su GitHub (tuo o di qualcun altro) e vuoi una copia sul tuo
    computer per lavorarci. Si fa con git clone, che scarica tutto — file, storia, branch — in una
    volta sola.
  


  

    
terminale — clonare un repository

    
$ git clone https://github.com/anna/mio-progetto.git
Cloning into 'mio-progetto'...
... done.

$ cd mio-progetto          # entra nella cartella scaricata
# è già un repository completo, col remoto "origin" GIÀ collegato!

  


  

    
git clone smontato scarica tutto, già collegato

    

      

        
git clone <url>

        
scarica una copia completa del repository remoto in una nuova cartella: tutti i file, tutta la
            storia, tutti i branch. In un colpo solo hai un repository locale funzionante.

        
già collegato a origin

        
clonare imposta automaticamente il remoto origin verso l'URL da cui hai clonato. Non
            serve git remote add: puoi subito fare push e pull (Cap. 11).

        
clone vs init + remote add

        
due modi di ottenere un repo locale collegato: init + remote add quando
            parti dal tuo locale (§10.3); clone quando parti da un remoto esistente.
            Stesso risultato, punti di partenza opposti.

      

    

    

      git clone è il modo in cui contribuisci a progetti altrui: cloni il loro repository, ci lavori in
      locale, e proponi le tue modifiche (con le pull request, Cap. 12). È il primo comando che dai quando ti
      unisci a un progetto esistente — al lavoro, o nell'open source. Ricorda dal Cap. 3: non si usa
      git init su un progetto che si clona; clone crea già tutto.
    

  



  
  
Sezione 10.5Laboratorio: il tuo primo repo su GitHub


  

    Pubblica il repository diario (dei capitoli scorsi) su GitHub. È il momento in cui il tuo lavoro
    locale diventa online e condivisibile.
  


  

    
terminale — dal locale a GitHub

    
# --- su github.com (browser): crea un nuovo repo vuoto "diario", copia il suo URL ---

# --- nel terminale, dentro la cartella "diario": ---
$ git remote add origin https://github.com/TUO-NOME/diario.git
$ git branch -M main          # assicura che il branch si chiami "main"
$ git push -u origin main    # manda tutto su GitHub

# ora ricarica la pagina del repo su GitHub: ci sono i tuoi file e le commit!

  


  

    
✅ Cosa hai ottenuto

    

      Il tuo repository diario è ora anche su GitHub: hai un backup online e un punto da cui
      condividere e collaborare. Su GitHub puoi vedere i file, sfogliare la storia delle commit, i branch — tutto
      il lavoro fatto in locale, ora nel cloud. Da qui in avanti, il ciclo quotidiano avrà un passo in più: dopo aver
      committato in locale, farai git push per mandare le novità su GitHub (Cap. 11). E se lavori da
      un'altra macchina, ti basterà git clone per riavere tutto. Il tuo Git è «uscito» dal computer: la
      collaborazione può iniziare.

Sezione 10.7Soluzioni


  

    
Soluzione 10.1

    

      Il repository locale è la copia sul tuo computer, dove lavori (commit, branch, merge). Il repository
      remoto è una copia su un server (es. GitHub), condivisa e usata come backup e punto d'incontro: non ci
      lavori direttamente, ci sincronizzi le modifiche. origin è il nome convenzionale del remoto
      principale a cui il tuo locale è collegato (quello da cui hai clonato o a cui ti sei collegato).
    

  


  

    
Soluzione 10.2

    

      
terminale

      
$ git remote add origin <url-del-repo-su-github>
$ git push -u origin main

    

    
Prima colleghi il remoto (remote add origin), poi mandi il branch
      main con push -u (il -u imposta il collegamento per i push successivi).

  


  

    
Soluzione 10.3

    

      Usi git clone quando il progetto esiste già su un remoto (GitHub) e vuoi scaricarne una copia
      locale: è il punto di partenza opposto a init (che si usa quando crei un progetto nuovo dal tuo
      computer). clone fa in più, automaticamente: scarica tutti i file, la storia e i branch,
      e imposta già il remoto origin verso l'URL clonato — quindi puoi subito fare push e pull senza
      git remote add.
    

  


  

    
Soluzione 10.4

    

      Perché un repository conserva tutta la storia: anche se cancelli un segreto in una commit successiva,
      resta visibile nelle commit precedenti. Su un repo pubblico chiunque potrebbe leggerlo; e se il repo è del
      tuo lavoro, il segreto è comunque esposto a tutti i collaboratori e a eventuali fughe. Password, chiavi API e
      token vanno tenuti fuori dal repository (in variabili d'ambiente o file esclusi con .gitignore,
      Cap. 13). Una credenziale finita per errore in un repo va considerata compromessa e cambiata.
