Capitolo 5

La storia del progetto

Hai iniziato a costruire una storia di commit: una sequenza di snapshot, ciascuno con il suo messaggio. Ma una storia ha valore solo se sai consultarla. In questo capitolo impari a esplorarla con tre strumenti che userai in continuazione: git log per scorrere l'elenco delle commit (chi, quando, cosa), git diff per vedere esattamente quali righe sono cambiate, e git show per ispezionare una singola commit nel dettaglio. Sono i comandi che trasformano la storia da «lista di etichette» a risorsa navigabile: ti permettono di rispondere a domande come «cosa è cambiato ieri?», «chi ha toccato questo file?», «quando è entrato questo comportamento?». Sono anche gli strumenti di base per il debug: quando qualcosa si rompe, la storia spesso contiene la risposta. In questo capitolo: git log (§5.1), leggere una commit (§5.2), git diff (§5.3), git show (§5.4).

🗺 Dove siamo

git log e le sue viste (§5.1), leggere una commit — hash, autore, data (§5.2), git diff per le modifiche riga per riga (§5.3), git show per una commit intera (§5.4). Nel laboratorio (§5.5) esplori una storia reale.

🔑 Perché ci tocca: la storia è una risorsa, se sai leggerla

Un progetto versionato accumula centinaia di commit. Saperle interrogare significa poter capire come e perché il codice è arrivato dov'è — indispensabile per correggere bug, riprendere un lavoro, o entrare in un progetto altrui. Chi non sa leggere la storia «subisce» Git; chi sa farlo lo usa come una macchina del tempo investigativa. Questi comandi sono di sola lettura: puoi esplorare liberamente, senza rischi.

Sezione 5.1git log: la linea del tempo

git log mostra l'elenco delle commit, dalla più recente alla più vecchia: la linea del tempo del tuo progetto. È il primo comando che lanci per «vedere cosa è successo».

terminale — git log completo
$ git log
commit e4f5g6h7i8j9... (HEAD -> main)
Author: Anna Rossi <anna@esempio.com>
Date:   Sat Jul 19 10:32:00 2026

    Aggiungi il secondo giorno

commit a1b2c3d4e5f6...
Author: Anna Rossi <anna@esempio.com>
Date:   Sat Jul 19 10:15:00 2026

    Aggiungi il primo giorno
Le viste di git log completo · oneline · graph
git log
la vista completa: per ogni commit mostra hash, autore, data e messaggio. Dettagliata ma lunga. Premi q per uscire se il log riempie lo schermo (Git lo apre in un «pager»).
git log --oneline
una riga per commit: hash breve + messaggio. La vista più usata per avere un colpo d'occhio rapido sulla storia.
git log --oneline --graph
aggiunge un grafico a sinistra che disegna i branch e i merge (Cap. 7-8). Utilissima quando la storia si ramifica.
git log -n 3 · git log nomefile
limita alle ultime N commit, o mostra solo le commit che hanno toccato un certo file. Modi per filtrare quando la storia è lunga.
git log --oneline sarà la tua vista quotidiana. Nota, nella vista completa, l'annotazione (HEAD -> main) sulla commit più recente: indica dove ti trovi nella storia (HEAD) e su quale branch (main) — concetti che chiariremo al Cap. 7. Per ora, ti basti sapere che segnala «l'ultima commit, quella in cima».

Sezione 5.2Leggere una commit: hash, autore, data

Ogni commit nel log ha alcuni elementi fissi. Capirli ti permette di riferirti a una commit precisa e di interpretare la storia.

Gli elementi di una commit hash · autore · data · messaggio
l'hash (es. a1b2c3d)
l'identificatore univoco della commit, un lungo codice esadecimale (mostrato accorciato nelle prime 7 cifre). È l'«indirizzo» della commit: lo usi per riferirti a lei in altri comandi (es. git show a1b2c3d, Cap. 6). Bastano le prime ~7 cifre, sono già univoche.
l'autore e l'email
chi ha creato la commit — proprio i user.name e user.email del Cap. 2. Ecco perché li avevi configurati: appaiono qui.
la data
quando è stata fatta la commit. Utile per orientarsi nel tempo.
il messaggio
la descrizione che hai scritto (Cap. 4). La parte che leggi per capire cosa fa quella commit — e il motivo per cui vanno scritti bene.
L'hash merita una nota: non è un numero progressivo (1, 2, 3…) ma un codice calcolato dal contenuto della commit. Sembra strano ma è potente: garantisce che ogni commit sia identificata in modo univoco e immutabile in tutto il mondo (anche tra repository diversi). Quando dovrai indicare «quella commit», userai il suo hash (o le sue prime cifre).

Sezione 5.3git diff: cosa è cambiato

git log ti dice quali commit ci sono; git diff ti mostra esattamente quali righe sono cambiate. È lo strumento per vedere le modifiche nel dettaglio, riga per riga.

terminale — git diff mostra le modifiche non ancora in staging
$ git diff
diff --git a/saluto.py b/saluto.py
@@ -1,2 +1,2 @@
 def saluta(nome):
-    print("ciao " + nome)
+    print(f"Ciao, {nome}!")
Leggere un diff righe − rosse, + verdi
le righe con − (rosse)
ciò che è stato rimosso (la versione vecchia).
le righe con + (verdi)
ciò che è stato aggiunto (la versione nuova). Una modifica appare come una riga tolta + una aggiunta, come nell'esempio.
le righe senza segno
il contesto immutato attorno alla modifica, per farti capire dove sei.
le varianti
git diff = modifiche non ancora in staging; git diff --staged = modifiche già in staging (cosa entrerà nella prossima commit); git diff a1b2c3d = differenze rispetto a una commit passata. Sempre lo stesso formato +/−.
Il formato diff (righe rosse tolte, verdi aggiunte) è uno standard universale: lo ritrovi identico su GitHub nelle pull request (Cap. 12), negli editor, ovunque si confronti codice. Imparare a leggerlo qui ti servirà ovunque. Un'abitudine utilissima: guarda il diff prima di ogni commit (git diff --staged) per controllare esattamente cosa stai per salvare — evita di committare per sbaglio codice di debug o modifiche indesiderate.
Esplora la storia: clicca una commit, vedi il diff come git log + git show
git log --oneline
    git show <commit>
    
      
    
    
    
    
    
  
    


  
  
    Sezione 5.4git show: guardare una commit
    

  
    
    git show unisce le due cose: prende una commit (per hash) e ne mostra tutto — i metadati (come
    git log) e le modifiche che ha introdotto (come git diff). È lo strumento per
    ispezionare «cosa ha fatto, di preciso, quella commit».
  
    

  
    
    
    terminale — git show di una commit
    
    
    $ git show a1b2c3d       # indichi la commit col suo hash
commit a1b2c3d4e5f6...
Author: Anna Rossi <anna@esempio.com>
Date:   Sat Jul 19 10:15:00 2026

    Aggiungi il primo giorno

diff --git a/giorno1.txt b/giorno1.txt
+Giorno 1: ho imparato git init.
    
  
    

  
    
    
    📖 log, diff, show: quando usare cosa
    
    
    
      Un riepilogo per non confonderli: git log = «elenco delle commit» (la lista);
      git diff = «cosa è cambiato ora» (le modifiche in corso, non ancora committate);
      git show <hash> = «cosa ha fatto quella commit» (metadati + modifiche di una commit
      passata). Prima trovi la commit con log, poi la ispezioni con show. Se ometti
      l'hash, git show mostra l'ultima commit — un modo rapido per rivedere cosa hai appena salvato.
    
    
  
    


  
  
    Sezione 5.5Laboratorio: esplorare la storia
    

  
    
    Usa i comandi di questo capitolo sul repository diario (con le commit del Cap. 4). Esplorare una
    storia reale fissa i concetti meglio di mille spiegazioni.
  
    

  
    
    
    terminale — un giro di esplorazione
    
    
    # 1) elenco compatto delle commit
$ git log --oneline
e4f5g6h Aggiungi il secondo giorno
a1b2c3d Aggiungi il primo giorno

# 2) ispeziona una commit specifica (copia il suo hash dal log)
$ git show a1b2c3d
... metadati + diff della prima commit ...

# 3) fai una modifica e guarda il diff PRIMA di committare
$ echo "riga aggiunta" >> giorno1.txt
$ git diff
+riga aggiunta

# 4) confronta due commit (differenze tra la prima e l'ultima)
$ git diff a1b2c3d e4f5g6h
    
  
    

  
    
    
    ✅ Cosa hai imparato a fare

      (show), e vedere le modifiche in corso o tra due punti (diff). Prova soprattutto
      il passo 3: guardare il diff prima di committare è un'abitudine che ti salverà da tante commit
      sbagliate. Sono tutti comandi di sola lettura: non modificano nulla, quindi esplora quanto vuoi. Nel
      prossimo capitolo useremo questi stessi hash per fare qualcosa di più potente: annullare e correggere.

    Sai creare la storia e ora sai leggerla. Ma cosa fai quando sbagli? Hai modificato un file e vuoi tornare

    Sezione 5.7Soluzioni
    

  
    
    
    Soluzione 5.1
    
    
    
      Per la vista compatta: git log --oneline (hash breve + messaggio, una riga per commit). Per uscire
      da git log (o da qualsiasi «pager» che riempie lo schermo) si preme q (quit).
    
    
  
    

  
    
    
    Soluzione 5.2
    
    
    
      Le righe con - (spesso rosse) sono state rimosse (versione vecchia); quelle con +
      (verdi) sono state aggiunte (versione nuova). Una riga modificata appare come due righe:
      quella vecchia con - e quella nuova con + — Git non ha un simbolo «modificato», la
      rappresenta come una rimozione seguita da un'aggiunta.
    
    
  
    

  
    
    
    Soluzione 5.3
    
    
    
      git log: mostra l'elenco delle commit della storia. git diff: mostra le
      modifiche riga per riga attualmente in corso (o tra due punti indicati). git show <hash>:
      mostra tutto di una singola commit — i suoi metadati e le modifiche che ha introdotto.
    
    
  
    

  
    
    
    Soluzione 5.4
    
    
    
      Perché git diff --staged mostra esattamente cosa sta per entrare nella commit (le modifiche
      in staging). Controllarlo prima di committare ti permette di verificare di salvare solo ciò che intendi —
      evitando di includere per sbaglio codice di debug, file temporanei, o modifiche che volevi lasciare fuori. È un
      controllo rapido che previene molte commit «sporche».
