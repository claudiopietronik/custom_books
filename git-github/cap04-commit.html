Capitolo 4

Salvare le modifiche: la commit

Eccoci al gesto centrale di Git, quello che ripeterai migliaia di volte: fare una commit, cioè salvare uno snapshot del tuo progetto nella storia. Dal capitolo scorso sai che una modifica compie un viaggio in tre aree; ora impariamo i due comandi che lo completano: git add (porta le modifiche in staging) e git commit (le registra per sempre, con un messaggio che descrive cosa hai fatto). Vedremo perché il messaggio di commit è tutt'altro che un dettaglio — è ciò che rende la storia leggibile e utile, a te e agli altri, anche a mesi di distanza. E consolideremo il ciclo add-commit: il battito cardiaco del lavoro quotidiano con Git. In questo capitolo: git add (§4.1), git commit (§4.2), il messaggio (§4.3) e il ciclo completo (§4.4). Dopo questo capitolo saprai davvero versionare un progetto: è il cuore di tutto Git.

🗺 Dove siamo

git add (§4.1), git commit (§4.2), il messaggio di commit (§4.3), il ciclo add-commit (§4.4). Nel laboratorio (§4.5) fai le tue prime commit e vedi la storia crescere.

🔑 Perché ci tocca: è l'azione che fai di più

Se c'è un gesto che definisce l'uso di Git, è fare commit. Ogni progresso, ogni correzione, ogni idea salvata è una commit. Farle bene — con contenuti coerenti e messaggi chiari — trasforma la storia del progetto in una risorsa preziosa (per capire, per tornare indietro, per collaborare). Farle male la rende un pasticcio. È un'abilità che vale la pena affinare fin da subito.

Sezione 4.1git add: preparare le modifiche

Il primo passo per salvare è git add: sposta le modifiche dalla working directory alla staging area (Cap. 3), cioè le «seleziona» per la prossima commit.

terminale — aggiungere alla staging area
$ git add giorno1.txt       # aggiungi un file specifico
$ git add .                 # aggiungi TUTTO ciò che è cambiato

$ git status
Changes to be committed:
        new file: giorno1.txt
git add smontato file singolo · tutto · lo stato «staged»
git add nomefile
mette in staging le modifiche di quel file. Utile per selezionare con precisione cosa includere nella prossima commit.
git add .
il punto significa «tutto ciò che è cambiato in questa cartella e sottocartelle». Il modo più comune e veloce quando vuoi salvare tutte le modifiche insieme.
dopo l'add: stato staged
git status ora elenca il file sotto «Changes to be committed» (in verde): è pronto per la commit. Ma non è ancora salvato — serve il commit.
ripensarci
hai messo in staging qualcosa per sbaglio? Puoi toglierlo senza perdere il lavoro (lo vedremo al Cap. 6, git restore --staged). Lo staging è reversibile.
Pensa a git add come al «mettere gli articoli nel carrello»: scegli cosa comprare, ma non hai ancora pagato. Il pagamento è il commit (prossima sezione). Puoi aggiungere più cose al carrello (più git add) prima di «pagare» una volta sola. Questa immagine — carrello e cassa — aiuta a ricordare perché servono due comandi.

Sezione 4.2git commit: salvare uno snapshot

Con le modifiche in staging, git commit le registra come uno snapshot permanente nella storia del repository. Ogni commit ha un messaggio che la descrive.

terminale — fare una commit
$ git commit -m "Aggiungi il primo giorno del diario"
[main (root-commit) a1b2c3d] Aggiungi il primo giorno del diario
 1 file changed, 1 insertion(+)
 create mode 100644 giorno1.txt
git commit smontato -m · l'hash · il riepilogo
git commit -m "messaggio"
crea la commit con il messaggio indicato. Il -m («message») ti fa scrivere il messaggio direttamente sulla riga di comando. Senza -m, Git aprirebbe un editor di testo per scriverlo.
l'hash a1b2c3d
ogni commit riceve un identificatore univoco (un codice lungo, di solito mostrato accorciato). È l'«impronta digitale» della commit: la userai per riferirti a lei (Cap. 5-6).
root-commit
appare solo per la primissima commit del repository (non ha commit precedenti). Dalla seconda in poi, non lo vedrai più.
il riepilogo
«1 file changed, 1 insertion»: Git riassume cosa è entrato nella commit (quanti file, quante righe aggiunte/rimosse). Un rapido controllo che hai salvato ciò che intendevi.
Fatto! La modifica è ora al sicuro nella storia: anche se cancelli o rovini il file, puoi sempre recuperare questa versione. La working directory torna «pulita» (git status dirà «nothing to commit»), pronta per le prossime modifiche. Hai completato il viaggio delle tre aree del Cap. 3: working → staging (add) → repository (commit).

Sezione 4.3Il messaggio di commit

Il messaggio non è una formalità: è la spiegazione di cosa hai cambiato e perché. Un buon messaggio rende la storia leggibile; uno pigro la rende inutile. Vale la pena imparare a scriverli bene fin da subito.

Anatomia di un buon messaggio chiaro · conciso · al presente
descrivi cosa fa la commit
una frase breve che riassume la modifica: «Aggiungi la validazione dell'email», «Correggi il calcolo del totale». Deve bastare a capire, senza aprire il codice.
usa l'imperativo presente
la convenzione: «Aggiungi…», «Correggi…», «Rimuovi…» (come un comando), non «Aggiunto» o «Aggiungendo». Suona come «questa commit fa X». È lo stile usato ovunque nel mondo Git.
sii specifico, non generico
evita messaggi vuoti come «modifiche», «fix», «aggiornamento», «asdf». Tra sei mesi non ti diranno nulla. Meglio «Correggi il bug del login con email maiuscole».
breve la prima riga (~50 caratteri)
la prima riga è il «titolo» della commit; tienila concisa. Se serve, aggiungi una riga vuota e altri dettagli sotto (per commit importanti).
messaggi: buoni vs pigri
# ✔ buoni (chiari, all'imperativo, specifici):
"Aggiungi la pagina di contatti"
"Correggi il crash quando la lista è vuota"
"Rimuovi il codice di debug inutilizzato"

# ✗ pigri (vaghi, inutili tra sei mesi):
"modifiche"    "fix"    "aggiornamento"    "asdf"    "."
Una prova pratica: un buon messaggio completa la frase «Questa commit, se applicata, …» — «…aggiunge la pagina di contatti». Se il tuo messaggio non ci sta bene, probabilmente è troppo vago. Non è pedanteria: la storia dei messaggi è ciò che leggi (con git log, Cap. 5) per capire l'evoluzione del progetto e per ritrovare quando è stato introdotto un cambiamento. Messaggi curati = storia utile.

Sezione 4.4Il ciclo add-commit

Ora unisci tutto: il lavoro con Git è, in gran parte, la ripetizione di un ciclo semplice. Modifichi, aggiungi, committi — e ricominci. È il ritmo quotidiano.

🔁 Il battito cardiaco di Git
il ciclo, ripetuto all'infinito
# 1. LAVORA: modifica i tuoi file (nell'editor)
# 2. CONTROLLA: cosa è cambiato?
$ git status
# 3. PREPARA: metti in staging
$ git add .
# 4. SALVA: fai la commit con un messaggio
$ git commit -m "Descrivi cosa hai fatto"
# ...e ricomincia dal punto 1

Questo ciclo è il 90% dell'uso quotidiano di Git. Modifica → status → add → commit. Quanto spesso committare? Regola pratica: una commit per ogni «unità logica» di lavoro compiuta — una funzionalità aggiunta, un bug corretto, una sezione scritta. Commit piccole e frequenti (Cap. 13) sono meglio di una gigante alla fine: più facili da capire e da annullare se serve. Prova il ciclo nel widget qui sotto.

Sezione 4.5Laboratorio: le prime commit


  

    Facciamo due commit vere nel repository del capitolo scorso, così vedi la storia nascere e crescere. Ti serve il
    repo diario (o creane uno nuovo con git init).
  


  

    
terminale — due commit dall'inizio

    
# crea un file e fai la PRIMA commit
$ echo "Giorno 1: ho imparato git init." > giorno1.txt
$ git add giorno1.txt
$ git commit -m "Aggiungi il primo giorno"
[main (root-commit) a1b2c3d] Aggiungi il primo giorno

# modifica/aggiungi e fai la SECONDA commit
$ echo "Giorno 2: ho imparato add e commit." > giorno2.txt
$ git add .
$ git commit -m "Aggiungi il secondo giorno"
[main e4f5g6h] Aggiungi il secondo giorno

# guarda la storia: due commit!
$ git log --oneline
e4f5g6h Aggiungi il secondo giorno
a1b2c3d Aggiungi il primo giorno

  


  

    
✅ Cosa hai costruito

    

      Hai creato una storia di due commit: due snapshot del progetto, ciascuno con il suo messaggio e il suo
      hash. git log --oneline te li mostra dal più recente al più vecchio. Questa è la «macchina del
      tempo» del Cap. 1, ora costruita da te: ogni commit è un punto a cui potrai tornare (Cap. 5-6). Nota il
      ritmo: modifica → add → commit, ripetuto. Continua a usarlo per ogni progresso del tuo diario, e la storia
      crescerà, ordinata e navigabile. Esploreremo git log a fondo nel prossimo capitolo.

Sezione 4.7Soluzioni


  

    
Soluzione 4.1

    

      git add file.txt mette in staging solo quel file; git add . mette in staging
      tutte le modifiche della cartella (e sottocartelle). Preferisci il primo quando hai cambiato più file ma
      vuoi includere in una commit solo alcuni di essi (per comporre una commit coerente, lasciando le altre
      modifiche per una commit successiva).
    

  


  

    
Soluzione 4.2

    

      
terminale

      
$ git add ricetta.txt
$ git commit -m "Aggiungi la ricetta della pizza"

    

    
Prima add (staging), poi commit (salvataggio con
      messaggio).

  


  

    
Soluzione 4.3

    

      Esempi di riscrittura (all'imperativo, specifici): "fix" → «Correggi il calcolo del totale con
      lo sconto»; "modifiche" → «Aggiorna i colori del tema in modalità scura»;
      "roba nuova" → «Aggiungi la funzione di ricerca nei contatti». L'importante è che chi legge
      capisca cosa fa la commit senza aprire il codice.
    

  


  

    
Soluzione 4.4

    

      git add = mettere gli articoli nel carrello (selezioni cosa vuoi salvare, nella staging
      area); git commit = pagare alla cassa (registri definitivamente ciò che è nel carrello,
      creando lo snapshot). Git separa le due fasi per darti controllo: puoi comporre con cura cosa mettere nel
      «carrello» (anche in più passaggi) prima di «pagare» una volta sola, ottenendo commit pulite e coerenti.
