Capitolo 12

Pull request e code review

Sai pushare e pullare — ma in un team pushare direttamente su main è pericoloso: chiunque potrebbe rompere il ramo principale, e nessuno controllerebbe il codice prima che entri. Il flusso professionale usa un meccanismo elegante, che GitHub ha reso lo standard mondiale: la pull request (PR). L'idea: invece di unire tu stesso il tuo branch in main, proponi le tue modifiche e chiedi al team di revisionarle. I colleghi le leggono, commentano, suggeriscono, approvano — è la code review — e solo dopo le modifiche vengono unite. È così che il codice migliora, la conoscenza si condivide, e main resta sano. In questo capitolo: il workflow a feature branch (§12.1), aprire una PR (§12.2), la code review (§12.3) e il merge della PR (§12.4). È il cuore della collaborazione moderna: come lavorano davvero i team di software.

🗺 Dove siamo

Il workflow a feature branch (§12.1), aprire una pull request (§12.2), la code review (§12.3), fare il merge della PR (§12.4). Nel laboratorio (§12.5) segui il ciclo completo di una PR.

🔑 Perché ci tocca: è come si lavora nel mondo reale

Praticamente ogni team di sviluppo, e ogni progetto open source, lavora con le pull request. Saperle usare è un requisito professionale, non un optional: al primo giorno di lavoro ti verrà chiesto di aprirne una. E la code review non è burocrazia: è il modo principale con cui si trovano i bug prima che arrivino agli utenti e con cui si impara dai colleghi. Padroneggiare questo flusso ti rende un collaboratore vero.

Sezione 12.1Il workflow a feature branch

Alla base della pull request c'è un'abitudine che già conosci (Cap. 7-8): non lavorare mai direttamente su main. Ogni modifica nasce su un branch dedicato, e main resta la versione «buona», sempre funzionante.

Il feature branch workflow branch → push → PR → review → merge
1 · crea un branch per la modifica
git switch -c aggiungi-login: un branch dedicato alla funzionalità o correzione. main resta intatto.
2 · lavora e committa, poi pusha il branch
fai le tue commit sul branch e lo mandi su GitHub: git push -u origin aggiungi-login. Il tuo branch è ora anche sul remoto, pronto per la proposta.
3 · apri una pull request
su GitHub, proponi di unire il tuo branch in main (§12.2). Parte la revisione.
4 · review e merge
i colleghi revisionano (§12.3); dopo l'approvazione, la PR viene unita in main (§12.4), e il branch si può cancellare. main cresce solo con codice revisionato.
Riconosci il ciclo del Cap. 8 (ramifica → lavora → unisci), ora con due aggiunte: il lavoro passa da GitHub (push del branch) e c'è un momento di revisione prima dell'unione. Questo trasforma il merge da atto solitario a decisione condivisa. È la differenza tra «Git per me» e «Git per il team».

Sezione 12.2Aprire una pull request

Una pull request è, letteralmente, una richiesta: «per favore, unite (pull) le mie modifiche in main». Si apre dal sito di GitHub, non da un comando Git, e diventa un luogo di discussione attorno al tuo codice.

📖 Cosa contiene una pull request

Dopo aver pushato il tuo branch, su GitHub compare un pulsante «Compare & pull request». Aprendola, indichi: il branch di partenza (il tuo, es. aggiungi-login) e quello di destinazione (di solito main); un titolo e una descrizione di cosa hai fatto e perché. GitHub mostra automaticamente il diff di tutte le tue modifiche (le righe +verdi/−rosse del Cap. 5) e l'elenco delle tue commit. La PR diventa una «pagina» dedicata alla tua proposta, dove il team può leggere il codice, commentarlo riga per riga, e discutere — prima che venga unito. È come dire: «ecco cosa vorrei aggiungere, che ne pensate?».

Sezione 12.3La code review


  

    Il cuore della pull request è la code review: uno o più colleghi leggono le tue modifiche e danno un
    feedback. Non è un giudizio su di te — è un lavoro di squadra per rendere il codice migliore.
  


  

    
Come funziona una review commenti · richieste · approvazione

    

      

        
commenti riga per riga

        
il revisore può commentare una riga specifica del diff: «qui manca un controllo», «bel modo di
            gestirlo», «e se la lista è vuota?». La discussione è ancorata al codice, precisa e utile.

        
richieste di modifica

        
se qualcosa va sistemato, il revisore chiede modifiche. Tu aggiorni il branch (nuove commit +
            push) e la PR si aggiorna da sola: il revisore rivede solo le novità.

        
approvazione

        
quando il codice va bene, il revisore approva la PR. Molti team richiedono almeno un'approvazione
            prima di poter unire. È il «via libera».

        
discussione e apprendimento

        
la review è anche un momento di scambio: si spiega perché una scelta è migliore, si condividono
            trucchi, si allineano gli stili. Si impara tantissimo leggendo e ricevendo review.

      

    

    

      Perché la review conta così tanto? Perché trova i problemi prima che arrivino agli utenti
      (quattro occhi vedono più di due), diffonde la conoscenza nel team (nessuno resta l'unico a capire una parte), e
      alza la qualità nel tempo. Ricevere feedback sul proprio codice all'inizio può intimidire, ma è una delle cose
      più formative del mestiere: accoglilo come un regalo, non come una critica personale.
    

  



  
  
Sezione 12.4Fare il merge della PR


  

    Approvata la review, la pull request può essere unita: le modifiche del branch entrano in main.
    Su GitHub si fa con un pulsante — niente comandi — e poi si fa pulizia.
  


  

    
Concludere la pull request merge · cancella branch · pull in locale

    

      

        
«Merge pull request»

        
il pulsante su GitHub che unisce il branch in main, direttamente sul remoto. Da
            questo momento le tue modifiche sono nella versione ufficiale.

        
cancella il branch

        
GitHub offre di cancellare il branch della feature, ormai unito (come il git branch -d del
            Cap. 8, ma sul remoto). Pulizia: la lista dei branch resta ordinata.

        
aggiorna il tuo locale

        
torna sul tuo computer, vai su main e fai git pull: scarichi la tua
            stessa modifica ora unita in main (più eventuali altre). Sei di nuovo allineato, pronto per la
            prossima feature.

      

    

    

      Il ciclo completo, che ripeterai a ogni funzionalità: switch -c feature → lavori → push del
      branch → apri PR → review → merge su GitHub → git switch main + git pull →
      cancella il branch locale. All'inizio sembra lungo, ma diventa naturale e velocissimo. È il ritmo del
      lavoro professionale, e ora lo conosci per intero.
    

  



  
  
Sezione 12.5Laboratorio: una pull request


  

    Fai una pull request completa sul tuo repo diario (su GitHub). Anche da solo puoi aprire e unire una
    PR: sarai tu il proponente e il revisore — perfetto per imparare il flusso.
  


  

    
terminale + GitHub — il flusso a PR

    
# --- nel terminale ---
$ git switch -c aggiungi-copertina
$ echo "# Il mio diario" > copertina.md
$ git add . && git commit -m "Aggiungi la copertina"
$ git push -u origin aggiungi-copertina

# --- su github.com ---
# 1. compare "Compare & pull request": cliccalo
# 2. scrivi titolo e descrizione, poi "Create pull request"
# 3. (review: leggi il diff; se da solo, approva o unisci direttamente)
# 4. "Merge pull request" -> "Confirm merge" -> "Delete branch"

# --- torna nel terminale, allineati ---
$ git switch main
$ git pull          # ora main locale ha la copertina
$ git branch -d aggiungi-copertina   # pulisci il branch locale

  


  

    
✅ Il flusso professionale, tuo

    

      Hai completato una pull request dall'inizio alla fine: branch → push → PR → (review) → merge → pull →
      pulizia. Questo è, letteralmente, come lavorano i team di software di tutto il mondo — GitHub, le
      aziende, i progetti open source. Da solo hai fatto sia il proponente sia il revisore; in un team, i ruoli si
      dividono, ma il flusso è identico. Nota come main non venga mai toccato «a mano»: cresce solo
      attraverso PR revisionate. Questa disciplina è ciò che tiene sano un progetto con tante persone. Ti manca solo
      un capitolo di «igiene» (cosa non committare, .gitignore) e poi metteremo tutto insieme.

Sezione 12.7Soluzioni


  

    
Soluzione 12.1

    

      Non si lavora direttamente su main per tenerlo sempre stabile e funzionante: ogni modifica
      cresce isolata su un branch dedicato e viene integrata solo dopo la revisione. Questa disciplina garantisce che
      in main entri solo codice revisionato e approvato, riducendo il rischio di rompere la
      versione principale e permettendo il controllo di qualità (code review) prima dell'unione.
    

  


  

    
Soluzione 12.2

    

      Una pull request è una richiesta di unire un branch (con le tue modifiche) in un altro (di solito
      main). A chi la revisiona mostra il diff completo delle modifiche (righe aggiunte/rimosse),
      l'elenco delle commit, e uno spazio per titolo, descrizione e commenti. Si apre dal sito di GitHub
      (non da un comando Git), dopo aver pushato il branch.
    

  


  

    
Soluzione 12.3

    

      Due benefici della code review: (1) trova bug e problemi prima che arrivino agli utenti (più occhi
      controllano); (2) condivide conoscenza nel team e alza la qualità/coerenza del codice (si impara
      dai colleghi). Se un revisore chiede una modifica, la applichi sul branch con nuove commit e fai
      push: la pull request si aggiorna automaticamente con le nuove modifiche, e il revisore rivede solo
      le novità.
    

  


  

    
Soluzione 12.4

    

      
terminale

      
$ git switch main
$ git pull
$ git branch -d nome-del-branch

    

    
Torni su main, fai pull per scaricare le modifiche ormai
      unite (inclusa la tua), e cancelli il branch locale ormai concluso.
