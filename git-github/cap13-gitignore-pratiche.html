Capitolo 13

Ignorare file e buone pratiche

Sai versionare e collaborare. Questo capitolo è di «igiene»: piccole accortezze che fanno la differenza tra un repository pulito e professionale e uno pasticciato. Prima cosa: non tutti i file di un progetto vanno versionati. File temporanei, dipendenze scaricabili, e soprattutto segreti (password, chiavi) non devono finire nel repository — e Git offre un meccanismo apposta per escluderli, il file .gitignore. Poi le buone abitudini che rendono la storia una risorsa e non un caos: messaggi di commit efficaci e commit piccole e frequenti. Sono cose semplici, spesso trascurate, che i team apprezzano moltissimo. In questo capitolo: il .gitignore (§13.1), cosa non committare mai (§13.2), messaggi efficaci (§13.3) e commit ben dimensionate (§13.4). La differenza tra «so usare Git» e «uso Git bene» sta spesso in questi dettagli.

🗺 Dove siamo

Il file .gitignore (§13.1), cosa non mettere in un repo — segreti in primis (§13.2), messaggi di commit efficaci (§13.3), commit piccole e frequenti (§13.4). Nel laboratorio (§13.5) crei un .gitignore e fai pulizia.

🔑 Perché ci tocca: reputazione e sicurezza

Un repository ordinato, senza file spazzatura e senza segreti, è il biglietto da visita di uno sviluppatore serio. E committare un segreto per errore è uno degli incidenti di sicurezza più comuni (e imbarazzanti): una chiave API finita in un repo pubblico può essere sfruttata in minuti. Queste buone pratiche ti proteggono e migliorano il lavoro di tutto il team.

Sezione 13.1Il file .gitignore

Il .gitignore è un file di testo, nella radice del progetto, in cui elenchi i file e le cartelle che Git deve ignorare: non li traccerà, non li mostrerà come «untracked», non li includerà nelle commit.

.gitignore — un esempio
# ambienti e dipendenze (si ricreano, non si versionano)
.venv/
node_modules/

# file di configurazione con segreti
.env

# file temporanei e di sistema
*.log
.DS_Store
__pycache__/
La sintassi del .gitignore nomi · cartelle · jolly · commenti
.env — un nome
ignora il file con quel nome esatto, ovunque nel progetto.
node_modules/ — una cartella
la barra finale indica una cartella: ignora quella cartella e tutto il suo contenuto. Tipico per le dipendenze (enormi e scaricabili: non si versionano).
*.log — un jolly
l'asterisco è un carattere jolly: *.log = «tutti i file che finiscono in .log». Utile per intere categorie di file.
# commento
le righe che iniziano con # sono commenti, per organizzare il file. Il .gitignore stesso va committato (è utile a tutto il team).
Un dettaglio importante: il .gitignore funziona solo sui file non ancora tracciati. Se hai già committato un file per sbaglio, aggiungerlo al .gitignore non lo rimuove dalla storia — devi toglierlo esplicitamente (git rm --cached nomefile) e committare. Per questo conviene creare il .gitignore all'inizio del progetto. Esistono .gitignore pronti per ogni linguaggio (GitHub li propone quando crei un repo): un ottimo punto di partenza.
.gitignore: cosa traccia Git, cosa ignora modifica le regole, guarda l'effetto
.gitignore (modificabile)
i file del progetto

    Sezione 13.2Cosa non mettere mai in un repo

    Alcune categorie di file non vanno mai versionate. Riconoscerle è parte del mestiere; la più critica sono i segreti, per ragioni di sicurezza.

    Le categorie da escludere segreti · dipendenze · generati · temporanei
    🔴 segreti (la priorità assoluta)
    password, chiavi API, token, certificati, il file .env. Mai nel repository: una volta committati restano nella storia (Cap. 10), e su un repo pubblico chiunque li vede. Vanno in variabili d'ambiente o file esclusi dal .gitignore.
    dipendenze scaricabili
    node_modules/, .venv/: enormi e ricreabili da un file di elenco (requirements.txt, package.json). Versioni la ricetta, non gli ingredienti.
    file generati
    output di compilazione, build, cache (__pycache__/, dist/). Si rigenerano dal codice sorgente: versionarli è inutile e crea rumore.
    file di sistema/editor
    .DS_Store (macOS), file temporanei dell'editor. Non riguardano il progetto: solo confusione.
    ⚠ Se committi un segreto, consideralo compromesso

    Se una password o chiave finisce per errore in un repository (anche privato, anche se la rimuovi subito), la regola è: cambiala. Rimuoverla dalla storia è complicato e non garantisce che nessuno l'abbia già copiata (specie su un repo pubblico, indicizzato in secondi). Prevenire con il .gitignore è l'unica strategia affidabile. Questa è la trappola di sicurezza numero uno per chi inizia: prendi l'abitudine di non far mai entrare i segreti nel repo.

    Sezione 13.3Messaggi di commit efficaci

    Torniamo sui messaggi di commit (Cap. 4), ora con un occhio al team. Messaggi buoni rendono la storia una risorsa consultabile; messaggi pigri la rendono inutile per tutti.

    Regole per messaggi utili il riassunto del Cap. 4, + il contesto
    prima riga breve e chiara
    ~50 caratteri, all'imperativo presente («Aggiungi…», «Correggi…»), specifica. È il «titolo» della commit, quello che si legge nel git log e nelle PR.
    spiega il «perché», se serve
    per modifiche non ovvie, dopo una riga vuota aggiungi il contesto: perché hai fatto questa scelta, quale problema risolve. Il «cosa» si vede dal diff; il «perché» no — ed è spesso la parte più preziosa a distanza di tempo.
    una commit, un tema
    se un messaggio deve dire «faccio X e Y e Z», probabilmente andavano più commit (§13.4). Un messaggio chiaro nasce da una commit coerente.
    convenzioni di team
    alcuni team usano prefissi (fix:, feat:) o riferimenti a ticket. Adatta lo stile al progetto: la coerenza col team conta più della «regola perfetta».
    Perché insistere tanto? Perché i messaggi sono l'unico modo per capire una modifica senza rileggere tutto il codice. Quando cercherai «quando e perché è entrato questo bug» (con git log), ti ritroverai a benedire chi ha scritto messaggi chiari — o a maledire chi ha scritto «fix» venti volte di fila. Sii il primo tipo di collega.

    Sezione 13.4Commit piccole e frequenti

    Un'ultima abitudine che semplifica tutto: fare commit piccole (una per «unità logica» di lavoro) e frequenti, invece di poche commit gigantesche.

    Perché piccolo è meglio leggibilità · reversibilità · meno conflitti
    più leggibili
    una commit che fa una cosa è facile da capire (nel log, nelle review). Una che ne fa venti è un muro impenetrabile. Commit piccole = storia comprensibile.
    più facili da annullare
    se una modifica va tolta (revert, Cap. 6), meglio che sia isolata in una sua commit, non impastata con dieci altre cose che vuoi tenere.
    meno conflitti
    modifiche piccole e frequenti si fondono più facilmente (Cap. 9): meno superficie di sovrapposizione con il lavoro altrui.
    la regola pratica
    una commit per ogni «passo» completo e coerente: una funzionalità aggiunta, un bug corretto, un refactoring. Non a metà di un lavoro (rischi commit «rotte»), non alla fine di tutto (commit monstre).
    L'equilibrio con l'esperienza viene naturale: «abbastanza piccola da spiegarsi in una frase, abbastanza completa da funzionare». Insieme ai messaggi chiari (§13.3), le commit ben dimensionate rendono la storia del progetto un vero diario leggibile — che è tutto il punto del controllo di versione. Queste due abitudini, semplici, ti distingueranno subito come collaboratore attento.

    Sezione 13.5Laboratorio: pulizia e .gitignore

    Aggiungi un .gitignore a un progetto e verifica che i file da escludere spariscano dal radar di Git.

    terminale — creare e verificare un .gitignore
    # in un repository, crea alcuni file (uno da ignorare)
$ echo "CHIAVE=segreta123" > .env       # un segreto: NON deve entrare!
$ echo "log del programma" > app.log

# senza .gitignore, Git li vedrebbe come untracked:
$ git status
Untracked files: .env  app.log  ...

# crea il .gitignore per escluderli
$ echo ".env" > .gitignore
$ echo "*.log" >> .gitignore

# ora Git NON vede più .env e app.log (ma vede il .gitignore, da committare):
$ git status
Untracked files: .gitignore   (niente .env, niente app.log!)

$ git add .gitignore && git commit -m "Aggiungi .gitignore"
    ✅ Cosa hai ottenuto

    Hai protetto il progetto: .env (il segreto) e app.log (rumore) ora sono invisibili a Git — non rischi di committarli per sbaglio. Il .gitignore, invece, va committato perché serve a tutto il team. Nota l'ordine giusto: creare il .gitignore prima di committare i file da escludere. Con questa accortezza, più i messaggi chiari e le commit piccole, il tuo repository è pulito e professionale. Ora hai tutti gli strumenti e le buone pratiche: nel capitolo finale li mettiamo insieme in un flusso di lavoro completo.

    🔑 Quello che devi portarti dietro dal Capitolo 13
    1. .gitignore: elenca file/cartelle da ignorare (.env, node_modules/, *.log). Va committato; funziona solo sui file non ancora tracciati (§13.1).
    2. Non committare mai: segreti (priorità!), dipendenze scaricabili, file generati, file di sistema. Un segreto committato va considerato compromesso → cambialo (§13.2).
    3. Messaggi efficaci: prima riga breve/imperativa/specifica; il «perché» nel corpo se serve; una commit, un tema (§13.3).
    4. Commit piccole e frequenti: più leggibili, più facili da annullare, meno conflitti. «Piccola da spiegare in una frase, completa da funzionare» (§13.4).

    Hai tutti gli strumenti (commit, branch, merge, remoto, PR) e tutte le buone pratiche (igiene, messaggi, commit dimensionate). È il momento di metterli insieme. Nel Capitolo 14, il gran finale: percorriamo un flusso di lavoro completo dall'inizio alla fine — dal clone di un progetto alla pull request unita — come si fa nel lavoro reale, e vediamo dove andare per continuare a crescere con Git. Tutto ciò che hai imparato, in un unico percorso.

    Sezione 13.6Esercizi

    Esercizio 13.1

    A cosa serve il file .gitignore? Scrivi le righe per ignorare: il file .env, la cartella build/, e tutti i file .tmp.

    Esercizio 13.2

    Perché le dipendenze (come node_modules/ o .venv/) non si versionano? Cosa si versiona al loro posto?

    Esercizio 13.3

    Hai committato per errore un file .env con una password in un repository pubblico, e te ne accorgi subito. Basta rimuoverlo con una nuova commit? Qual è la cosa più importante da fare?

    Esercizio 13.4

    Elenca due vantaggi di fare commit piccole e frequenti invece di poche commit molto grandi.

    Sezione 13.7Soluzioni

    Soluzione 13.1

    Il .gitignore elenca i file e le cartelle che Git deve ignorare (non tracciare né committare).

    .gitignore
    .env
build/
*.tmp
    Soluzione 13.2

    Le dipendenze non si versionano perché sono enormi e ricreabili: si possono riscaricare/reinstallare in qualsiasi momento a partire da un file di elenco. Versionarle gonfierebbe il repository inutilmente. Al loro posto si versiona la «ricetta»: il file che elenca le dipendenze e le loro versioni (es. requirements.txt per Python, package.json per Node), da cui chiunque può ricreare l'ambiente.

    Soluzione 13.3

    No, non basta rimuoverlo con una nuova commit: la password resta comunque nella storia (nelle commit precedenti), e su un repo pubblico potrebbe essere già stata copiata o indicizzata. La cosa più importante è cambiare immediatamente la password (o rigenerare la chiave): va considerata compromessa. Poi si può ripulire la storia (operazione avanzata) e, soprattutto, aggiungere il file al .gitignore per evitare che ricapiti.

    Soluzione 13.4

    Due vantaggi: (1) sono più leggibili — una commit che fa una sola cosa è facile da capire nel log e nelle review, mentre una enorme è un muro; (2) sono più facili da annullare selettivamente (con revert) senza toccare altro lavoro. In più generano meno conflitti (minore sovrapposizione) e permettono messaggi di commit più chiari e specifici.
