Capitolo 7

I branch

Siamo al concetto che rende Git davvero potente: i branch (rami). Finora abbiamo lavorato su un'unica linea di sviluppo, aggiungendo commit una dopo l'altra. Ma immagina di voler provare una nuova funzionalità rischiosa, senza toccare la versione che funziona; o di lavorare in tre su cose diverse dello stesso progetto, senza pestarvi i piedi. La soluzione è creare linee di sviluppo parallele: branch, appunto. Ognuno è una «diramazione» indipendente dove puoi fare commit in totale isolamento; quando l'idea è pronta, la unisci al ramo principale (Cap. 8), altrimenti la butti via senza conseguenze. È il meccanismo su cui si regge tutto lo sviluppo moderno e la collaborazione. In questo capitolo: cos'è un branch (§7.1), come crearli e spostarsi (§7.2), cos'è HEAD (§7.3), e la potenza dell'isolamento (§7.4). I branch sono la porta della seconda metà di Git: sperimentare senza paura, collaborare senza caos.

🗺 Dove siamo

Cos'è un branch (§7.1), creare e spostarsi con git branch/git switch (§7.2), HEAD — dove sei adesso (§7.3), lavorare in isolamento (§7.4). Nel laboratorio (§7.5) crei il tuo primo branch.

🔑 Perché ci tocca: è come si lavora davvero

Nessun team serio lavora tutto su una sola linea: ogni funzionalità, ogni correzione nasce e cresce su un branch dedicato, e viene unita solo quando è pronta e verificata. Capire i branch è il passaggio da «uso Git per me» a «uso Git per collaborare», ed è la base delle pull request (Cap. 12). È forse il concetto più importante dell'intero libro dopo le tre aree.

Sezione 7.1Cos'è un branch e a cosa serve

Un branch è una linea di sviluppo indipendente: una sequenza di commit che può divergere da un'altra. Il branch principale, creato in automatico, si chiama di solito main.

📖 Un branch è un'etichetta mobile

Ecco il modello mentale corretto (e sorprendentemente semplice): un branch è solo un'etichetta (un puntatore) che indica una certa commit, e che avanza automaticamente ogni volta che fai una nuova commit. Quando crei un nuovo branch, crei una seconda etichetta sulla stessa commit; da lì, le due etichette possono avanzare indipendentemente — ognuna sulla propria sequenza di commit. Non si «copia» il progetto: si crea solo un nuovo puntatore, quindi i branch sono leggerissimi e istantanei da creare. È questa leggerezza che rende naturale crearne uno per ogni piccola cosa.

A cosa servono i branch isolamento · sperimentazione · collaborazione
sperimentare senza rischi
provi un'idea su un branch; se funziona, la unisci; se no, la butti — main resta sempre pulito e funzionante.
lavorare in parallelo
tu su un branch, un collega su un altro: nessuno disturba il lavoro dell'altro finché non decidete di unire.
organizzare per funzionalità
la pratica standard: un branch per ogni feature o correzione (es. login-google, fix-bug-carrello). La storia resta ordinata e comprensibile.

Sezione 7.2Creare e spostarsi tra branch

Due operazioni: creare un branch e spostarti su di esso. Git moderno le fa con git branch e git switch (comandi chiari e recenti).

terminale — creare e spostarsi
# elenca i branch (l'asterisco indica quello attuale)
$ git branch
* main

# crea un nuovo branch chiamato "nuova-funzione"
$ git branch nuova-funzione

# spostati su di esso
$ git switch nuova-funzione
Switched to branch 'nuova-funzione'

# scorciatoia: crea E spostati in un colpo solo
$ git switch -c altra-funzione
I comandi dei branch branch · switch · switch -c
git branch
senza argomenti, elenca i branch; l'asterisco * segna quello su cui sei ora. Con un nome (git branch nome), crea un nuovo branch (ma non ci si sposta).
git switch nome
ti sposta sul branch indicato: da quel momento le tue commit vanno su quel branch. La working directory cambia per riflettere quel branch.
git switch -c nome
«create and switch»: crea il branch e ci si sposta subito. La scorciatoia più usata quando inizi una nuova funzionalità.
nota storica: git checkout
prima esisteva git checkout per spostarsi tra branch (git checkout -b nome). Lo vedrai ancora spesso; switch è la versione moderna e più chiara, ma i due convivono.
La convenzione per i nomi dei branch: minuscolo, parole separate da trattini, descrittivo — login-utente, fix-crash-avvio. Un buon nome dice a cosa serve il branch, esattamente come un buon messaggio di commit. Prova il widget qui sotto per vedere un branch nascere e divergere.
Branch dal vivo: crea, spostati, committa guarda le linee divergere

Sezione 7.3HEAD: dove sei adesso

C'è una parola che compare spesso e vale la pena chiarire: HEAD. È il puntatore che indica dove ti trovi nella storia — di solito, la commit più recente del branch su cui sei.

HEAD, il «tu sei qui» il puntatore alla tua posizione
HEAD
indica la commit corrente, quella su cui «poggi». Normalmente punta al branch attivo, che a sua volta punta alla sua ultima commit. «Dove sei adesso» nella storia.
lo vedi in git log
l'annotazione (HEAD -> main) del Cap. 5 significa: «HEAD punta al branch main, che è su questa commit». Cambiando branch con switch, HEAD si sposta con te.
a cosa serve saperlo
molti comandi si riferiscono a HEAD: HEAD = commit attuale, HEAD~1 = quella precedente, e così via. È il punto di riferimento «da qui» per navigare la storia.
Non serve pensarci troppo all'inizio: ricorda solo che HEAD = «dove sono ora». Quando fai una commit, la nuova commit viene aggiunta «dopo» HEAD, e sia HEAD sia il branch avanzano su di essa. Quando cambi branch, HEAD ti segue. È il filo che tiene insieme «tu», «il branch attivo» e «la commit corrente».

Sezione 7.4Lavorare in isolamento

Ecco il beneficio concreto dei branch: quando lavori su un branch, le tue modifiche e commit non toccano gli altri branch. Ogni linea vive per conto suo, fino al momento dell'unione.

🔒 Ogni branch, un mondo a sé

Immagina di essere su un branch nuova-grafica e di fare tre commit sperimentali. Su main, nel frattempo, non è cambiato nulla: le tue commit vivono solo sul tuo branch. Se torni su main (con git switch main), i file tornano com'erano lì — la tua grafica sperimentale «sparisce» dalla vista (è al sicuro sull'altro branch, non persa). Puoi passare avanti e indietro tra branch, e ognuno mostra il suo stato. Questo isolamento è ciò che ti permette di: provare idee folli senza rischi; tenere main sempre stabile; lavorare in parallelo. Se l'idea è buona, la unisci (Cap. 8); se no, cancelli il branch e non è successo nulla.

⚠ Committa prima di cambiare branch

Una cautela pratica: prima di spostarti tra branch con switch, è buona norma aver committato (o messo da parte) le modifiche in corso. Se hai modifiche non salvate, Git potrebbe impedirti lo spostamento per non perderle (o le porterebbe con sé, creando confusione). Regola semplice: una commit pulita prima di cambiare branch. Se controlli git status (dovrebbe dire «working tree clean») prima di ogni switch, sei sempre al sicuro.

Sezione 7.5Laboratorio: il tuo primo branch

Crea un branch, fai una commit «di lato», e osserva come main resta intatto. È l'esperimento che fa «scattare» il concetto di branch.

terminale — un branch in isolamento
# parti da main, crea e spostati su un nuovo branch
$ git switch -c esperimento
Switched to a new branch 'esperimento'

# fai una modifica e committala: vive SOLO su questo branch
$ echo "idea sperimentale" > bozza.txt
$ git add . && git commit -m "Prova un'idea"

# torna su main: bozza.txt NON c'è! (è al sicuro su esperimento)
$ git switch main
$ ls
giorno1.txt  giorno2.txt          # bozza.txt non è qui

# ricontrolla: torna su esperimento e bozza.txt riappare
$ git switch esperimento
$ ls
bozza.txt  giorno1.txt  giorno2.txt
✅ Il momento «aha»

Hai appena visto la magia dei branch: bozza.txt esiste su esperimento ma non su main. Spostandoti tra i due, la working directory cambia per riflettere il branch attivo. Le due linee di sviluppo sono davvero indipendenti: puoi accumulare commit su esperimento senza toccare minimamente main. Se l'esperimento non ti convince, torni su main e cancelli il branch (git branch -d esperimento, Cap. 8): come se non fosse mai successo. Se invece è buono, lo unisci — ed è proprio ciò che impariamo nel prossimo capitolo.

🔑 Quello che devi portarti dietro dal Capitolo 7
  1. Un branch è una linea di sviluppo indipendente — tecnicamente un'etichetta mobile che punta a una commit e avanza. Leggerissimo da creare (§7.1).
  2. git branch elenca/crea; git switch nome ti sposta; git switch -c nome crea e sposta. (Vecchio: git checkout) (§7.2).
  3. HEAD = «dove sei ora» (la commit/branch corrente); ti segue quando cambi branch (§7.3).
  4. I branch danno isolamento: le commit di un branch non toccano gli altri; cambiando branch la working directory riflette quello attivo. Committa prima di fare switch (§7.4).

Sai creare linee di sviluppo parallele e lavorarci in isolamento. Ma un branch «di lato» ha senso solo se, a lavoro finito, puoi riportarne il risultato nel ramo principale. Come si uniscono due branch? Cosa succede alle commit dell'uno e dell'altro? Nel Capitolo 8 impariamo il merge: l'operazione che fonde una linea di sviluppo in un'altra, completando il ciclo «ramifica → lavora → unisci» che è il cuore del lavoro con Git.

Sezione 7.7Soluzioni

Soluzione 7.1

Un branch è una linea di sviluppo indipendente: una sequenza di commit che può divergere dalle altre. Tecnicamente è solo un'etichetta (un puntatore) che indica una commit e avanza automaticamente a ogni nuova commit. È «leggero» perché creando un branch non si copia il progetto né i file: si crea solo un nuovo puntatore su una commit esistente. Per questo è istantaneo e non occupa spazio.

Soluzione 7.2
terminale
$ git switch -c pagina-contatti

Il -c («create») crea il branch e ci si sposta subito. (Equivalente vecchio: git checkout -b pagina-contatti.)

Soluzione 7.3

No: la commit fatta su esperimento non appare su main. I branch sono linee di sviluppo indipendenti: le commit di un branch restano su quel branch finché non lo si unisce esplicitamente (merge, Cap. 8). Tornando su main, vedi solo le commit di main; la commit dell'esperimento è al sicuro sul suo branch, ma invisibile da main.

Soluzione 7.4

HEAD è il puntatore che indica dove ti trovi ora nella storia: normalmente la commit più recente del branch attivo («tu sei qui»). Quando esegui git switch un-altro-branch, HEAD si sposta a puntare a quel branch (e alla sua ultima commit): ti «segue» sul nuovo branch, e la working directory si aggiorna per rifletterlo.
