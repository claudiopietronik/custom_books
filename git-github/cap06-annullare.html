Capitolo 6

Annullare e correggere

Uno dei più grandi doni di Git è questo: sbagliare non fa più paura. Hai modificato un file e vuoi tornare com'era? Hai messo in staging la cosa sbagliata? Hai scritto un messaggio di commit con un errore? Hai fatto una commit che non doveva esistere? Per ognuno di questi casi Git ha una risposta pulita e sicura. La difficoltà, per i principianti, non è che manchino gli strumenti — è sapere quale usare, perché Git ne ha diversi che sembrano simili ma agiscono su aree diverse. In questo capitolo li mettiamo in ordine, guidati dalle tre aree del Cap. 3: scartare modifiche non salvate con restore (§6.1), togliere qualcosa dallo staging (§6.2), correggere l'ultima commit con amend (§6.3), e annullare in sicurezza una commit già condivisa con revert (§6.4). Alla fine avrai una «rete di sicurezza» mentale: per ogni errore, saprai esattamente quale comando ti salva.

🗺 Dove siamo

Scartare modifiche non salvate (restore, §6.1), togliere dallo staging (§6.2), correggere l'ultima commit (amend, §6.3), annullare una commit pubblica (revert, §6.4). Nel laboratorio (§6.5) rimedi ad alcuni errori tipici.

🔑 Perché ci tocca: la libertà di sperimentare

Sapere di poter tornare indietro cambia il modo in cui lavori: provi idee, fai modifiche audaci, senza il terrore di rompere tutto. La rete di sicurezza di Git è ciò che rende la programmazione meno stressante e più creativa. Ma va usata con criterio: alcuni comandi «riscrivono» la storia (pericolosi se il lavoro è già condiviso), altri no. Capire la differenza è la chiave — e questo capitolo te la dà.

Sezione 6.1Scartare modifiche non salvate (restore)

Il caso più semplice: hai modificato un file nella working directory, non ti piace, e vuoi riportarlo com'era all'ultima commit. Lo strumento è git restore.

terminale — scartare le modifiche di un file
# hai modificato saluto.py e vuoi annullare le modifiche NON committate
$ git restore saluto.py

# il file torna com'era all'ultima commit. La modifica è persa.
⚠ Questa modifica non torna indietro

git restore nomefile butta via le modifiche non salvate di quel file, riportandolo all'ultima versione committata. Attenzione: le modifiche scartate sono perse per sempre (non erano in nessuna commit, quindi Git non le può recuperare). Usalo solo quando sei sicuro di voler buttare via quel lavoro. È il rovescio della medaglia della sua semplicità: comodo, ma definitivo. Se hai dubbi, committa prima (o usa git stash, un «cassetto temporaneo» per mettere da parte modifiche — un comando avanzato che potrai approfondire).

Sezione 6.2Togliere dallo staging

Un altro caso comune: hai fatto git add di un file, ma ti accorgi che non volevi includerlo nella prossima commit. Vuoi toglierlo dallo staging senza perdere le modifiche.

terminale — togliere dallo staging (senza perdere il lavoro)
# hai fatto per sbaglio: git add config.txt
$ git restore --staged config.txt

# il file esce dallo staging ma le sue modifiche RESTANO nella working directory
restore vs restore --staged due azioni diverse, stessa parola
git restore nomefile
agisce sulla working directory: scarta le modifiche del file (le perde). §6.1.
git restore --staged nomefile
agisce sulla staging area: toglie il file dallo staging ma conserva le modifiche nella working directory. Non perdi nulla — annulli solo l'add.
ricorda le tre aree
la differenza è quale area tocchi: --staged = staging (annulla l'add); senza = working (scarta la modifica). Git te lo suggerisce sempre in git status, che indica il comando esatto per ogni situazione.
Non serve memorizzare la sintassi a memoria: quando fai git status, Git ti mostra il comando da usare sia per «togliere dallo staging» sia per «scartare le modifiche». È il vantaggio di controllare spesso lo stato: Git ti guida. La cosa da capire (e ricordare) è il concetto: c'è un modo per annullare l'add senza perdere il lavoro, e uno per buttare via davvero le modifiche.

Sezione 6.3Correggere l'ultima commit (amend)

Hai appena fatto una commit e ti accorgi di un errore: un refuso nel messaggio, o hai dimenticato di includere un file. git commit --amend ti lascia correggere l'ultima commit invece di farne una nuova.

terminale — correggere l'ultima commit
# CASO A: solo correggere il messaggio dell'ultima commit
$ git commit --amend -m "Messaggio corretto"

# CASO B: hai dimenticato un file? Aggiungilo e "fondilo" nell'ultima commit
$ git add file-dimenticato.txt
$ git commit --amend --no-edit   # --no-edit: tieni il messaggio com'è
⚠ amend riscrive la storia: solo su commit non condivise

--amend non «modifica» la commit: ne crea una nuova che la sostituisce (con un hash diverso). Questo si chiama riscrivere la storia. Va bene solo se la commit è ancora locale, non ancora condivisa (non «pushata» su GitHub, Cap. 10). Se la commit è già stata condivisa con altri, riscriverla crea disallineamenti e confusione per chi l'aveva già scaricata. La regola d'oro: amend liberamente le commit locali; mai quelle già pubblicate. Per quelle pubbliche c'è revert (§6.4).

Sezione 6.4Annullare una commit pubblica (revert)

E se una commit già condivisa ha introdotto un problema? Non puoi riscriverla (danneggeresti gli altri). La soluzione pulita è git revert: crea una nuova commit che annulla gli effetti di quella problematica, lasciando la storia intatta.

terminale — annullare in sicurezza una commit
# la commit a1b2c3d ha introdotto un bug. Annulliamone gli effetti:
$ git revert a1b2c3d

# Git crea una NUOVA commit che disfa le modifiche di a1b2c3d.
# La storia ora ha: ...a1b2c3d (il bug) ... + una commit "Revert" che lo annulla.
revert vs amend/reset annullare aggiungendo, non cancellando
git revert <hash>
crea una nuova commit che fa l'opposto di quella indicata, annullandone gli effetti. Non cancella nulla dalla storia: la aggiunge. Per questo è sicuro anche su commit pubbliche — non riscrive il passato, lo corregge andando avanti.
perché è la scelta giusta per il lavoro condiviso
chi ha già scaricato la storia riceverà semplicemente una commit in più (il revert), senza disallineamenti. Nessuno viene «tradito» da un passato cambiato.
e git reset?
esiste anche git reset, che «sposta indietro» il branch cancellando commit — potente ma pericoloso e riservato a commit locali (riscrive la storia). Per i principianti: usa revert per annullare commit, ed evita reset finché non ne capisci bene i rischi.
La distinzione fondamentale del capitolo: alcuni comandi riscrivono la storia (amend, reset), altri la estendono (revert). Riscrivere va bene solo in locale, prima di condividere; una volta che il lavoro è pubblico, si annulla aggiungendo (revert), non cancellando. Tieni questa regola come bussola: ti eviterà i guai più seri con Git. Il widget qui sotto ti aiuta a scegliere lo strumento giusto per ogni situazione.

Sezione 6.5Laboratorio: rimediare agli errori

Proviamo due dei rimedi più comuni sul repository diario: scartare una modifica e correggere il messaggio dell'ultima commit.

terminale — due rimedi in pratica
# --- Scartare una modifica non voluta ---
$ echo "riga sbagliata" >> giorno1.txt
$ git status          # mostra giorno1.txt come "modified"
$ git restore giorno1.txt   # torna com'era: la riga sparisce
$ git status          # "working tree clean" di nuovo

# --- Correggere il messaggio dell'ultima commit ---
$ git commit --amend -m "Aggiungi il primo giorno del diario"
$ git log --oneline   # il messaggio ora è quello corretto
✅ La tua rete di sicurezza

Hai visto due rimedi in azione: restore per buttare via una modifica non voluta, e --amend per correggere l'ultima commit locale. Ricorda la regola d'oro: correggi liberamente ciò che è ancora solo tuo (locale); per annullare qualcosa di già condiviso, usa revert (che aggiunge, non riscrive). Con questi strumenti, sbagliare non è più un dramma: c'è sempre una via d'uscita ordinata. E ricorda che git status ti suggerisce quasi sempre il comando giusto per la situazione.

🔑 Quello che devi portarti dietro dal Capitolo 6
  1. git restore nomefile: scarta le modifiche non committate di un file (le perde) (§6.1).
  2. git restore --staged nomefile: toglie dallo staging ma conserva le modifiche (annulla l'add) (§6.2).
  3. git commit --amend: corregge l'ultima commit (messaggio o file dimenticato). Riscrive la storia → solo su commit locali (§6.3).
  4. git revert <hash>: annulla una commit aggiungendone una opposta. Sicuro anche su commit pubbliche. Regola: riscrivi solo il locale, annulla il pubblico con revert (§6.4).

Hai una rete di sicurezza per correggere gli errori su una singola linea di sviluppo. Ma il vero superpotere di Git — quello che lo rende insostituibile per sperimentare e collaborare — sono i branch: linee di sviluppo parallele, dove provare idee in totale isolamento senza toccare il lavoro principale. Nel Capitolo 7 entriamo nel mondo dei branch: cosa sono, come si creano, come ci si sposta tra loro. È il concetto che apre la seconda metà di Git.

Sezione 6.6Esercizi

Esercizio 6.1

Esercizio 6.2

Hai appena committato (in locale, non pushato) e ti accorgi di aver dimenticato un file. Quali comandi usi per includerlo nella stessa commit, senza crearne una nuova?

Esercizio 6.3

Perché git commit --amend non va usato su una commit già condivisa con altri (già pushata)? Cosa si usa invece in quel caso?

Esercizio 6.4

Cosa fa git revert <hash>? Perché è «sicuro» annullare così una commit pubblica, mentre riscrivere la storia non lo è?

Sezione 6.7Soluzioni

Soluzione 6.1

git restore file.txt agisce sulla working directory: scarta le modifiche non committate del file, riportandolo all'ultima commit — perdi quelle modifiche. git restore --staged file.txt agisce sulla staging area: toglie il file dallo staging (annulla l'add) ma conserva le modifiche nella working directory — non perdi nulla. In sintesi: la versione senza --staged butta via le modifiche; quella con --staged annulla solo la preparazione.

Soluzione 6.2
terminale
$ git add file-dimenticato.txt
$ git commit --amend --no-edit

Aggiungi il file allo staging, poi --amend --no-edit lo «fonde» nell'ultima commit mantenendo il messaggio invariato.

Soluzione 6.3

Perché --amend riscrive la storia: crea una nuova commit (con hash diverso) che sostituisce l'ultima. Se quella commit era già stata condivisa (pushata), chi l'aveva scaricata si ritrova con una storia divergente, e nascono disallineamenti e conflitti fastidiosi. Su una commit già pubblica si usa invece git revert, che aggiunge una commit di annullamento senza toccare il passato.

Soluzione 6.4

git revert <hash> crea una nuova commit che fa l'opposto di quella indicata, annullandone gli effetti (senza cancellarla dalla storia). È «sicuro» su una commit pubblica perché non riscrive il passato: chi ha già scaricato la storia riceve semplicemente una commit in più (il revert), senza disallineamenti. Riscrivere la storia (amend/reset), invece, cambia commit già viste da altri, causando divergenze — per questo è sconsigliato su ciò che è già condiviso.
