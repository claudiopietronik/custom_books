Capitolo 9

Risolvere i conflitti

Finora i merge sono andati lisci, perché le modifiche dei due branch non si sovrapponevano. Ma cosa succede se due branch cambiano la stessa riga dello stesso file, in modi diversi? Git non può sapere quale versione sia quella «giusta» — sarebbe come chiedere a un correttore automatico di decidere tra due frasi opposte. Allora si ferma e chiede aiuto a te: nasce un conflitto di merge. Per molti principianti la parola «conflitto» suona come un disastro; in realtà è una situazione normale e gestibile, che si risolve con calma in pochi passi. Git ti segnala esattamente dove è il problema (con dei marcatori nel file), tu decidi quale versione tenere (o come combinarle), e concludi il merge. In questo capitolo: perché nascono i conflitti (§9.1), come leggere i marcatori (§9.2), come risolverli e completare il merge (§9.3), e come evitarli con buone abitudini (§9.4). Dopo questo capitolo, i conflitti smetteranno di farti paura: saranno solo un passo in più, ogni tanto.

🗺 Dove siamo

Perché nascono i conflitti (§9.1), leggere i marcatori (§9.2), risolvere e completare il merge (§9.3), evitarli con buone abitudini (§9.4). Nel laboratorio (§9.5) risolvi un conflitto passo per passo.

🔑 Perché ci tocca: è la paura numero uno dei principianti

«E se creo un conflitto?» è il timore che blocca chi inizia a collaborare. Ma un conflitto non rompe nulla: Git mette in pausa il merge e aspetta la tua decisione, senza toccare le tue commit. Imparare a risolverli — sono più semplici di quanto sembri — ti dà la tranquillità per collaborare davvero, che è il punto di tutto GitHub (prossimi capitoli).

Sezione 9.1Perché nascono i conflitti

Un conflitto nasce da una situazione precisa: due branch modificano la stessa parte dello stesso file, in modo incompatibile. Al momento del merge, Git non sa quale delle due versioni scegliere.

📖 Quando Git può decidere da solo, e quando no

Ricorda dal Cap. 8: nel merge, Git fonde le modifiche file per file, riga per riga. Se i due branch hanno toccato parti diverse (file diversi, o righe diverse dello stesso file), Git le combina automaticamente, senza problemi — è il caso normale. Il conflitto scatta solo quando entrambi hanno cambiato le stesse righe in modo diverso: lì Git non ha una regola per scegliere, quindi si ferma e chiede a te. In un progetto ben organizzato i conflitti sono rari (perché le persone lavorano su parti diverse), ma prima o poi capitano — ed è giusto saperli affrontare.

Sezione 9.2Leggere i marcatori di conflitto

Quando un merge produce un conflitto, Git modifica il file «conteso» inserendo dei marcatori speciali che mostrano entrambe le versioni, una sotto l'altra. Imparare a leggerli è metà del lavoro.

titolo.txt — come appare durante un conflitto
<<<<<<< HEAD
# Il mio diario di viaggio        <- la TUA versione (branch attuale)
=======
# Diario 2026                     <- l'ALTRA versione (branch che unisci)
>>>>>>> aggiungi-titolo
I tre marcatori <<<<<<< ======= >>>>>>>
<<<<<<< HEAD
segna l'inizio della tua versione (quella del branch su cui sei, HEAD). Tutto ciò che segue, fino a =======, è «ciò che avevi tu».
=======
il separatore: divide le due versioni. Sopra la tua, sotto quella dell'altro branch.
>>>>>>> nome-branch
segna la fine della versione dell'altro branch (quello che stai unendo). Tutto tra ======= e questo marcatore è «ciò che aveva l'altro».
cosa devi fare
decidere come deve restare quel pezzo: tenere la tua, tenere l'altra, o combinarle. Poi cancellare i marcatori e lasciare solo il testo finale corretto.
Il punto chiave: Git non ha «sbagliato», ti sta mostrando le due alternative perché tu scelga. Il file, con i marcatori, è temporaneamente «non valido» (contiene entrambe le versioni + i simboli): il tuo compito è renderlo di nuovo valido, decidendo cosa tenere. git status durante un conflitto ti elenca i file «both modified» che aspettano la tua decisione.

Sezione 9.3Risolvere e completare il merge

Risolvere un conflitto è un processo in tre passi semplici: modifica il file lasciando la versione giusta, aggiungi alla staging, committa per concludere il merge.

terminale — i tre passi per risolvere
# 1) apri il file nell'editor e SISTEMA il conflitto:
#    cancella i marcatori (<<<, ===, >>>) e lascia solo il testo finale che vuoi

# 2) segna il conflitto come risolto aggiungendo il file
$ git add titolo.txt

# 3) completa il merge con una commit (Git propone un messaggio adatto)
$ git commit
[main a9b8c7d] Merge branch 'aggiungi-titolo'
Il processo di risoluzione modifica → add → commit
1 · modifica il file
apri il file nell'editor, guarda le due versioni tra i marcatori, e lascia il testo finale corretto (la tua, l'altra, o una combinazione). Fondamentale: cancella tutti e tre i marcatori (<<<, ===, >>>). Ripeti per ogni conflitto nel file.
2 · git add nomefile
aggiungere il file segnala a Git che il conflitto è risolto. È il modo di dire «ho sistemato». Fallo per ogni file che era in conflitto.
3 · git commit
conclude il merge creando il merge commit (Cap. 8). Git propone già un messaggio adatto («Merge branch…»): di solito lo accetti. Fatto: il merge è completo, il conflitto risolto.
via d'uscita: git merge --abort
se ti sei confuso e vuoi annullare tutto e tornare a prima del merge: git merge --abort. Nessun danno: torni al punto di partenza e puoi riprovare con calma.
Nota che non c'è un comando magico che risolve il conflitto: la decisione è umana (solo tu sai cosa deve dire davvero quel codice). Git ti dà gli strumenti — ti mostra le alternative e attende — ma la scelta è tua. Ed è giusto così: un conflitto è, in fondo, una decisione di merito («quale versione è corretta?») che nessun automatismo può prendere al posto tuo. Prova a risolverne uno nel widget qui sotto.

Sezione 9.4Evitare i conflitti (buone abitudini)


  

    I conflitti si risolvono facilmente, ma è ancora meglio ridurne la frequenza. Poche abitudini, soprattutto
    quando lavori in team, li rendono rari.
  


  

    
Come tenerli al minimo commit piccole · sincronizza spesso · dividi il lavoro

    

      

        
commit piccole e frequenti

        
modifiche piccole si fondono più facilmente di enormi blocchi cambiati insieme. Meno superficie di
            sovrapposizione, meno conflitti (Cap. 13).

        
sincronizzati spesso

        
in team, aggiorna spesso il tuo branch con le ultime modifiche di main (con pull,
            Cap. 11): così i tuoi cambiamenti restano «vicini» a quelli degli altri, e le divergenze non si accumulano.

        
dividi bene il lavoro

        
se ognuno lavora su file o aree diverse del progetto, le sovrapposizioni sono naturalmente poche.
            Una buona organizzazione previene i conflitti alla radice.

        
branch di vita breve

        
un branch che vive giorni, non settimane, diverge meno da main. Finisci una funzionalità,
            uniscila, cancella il branch — e ricomincia (il flusso del Cap. 8).

      

    

    

      Il messaggio da portare a casa: i conflitti sono un sintomo normale del lavoro parallelo, non un errore.
      Puoi ridurli con le buone abitudini, ma non eliminarli del tutto — ed è ok, perché ora sai risolverli. Non aver
      paura di creare branch e fare merge per timore dei conflitti: sarebbe come non guidare per paura di parcheggiare.
    

  



  
  
Sezione 9.5Laboratorio: risolvi un conflitto


  

    Creiamo un conflitto di proposito (il modo migliore per imparare) e risolviamolo. Servono due branch che
    cambiano la stessa riga.
  


  

    
terminale — creare e risolvere un conflitto

    
# su main: crea un file e committa
$ echo "colore preferito: blu" > gusti.txt
$ git add . && git commit -m "Aggiungi i gusti"

# crea un branch e cambia la riga in modo diverso
$ git switch -c altro-colore
$ echo "colore preferito: verde" > gusti.txt
$ git add . && git commit -m "Cambia colore in verde"

# torna su main e cambia la STESSA riga in modo diverso ancora
$ git switch main
$ echo "colore preferito: rosso" > gusti.txt
$ git add . && git commit -m "Cambia colore in rosso"

# ora unisci: CONFLITTO! (entrambi hanno cambiato la stessa riga)
$ git merge altro-colore
CONFLICT (content): Merge conflict in gusti.txt
Automatic merge failed; fix conflicts and then commit the result.

# apri gusti.txt, scegli la versione giusta, cancella i marcatori, poi:
$ git add gusti.txt
$ git commit          # completa il merge

      seguire. Se durante la risoluzione ti fossi confuso, git merge --abort ti avrebbe riportato al

      ramificare, unire e risolvere conflitti — tutto in locale, sul tuo computer. È il momento di aprirsi al mondo:
      dal prossimo capitolo entra in scena GitHub.

Sezione 9.7Soluzioni


  

    
Soluzione 9.1

    

      Git genera un conflitto quando i due branch hanno modificato le stesse righe dello stesso file in modo
      diverso: non ha una regola per decidere quale versione tenere, quindi si ferma e chiede a te. Riesce invece a
      fondere automaticamente quando le modifiche riguardano parti diverse (file diversi, o righe diverse
      dello stesso file): non c'è sovrapposizione, quindi Git le combina senza ambiguità.
    

  


  

    
Soluzione 9.2

    

      La parte tra <<<<<<< HEAD e ======= è la tua versione: quella del
      branch su cui ti trovi (HEAD). La parte tra ======= e
      >>>>>>> nome-branch è la versione dell'altro branch, quello che stai unendo. Git
      te le mostra entrambe perché tu scelga cosa tenere.
    

  


  

    
Soluzione 9.3

    

      I tre passi: 1) modifica il file scegliendo il testo finale (la tua versione, l'altra, o una
      combinazione); 2) git add nomefile per segnare il conflitto come risolto; 3)
      git commit per completare il merge. Al file, prima di aggiungerlo, non devi dimenticare di
      cancellare tutti i marcatori (<<<<<<<, =======,
      >>>>>>>): devono restare solo le righe di testo valide.
    

  


  

    
Soluzione 9.4

    

      git merge --abort: annulla il merge in corso e riporta il repository allo stato precedente
      all'inizio del merge, come se non l'avessi mai avviato. Nessuna commit viene persa: puoi riprovare con calma. È
      la «via d'uscita» sicura quando un merge conflittuale ti ha confuso.
