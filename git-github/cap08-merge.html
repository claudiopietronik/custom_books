Capitolo 8

Unire i branch: il merge

Un branch «di lato» ha senso solo se, a lavoro finito, puoi riportarne il risultato nel ramo principale. Questa operazione — fondere una linea di sviluppo in un'altra — si chiama merge, ed è il gesto che chiude il ciclo fondamentale di Git: ramifica → lavora → unisci. Il merge prende le commit fatte su un branch e le integra in un altro, così le tue modifiche sperimentali diventano parte del progetto ufficiale. Git gestisce due situazioni: quando l'unione è banale (il fast-forward) e quando i due branch sono davvero divergenti (il merge commit). Capire la differenza ti fa leggere la storia con occhi nuovi. In questo capitolo: riportare il lavoro su main (§8.1), il fast-forward (§8.2), il merge commit (§8.3) e come cancellare un branch ormai unito (§8.4). Completiamo così il ciclo branch-e-merge, il modo in cui si lavora davvero — da soli e, presto, in team.

🗺 Dove siamo

Riportare il lavoro sul branch principale con git merge (§8.1), il fast-forward (§8.2), il merge commit (§8.3), cancellare un branch finito (§8.4). Nel laboratorio (§8.5) crei e unisci un branch.

🔑 Perché ci tocca: chiude il ciclo di lavoro

Creare branch senza saperli unire sarebbe metà del lavoro. Il merge è ciò che rende utili i branch: trasforma un esperimento isolato in un miglioramento del progetto. Ed è l'operazione dietro le pull request (Cap. 12), il cuore della collaborazione. Capirla bene — inclusi i suoi due «modi» — ti rende padrone del flusso di lavoro completo.

Sezione 8.1Riportare il lavoro sul branch principale

Il merge ha una regola di direzione da tenere a mente: ti sposti sul branch che vuoi far crescere (di solito main) e da lì «tiri dentro» l'altro branch. In altre parole: «unisci verso dove sei».

terminale — unire feature dentro main
# 1) spostati sul branch che riceve (la destinazione)
$ git switch main

# 2) unisci l'altro branch dentro quello attuale
$ git merge nuova-funzione
Updating a1b2c3d..e4f5g6h
Fast-forward
 funzione.txt | 3 +++
La direzione del merge vai sulla destinazione, poi merge
1 · git switch main
vai sul branch destinazione, quello che vuoi arricchire con il lavoro dell'altro. Il merge aggiorna il branch su cui ti trovi.
2 · git merge nuova-funzione
«porta le commit di nuova-funzione dentro il branch attuale (main)». Dopo questo, main contiene anche il lavoro fatto sull'altro branch.
l'errore da evitare
invertire la direzione: fare merge main mentre sei su nuova-funzione porterebbe main dentro la funzione, non il contrario. Ricorda: prima vai dove vuoi ricevere, poi unisci l'altro.
Il risultato del merge dipende da come sono messi i due branch. Se main non è avanzato da quando hai creato il branch, Git fa un fast-forward (§8.2); se invece entrambi hanno nuove commit, crea un merge commit (§8.3). Non devi scegliere tu: Git capisce da solo quale caso è. Ma conoscere i due ti fa leggere meglio la storia.

Sezione 8.2Il fast-forward

Il caso più semplice. Se dopo aver creato il branch non hai più fatto commit su main, allora main è «rimasto indietro» e il branch è semplicemente più avanti sulla stessa linea. Unire significa solo far avanzare l'etichetta main fino al branch.

⏩ Fast-forward: spostare l'etichetta in avanti

Ricordi che un branch è un'etichetta mobile (Cap. 7)? Nel fast-forward, poiché main non ha commit «sue» dopo la biforcazione, non c'è nulla da «fondere» davvero: Git fa semplicemente scivolare l'etichetta main in avanti fino a raggiungere l'ultima commit del branch. La storia resta una linea dritta, come se avessi fatto quelle commit direttamente su main. Il messaggio di Git lo dice esplicitamente: «Fast-forward». È il caso tipico quando lavori da solo e nessun altro ha toccato main nel frattempo. Semplice, pulito, nessun «nodo» nella storia.

Sezione 8.3Il merge commit

Il caso più interessante. Se entrambi i branch hanno accumulato nuove commit da quando si sono separati (tu sul tuo branch, qualcun altro — o tu stesso — su main), le due linee sono davvero divergenti. Git le fonde creando una commit speciale: il merge commit.

Il merge commit una commit con due «genitori»
cos'è
una nuova commit che unisce le due linee divergenti. Ha due commit «genitori» (una per branch), invece di una sola: è il «nodo» dove i due rami si ricongiungono.
cosa contiene
il risultato della combinazione delle modifiche di entrambi i branch. Git le fonde automaticamente file per file. Se le modifiche non si sovrappongono, tutto è automatico; se invece toccano le stesse righe, nasce un conflitto (Cap. 9!).
nella storia
con git log --graph vedi le due linee che si separano e poi si ricongiungono al merge commit. La storia «ramifica e si riunisce» — la forma tipica dei progetti collaborativi.
il messaggio
Git propone un messaggio automatico («Merge branch 'nuova-funzione'»); di solito lo accetti così com'è.
La differenza in una frase: fast-forward = «main scivola in avanti», nessuna commit nuova; merge commit = «le due linee si fondono» in una commit apposita con due genitori. Entrambi portano il lavoro del branch in main; cambia solo la forma della storia. Prova il widget qui sotto per vedere i due casi.

Sezione 8.4Cancellare un branch finito

Una volta che un branch è stato unito, il suo scopo è esaurito: le sue commit ora vivono anche in main. È buona pratica cancellarlo per tenere la lista dei branch pulita.

terminale — cancellare un branch unito
# dopo aver fatto il merge in main:
$ git branch -d nuova-funzione
Deleted branch nuova-funzione (was e4f5g6h).
Cancellare un branch -d (sicuro) vs -D (forzato)
git branch -d nome
cancella il branch, ma solo se è già stato unito. È la versione sicura: se il branch ha commit non ancora unite (che andrebbero perse), Git si rifiuta e ti avvisa.
git branch -D nome (D maiuscola)
cancella comunque, anche se non unito. Da usare con cautela: butti via le commit di quel branch. Serve quando vuoi davvero abbandonare un esperimento fallito.
cosa non cancelli
cancellare un branch non cancella le commit già unite in altri branch: quelle restano in main. Butti via solo l'etichetta (e, con -D, le eventuali commit non unite). I file uniti sono al sicuro.
Il flusso completo, che ripeterai in continuazione: switch -c feature → lavori e committi → switch main → merge feature → branch -d feature. Ramifica, lavora, unisci, pulisci. È il ritmo dello sviluppo con Git — e, con GitHub, diventerà il flusso a pull request (Cap. 12).

Sezione 8.5Laboratorio: crea e unisci un branch

Facciamo il ciclo completo: crea un branch, lavoraci, uniscilo in main, cancellalo. È il flusso che userai ogni giorno.

terminale — il ciclo branch-e-merge completo
# 1) crea un branch per una nuova funzionalità e spostati
$ git switch -c aggiungi-titolo

# 2) lavora e committa (sul branch)
$ echo "# Il mio diario" > titolo.txt
$ git add . && git commit -m "Aggiungi il titolo del diario"

# 3) torna su main e unisci
$ git switch main
$ git merge aggiungi-titolo
Fast-forward ...          # titolo.txt ora è in main!

# 4) pulisci: cancella il branch ormai unito
$ git branch -d aggiungi-titolo

# 5) verifica la storia
$ git log --oneline --graph
✅ Il ciclo che padroneggi ora

Hai completato il ciclo ramifica → lavora → unisci → pulisci: creato un branch, fatto una commit lì, unita in main, cancellato il branch. Questo è il flusso di lavoro fondamentale di Git, lo stesso che usano i team di tutto il mondo (con qualche passo in più per la collaborazione, che vedremo). Prova a rifarlo creando anche una commit su main prima del merge: otterrai un merge commit invece di un fast-forward — guardalo con git log --graph. Sperimenta: ora hai la rete di sicurezza del Cap. 6 se qualcosa va storto.

🔑 Quello che devi portarti dietro dal Capitolo 8
  1. Il merge unisce un branch in un altro. Regola di direzione: vai sulla destinazione (git switch main), poi git merge nome-branch (§8.1).
  2. Fast-forward: se main non è avanzato, l'etichetta main scivola in avanti; storia dritta, nessuna commit nuova (§8.2).
  3. Merge commit: se entrambi i branch hanno nuove commit, Git crea una commit con due genitori che fonde le linee. Se toccano le stesse righe → conflitto (Cap. 9) (§8.3).
  4. Dopo l'unione, git branch -d nome cancella il branch (sicuro; -D forza). Butti l'etichetta, non le commit già unite (§8.4).

Sai unire branch — e finora è andato tutto liscio, perché le modifiche non si sovrapponevano. Ma cosa succede quando due branch cambiano le stesse righe dello stesso file in modi diversi? Git non può decidere da solo quale tenere: nasce un conflitto. Sembra spaventoso, ma è normale e si risolve con calma. Nel Capitolo 9 impariamo a leggere e risolvere i conflitti di merge — l'ultima abilità «di base» di Git prima di passare a GitHub e alla collaborazione vera.

Sezione 8.7Soluzioni

Soluzione 8.1
terminale
$ git switch main
$ git merge login

Devi trovarti sul branch destinazione (main) prima del merge, perché il merge aggiorna il branch su cui sei. Poi «tiri dentro» login.

Soluzione 8.2

Il fast-forward avviene quando il branch destinazione (es. main) non ha nuove commit dopo la biforcazione: Git fa semplicemente avanzare la sua etichetta fino al branch, senza creare commit; la storia resta una linea dritta. Il merge commit avviene quando entrambi i branch hanno accumulato nuove commit (sono davvero divergenti): Git crea una commit apposita che fonde le due linee.

Soluzione 8.3

Due genitori. Un merge commit rappresenta il punto in cui due linee di sviluppo si ricongiungono: per questo ha due commit precedenti (una per ciascun branch unito), a differenza di una commit normale che ne ha una sola. È proprio ciò che «segna» nella storia il ricongiungimento dei due rami.

Soluzione 8.4

git branch -d è la versione sicura: cancella il branch solo se è già stato unito (altrimenti si rifiuta, per non farti perdere commit). git branch -D (D maiuscola) forza la cancellazione anche se il branch non è unito. Cancellare un branch già unito non fa perdere le sue commit: quelle vivono ora anche in main; si elimina solo l'etichetta del branch, non il lavoro integrato.
