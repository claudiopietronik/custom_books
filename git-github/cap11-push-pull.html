Capitolo 11

Collaborare: push, pull, fetch

Il tuo repository è su GitHub: ora impariamo a sincronizzarlo, cioè a scambiare commit tra la tua copia locale e quella remota. Sono due movimenti opposti: push per mandare le tue commit al remoto («salire»), pull per ricevere quelle nuove dal remoto («scendere»). Con questi due comandi, più persone possono lavorare sullo stesso progetto: ciascuno fa commit in locale, le pusha su GitHub, e pulla il lavoro degli altri. Vedremo anche la differenza tra pull e fetch (un dettaglio che chiarisce come Git ragiona) e le abitudini per restare allineati in team senza pestarsi i piedi. In questo capitolo: git push (§11.1), git pull (§11.2), fetch vs pull (§11.3), e la sincronizzazione in team (§11.4). È il ritmo quotidiano del lavoro collaborativo: committa, pusha, pulla, ripeti.

🗺 Dove siamo

git push per mandare (§11.1), git pull per ricevere (§11.2), fetch vs pull (§11.3), restare sincronizzati in team (§11.4). Nel laboratorio (§11.5) simuli un flusso di sincronizzazione.

🔑 Perché ci tocca: è la collaborazione, in due comandi

Push e pull sono il battito del lavoro di squadra: senza di essi, il repository remoto sarebbe solo un backup statico. Con essi, il codice fluisce tra te, il team e il progetto ufficiale. Capire quando pushare e pullare — e cosa succede quando anche altri lo fanno — è la base per collaborare senza confusione (e per le pull request del prossimo capitolo).

Sezione 11.1git push: mandare le commit al remoto

git push invia le commit che hai fatto in locale al repository remoto (GitHub), aggiornandolo. È il passo che rende visibile agli altri (e mette al sicuro) il tuo lavoro.

terminale — mandare le commit su GitHub
# hai committato in locale; ora manda le commit a origin
$ git push
... To https://github.com/anna/diario.git
   a1b2c3d..e4f5g6h  main -> main
git push smontato manda ciò che è solo in locale
git push
manda al remoto le commit che hai in locale ma non ancora sul remoto. Solo le novità: Git sa cosa manca «lassù» e invia quello. Dopo il primo push -u (Cap. 10), basta git push.
la forma esplicita git push origin main
«manda il branch main al remoto origin». Utile per essere precisi o quando pushi un branch nuovo per la prima volta.
cosa succede
il remoto avanza: le tue commit ora sono su GitHub, visibili nella pagina del repo e agli altri collaboratori. Il tuo lavoro è condiviso e al sicuro.
quando pushare
tipicamente dopo aver completato un pezzo di lavoro (una o più commit coerenti). Non serve pushare a ogni singola commit, ma farlo spesso è una buona abitudine (backup + gli altri vedono i progressi).
Un dettaglio importante: puoi pushare solo se il tuo locale è «aggiornato» rispetto al remoto. Se nel frattempo qualcun altro ha pushato commit che tu non hai, Git rifiuta il tuo push e ti chiede di fare prima git pull (§11.2) per allinearti. È una protezione: evita di sovrascrivere il lavoro altrui. Vediamo appunto il pull.

Sezione 11.2git pull: ricevere le modifiche

git pull fa l'opposto di push: scarica dal remoto le commit nuove (fatte da te su un'altra macchina, o dai collaboratori) e le integra nel tuo repository locale.

terminale — ricevere le novità dal remoto
$ git pull
Updating a1b2c3d..f9e8d7c
Fast-forward
 collaboratore.txt | 5 +++++
 1 file changed, 5 insertions(+)
git pull smontato scarica + integra
git pull
scarica le commit nuove dal remoto e le fonde nel tuo branch locale, in un colpo solo. Il tuo locale si allinea al remoto: hai anche le commit degli altri.
pull = fetch + merge
«dietro le quinte» pull fa due cose: scarica (fetch) le commit remote e le unisce (merge) al tuo lavoro. Per questo può generare un conflitto (Cap. 9!), se tu e un altro avete cambiato le stesse righe.
quando pullare
all'inizio di una sessione di lavoro (per partire dall'ultima versione) e prima di pushare (per allinearti). Pulla spesso: meno divergenza, meno conflitti.
il ciclo collaborativo
pull (parti aggiornato) → lavora e committa in locale → pull di nuovo (allinea) → push (condividi). Questo ritmo tiene tutti sincronizzati.
Il caso classico da tenere a mente: provi a push, Git lo rifiuta («updates were rejected») perché il remoto è più avanti; fai git pull (che scarica le novità e magari ti fa risolvere un conflitto), poi git push di nuovo — e stavolta va. «Push rifiutato → pull → push» è una sequenza che incontrerai spesso: ora sai perché e cosa fare.
Sincronizzazione: push e pull tra te e il team le commit salgono e scendono
💻 tu (locale)
⬆ push
⬇ pull
☁️ GitHub (origin)

Sezione 11.3fetch vs pull

C'è un cugino di pull che vale la pena conoscere: git fetch. Fa metà del lavoro di pull — scarica, ma non fonde — ed è utile quando vuoi guardare le novità prima di integrarle.

fetch vs pull scarica-e-basta vs scarica-e-unisci
git fetch
scarica le commit nuove dal remoto senza toccare il tuo lavoro. Le mette «da parte» (in riferimenti come origin/main) così puoi ispezionarle (con git log origin/main, git diff) prima di decidere.
git pull
= git fetch + git merge: scarica e integra subito nel tuo branch. Più diretto, ma «impegna» subito le modifiche remote (con possibile conflitto).
quando usare quale
pull nell'uso quotidiano, quando vuoi semplicemente allinearti. fetch quando vuoi prima vedere cosa è cambiato (utile in progetti attivi, o se sei prudente). fetch è sempre «sicuro»: non modifica il tuo lavoro, solo aggiorna la conoscenza del remoto.
Un modo di pensarci: fetch = «vai a vedere la posta ma non aprirla»; pull = «vai a prendere la posta e sistemala subito». All'inizio userai quasi sempre pull (più semplice); fetch diventa utile quando vuoi controllo e prudenza. Sapere che pull = fetch + merge ti aiuta anche a capire perché un pull può generare un conflitto: la parte «merge».

Sezione 11.4Restare sincronizzati in team

Quando più persone lavorano sullo stesso repository, la chiave è sincronizzarsi spesso. Poche abitudini rendono la collaborazione fluida ed evitano il caos (e i conflitti).

Le abitudini del lavoro in team pull spesso · push spesso · branch propri
pull all'inizio, pull prima di push
comincia ogni sessione con git pull (parti dall'ultima versione); fai pull anche prima di ogni push (per allinearti ed evitare il rifiuto). Meno sorprese.
push frequenti
condividi il tuo lavoro spesso, così gli altri vedono i progressi e le divergenze restano piccole. Un lavoro tenuto in locale per settimane è una bomba di conflitti.
lavora su branch tuoi
ognuno la sua feature sul suo branch (Cap. 7), poi unione controllata. Non tutti su main insieme: sarebbe un campo minato. È il tema delle pull request (Cap. 12).
comunica
Git è uno strumento, non un sostituto della comunicazione. Dirsi «sto lavorando su questo file» previene sovrapposizioni meglio di qualsiasi comando.
Il messaggio di fondo: la collaborazione con Git è fatta di piccoli scambi frequenti, non di grandi travasi occasionali. Pull spesso, push spesso, branch separati, e comunicazione. Con queste abitudini, lavorare in dieci sullo stesso progetto diventa naturale — ed è esattamente ciò che il flusso a pull request del prossimo capitolo formalizza e rende ordinato.

Sezione 11.5Laboratorio: sincronizzare

Simuliamo il ciclo collaborativo completo sul tuo repo diario già su GitHub (Cap. 10). Anche da solo, puoi esercitare push e pull (immagina che il «collaboratore» sia te stesso da un'altra macchina).

terminale — il ciclo pull → lavora → push
# 1) parti aggiornato: scarica eventuali novità dal remoto
$ git pull
Already up to date.          # (se non ci sono novità)

# 2) lavora e committa in locale (il ciclo del Cap. 4)
$ echo "Giorno 4: ho imparato push e pull." > giorno4.txt
$ git add . && git commit -m "Aggiungi il quarto giorno"

# 3) allinea di nuovo (buona abitudine) e condividi
$ git pull          # nel caso siano arrivate novità nel frattempo
$ git push          # manda la tua commit su GitHub

# ricarica GitHub: il quarto giorno è online!
✅ Il ritmo che hai imparato

Hai praticato il ciclo collaborativo: pull (parti aggiornato) → lavora e committa → pull (allinea) → push (condividi). Questo ritmo, ripetuto, è la sincronizzazione in team. Se durante un pull comparisse un conflitto, ora sai risolverlo (Cap. 9). E se un push venisse rifiutato, sai che basta un pull prima. Hai tutti gli strumenti per lavorare con altri su un progetto condiviso. Manca solo un pezzo — il più «sociale» — per rendere la collaborazione ordinata e revisionata: le pull request.

🔑 Quello che devi portarti dietro dal Capitolo 11
  1. git push manda le tue commit locali al remoto (origin). Puoi pushare solo se sei allineato: altrimenti Git rifiuta e chiede un pull (§11.1).
  2. git pull scarica e integra le commit nuove dal remoto (= fetch + merge). Può generare un conflitto (§11.2).
  3. git fetch scarica senza integrare (sicuro, per «guardare prima»); pull = fetch + merge (§11.3).
  4. In team: pull spesso (all'inizio e prima di push), push spesso, branch propri, comunica. Scambi piccoli e frequenti (§11.4).

Sai scambiare commit con il remoto. Ma pushare direttamente su main in un team non è il modo migliore: manca un momento di revisione, e il rischio di rompere il ramo principale è alto. Il flusso professionale usa le pull request: proponi le tue modifiche su un branch, i colleghi le revisionano e discutono, e solo dopo si uniscono. Nel Capitolo 12 impariamo questo flusso — il cuore della collaborazione moderna su GitHub — con la code review che rende il codice migliore e condiviso.

Sezione 11.6Esercizi

Esercizio 11.1

In una frase ciascuno, cosa fanno git push e git pull? In che «direzione» va ognuno (dal locale al remoto o viceversa)?

Esercizio 11.2

Provi a fare git push e Git lo rifiuta dicendo che il remoto ha commit che tu non hai. Cosa fai per risolvere e riuscire a pushare?

Esercizio 11.3

Qual è la differenza tra git fetch e git pull? A cosa equivale git pull «sotto il cofano»?

Esercizio 11.4

Perché, lavorando in team, è consigliato fare git pull prima di git push? Cosa si rischia a tenere il proprio lavoro in locale per settimane senza sincronizzare?

Sezione 11.7Soluzioni

Soluzione 11.1

git push manda le tue commit locali al repository remoto (direzione: dal locale al remoto, «salgono»). git pull scarica e integra le commit nuove dal remoto nel tuo repository locale (direzione: dal remoto al locale, «scendono»). In breve: push condivide il tuo lavoro, pull ti aggiorna con quello degli altri.

Soluzione 11.2

Il push è rifiutato perché il remoto è «più avanti» del tuo locale (qualcuno ha pushato commit che tu non hai). La soluzione: fare prima git pull per scaricare e integrare quelle commit (risolvendo un eventuale conflitto), così il tuo locale si allinea al remoto; poi git push di nuovo, che stavolta va a buon fine. La sequenza è: push rifiutato → pull → push.

Soluzione 11.3

git fetch scarica le commit nuove dal remoto ma non le integra nel tuo branch (le mette «da parte» per ispezionarle): è sicuro, non tocca il tuo lavoro. git pull scarica e integra subito. «Sotto il cofano», git pull equivale a git fetch + git merge: prima scarica, poi fonde le commit remote nel tuo branch (motivo per cui può generare un conflitto).

Soluzione 11.4

Fare pull prima di push ti fa allineare il locale con le eventuali commit nuove del remoto: così il push non viene rifiutato e integri per tempo il lavoro altrui (risolvendo subito piccoli conflitti). Tenere il proprio lavoro in locale per settimane senza sincronizzare fa divergere molto la tua copia da quella condivisa: quando finalmente sincronizzi, rischi conflitti grandi e numerosi, difficili da risolvere. Meglio scambi piccoli e frequenti.
