Capitolo 17

Reverse proxy e HTTPS: nginx e Caddy

Il nostro backend, avviato con Compose, funziona — ma esporlo direttamente a Internet sarebbe un errore. Serve un «portiere» davanti all'applicazione: un programma robusto che riceve tutte le connessioni dal mondo, le smista all'app, gestisce l'HTTPS (il lucchetto verde), serve i file statici e protegge ciò che sta dietro. Questo portiere è, tradizionalmente, nginx: un web server velocissimo, usato come reverse proxy. Lo hai già incontrato nel libro «Reti» come server web; qui lo vedi nel suo ruolo più prezioso per il backend. Ma c'è un'alternativa moderna che vale assolutamente la pena conoscere, Caddy, che fa la stessa cosa con l'HTTPS automatico e una configurazione di poche righe. In questo capitolo capisci cos'è un reverse proxy (§17.1), come mettere nginx davanti a FastAPI (§17.2), come servire i file statici (§17.3), come attivare l'HTTPS gratuito con Let's Encrypt (§17.4) e come Caddy semplifica tutto (§17.5). È il penultimo pezzo: dopo questo, il nostro backend sarà pronto per affacciarsi su Internet in modo sicuro.

🗺 Dove siamo

Cos'è un reverse proxy (§17.1), nginx davanti a FastAPI (§17.2), servire i file statici (§17.3), l'HTTPS con Let's Encrypt (§17.4), Caddy e l'HTTPS automatico (§17.5). Nel laboratorio (§17.6) configuri nginx con un certificato.

🔑 Perché ci tocca: la porta d'ingresso sicura

Nessun backend serio è esposto «nudo» a Internet: davanti c'è sempre un reverse proxy come nginx, che fa da scudo e da smistatore. È lì che si attiva l'HTTPS — oggi obbligatorio (i browser marchiano come «non sicuro» i siti senza) — e si gestiscono connessioni, file statici, e la sicurezza di base. Capire questo strato è indispensabile per portare online un'applicazione in modo professionale.

Sezione 17.1Cos'è un reverse proxy

Un reverse proxy è un server che sta davanti alle tue applicazioni e fa da intermediario: riceve tutte le richieste dai client e le inoltra al servizio giusto dietro di lui, poi rimanda indietro la risposta. Il client parla solo col proxy; non sa (e non deve sapere) cosa c'è dietro.

Cosa fa il portiere i compiti del reverse proxy
punto d'ingresso unico
tutte le richieste arrivano a nginx (sulle porte 80/443), che le smista. L'app FastAPI non è mai esposta direttamente: sta «al sicuro» dietro il proxy.
gestisce l'HTTPS (TLS)
nginx si occupa della cifratura (il lucchetto): decifra le richieste in arrivo e cifra le risposte, così la tua app può parlare in HTTP semplice dietro le quinte. Si dice «terminazione TLS».
smista per percorso o dominio
può mandare /api/… all'app FastAPI, /static/… ai file, domini diversi ad app diverse. Un solo ingresso, tante destinazioni.
protegge e ottimizza
serve i file statici velocemente, può fare cache, limitare gli abusi, bilanciare il carico su più copie dell'app (Cap. 15). Fa lo «sporco lavoro» di rete, sollevando l'app.
«Reverse» (inverso) perché è l'opposto di un proxy classico: un proxy normale sta davanti ai client (li rappresenta verso Internet); un reverse proxy sta davanti ai server (li rappresenta verso i client). Per il client, nginx è il tuo sito; l'app dietro è invisibile. Questo isolamento è ottimo per sicurezza e flessibilità.

Sezione 17.2nginx davanti a FastAPI

La configurazione di base di nginx come reverse proxy per la nostra app è breve: «tutto ciò che arriva, passalo all'app FastAPI». La direttiva chiave è proxy_pass.

nginx.conf (estratto) — reverse proxy verso FastAPI
server {
    listen 80;                        # ascolta sulla porta HTTP
    server_name mio-sito.com;

    location / {                     # tutte le richieste...
        proxy_pass http://api:80;      # ...inoltrale all'app FastAPI
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
La configurazione smontata server · listen · location · proxy_pass
server { ... }
un «sito» servito da nginx. Puoi averne più d'uno (per domini diversi).
listen 80; · server_name
su quale porta ascoltare e per quale dominio. La 80 è HTTP; la 443 sarà HTTPS (§17.4).
location / { ... }
una regola per un gruppo di percorsi. / = «tutto». Puoi avere più location per smistare percorsi diversi (§17.3).
proxy_pass http://api:80;
il cuore del reverse proxy: «inoltra questa richiesta all'app api sulla porta 80». Nota il nome api: è il servizio del Compose (Cap. 16)! Se nginx è nello stesso stack, lo raggiunge per nome.
proxy_set_header ...
passa all'app informazioni utili sulla richiesta originale (l'host chiesto, l'IP reale del client), altrimenti perse dietro il proxy.
Nella pratica, nginx diventa un altro servizio del tuo docker-compose.yml (Cap. 16): è lui a esporre le porte 80/443 verso il mondo, mentre l'app api resta interna, raggiungibile solo dal proxy. Il client parla con nginx; nginx parla con l'app. Questo è lo schema di produzione più diffuso per un backend FastAPI.

Sezione 17.3Servire file statici

nginx è eccezionale nel servire file statici (immagini, CSS, JavaScript, pagine HTML): lo fa molto più velocemente di un'app Python. Conviene affidargli quel compito, lasciando all'app solo la logica.

nginx.conf — statici da nginx, resto all'app
server {
    listen 80;

    location /static/ {              # i file statici...
        root /var/www;              # ...servili direttamente da questa cartella
    }

    location / {                    # tutto il resto...
        proxy_pass http://api:80;     # ...all'app (le API)
    }
}
📖 Divisione dei compiti

nginx confronta il percorso con le location, dalla più specifica: una richiesta a /static/logo.png viene servita direttamente come file (velocissimo, senza toccare l'app); una richiesta a /api/contatti viene inoltrata a FastAPI. Ognuno fa ciò in cui è bravo: nginx i file, l'app la logica. È il pattern che hai visto nel widget del §17.1. Nota: per un backend puramente «API» potresti non avere statici da servire (il front-end sta altrove); ma spessissimo nginx serve anche il front-end compilato, ed è utile saperlo fare.

Sezione 17.4HTTPS con Let's Encrypt

L'HTTPS (il lucchetto) cifra il traffico tra client e server: oggi è indispensabile — protegge i dati (e i token, Cap. 13!) e i browser lo pretendono. La bella notizia: ottenere un certificato è gratis e automatico, grazie a Let's Encrypt.

HTTPS in pratica certificato · Let's Encrypt · certbot
il certificato TLS
un file che «certifica» l'identità del tuo dominio e permette la cifratura (Cap. «Reti», HTTPS/TLS). Lo installa nginx per servire su HTTPS (porta 443).
Let's Encrypt
un'autorità di certificazione gratuita che rilascia certificati validi, riconosciuti da tutti i browser. Ha reso l'HTTPS gratuito e universale.
certbot
lo strumento che ottiene e rinnova automaticamente i certificati di Let's Encrypt e li configura in nginx. Un comando, e il tuo sito è in HTTPS; i certificati (che scadono ogni 90 giorni) si rinnovano da soli.
redirect HTTP → HTTPS
si configura nginx perché ogni richiesta in HTTP (porta 80) venga reindirizzata a HTTPS (443): così nessuno naviga «in chiaro».
terminale — certbot ottiene e installa il certificato
$ sudo certbot --nginx -d mio-sito.com
... Certificato ottenuto e configurato in nginx.
    Il tuo sito è ora raggiungibile in HTTPS! Rinnovo automatico attivo.
Serve un dominio reale che punti al tuo server (§17.6, e Cap. 18). Ma il processo è sorprendentemente semplice: certbot --nginx parla con Let's Encrypt, ottiene il certificato, modifica la config di nginx per usarlo, e imposta il rinnovo automatico. HTTPS professionale, gratis, in un comando — una delle grandi conquiste del web moderno. Ricorda dal Cap. 13: i token e le password viaggiano al sicuro solo su HTTPS.

Sezione 17.5Caddy: HTTPS automatico

Tutto quello che hai visto con nginx è lo standard collaudato, ma richiede qualche passaggio: scrivere la config, installare certbot, lanciarlo, gestire il rinnovo. Esiste un'alternativa più recente che fa le stesse cose con molta meno fatica: Caddy. Merita di stare accanto a nginx nella tua cassetta degli attrezzi.

📖 Cos'è Caddy e perché piace

Caddy è un web server e reverse proxy moderno (scritto in Go), pensato per essere semplice e sicuro «di default». Fa le stesse cose di nginx — punto d'ingresso, reverse proxy, file statici, HTTPS — ma con una differenza che cambia la vita: ottiene e rinnova i certificati HTTPS da solo, in automatico, senza certbot e senza che tu debba fare nulla. La prima volta che avvii Caddy con un dominio, lui contatta Let's Encrypt, ottiene il certificato, configura l'HTTPS e imposta il rinnovo — tutto senza un comando in più. E la sua configurazione (il Caddyfile) è così essenziale che spesso bastano due righe.

Caddyfile — reverse proxy verso FastAPI, con HTTPS automatico
mio-sito.com {                     # il tuo dominio
    reverse_proxy api:80            # inoltra tutto all'app FastAPI
}
✨ Sì, è davvero tutto qui

Confronta queste due righe con la config di nginx del §17.2 più il comando certbot del §17.4: Caddy ottiene lo stesso risultato — reverse proxy e HTTPS con rinnovo automatico — senza nessun passaggio aggiuntivo. Basta indicare il dominio (quella prima riga) e Caddy capisce che vuoi l'HTTPS e se ne occupa. Reindirizza pure da solo l'HTTP (porta 80) all'HTTPS (443), così nessuno naviga in chiaro. Meno codice = meno errori, meno cose da ricordare, meno certificati scaduti per dimenticanza.

Caddyfile — con file statici (equivalente al §17.3)
mio-sito.com {
    handle_path /static/* {         # i file statici...
        root * /var/www            # ...servili da questa cartella
        file_server
    }
    reverse_proxy api:80            # tutto il resto → l'app
}
docker-compose.yml — Caddy al posto di nginx (nota: niente certbot)
services:
  caddy:
    image: caddy
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./Caddyfile:/etc/caddy/Caddyfile       # la tua config
      - caddy_data:/data                     # qui salva i certificati (persistono)
    depends_on: [api]
  # ... api, db, minio come prima ...

volumes:
  caddy_data:
nginx o Caddy? quando scegliere l'uno o l'altro
Caddy — la semplicità
HTTPS automatico, config brevissima, ottime scelte di sicurezza «di serie». Perfetto per partire in fretta, per progetti piccoli e medi, per chi non vuole gestire certbot a mano. Meno cose da sapere per ottenere un sito sicuro e online.
nginx — la maturità
è lo standard da vent'anni: enorme diffusione, tantissima documentazione ed esempi, controllo finissimo su ogni dettaglio, prestazioni altissime su carichi estremi. Se erediti un sistema esistente, molto probabilmente c'è nginx; e per configurazioni molto complesse offre più leve.
quale imparare?
Entrambi, ed è ciò che stai facendo. Conoscere nginx è quasi obbligatorio (lo incontrerai ovunque); conoscere Caddy ti dà una scorciatoia preziosa quando vuoi andare online in fretta e senza grattacapi. Fanno lo stesso mestiere: cambia la filosofia — nginx «configuri tutto tu», Caddy «i buoni default ci sono già».
Esistono anche altri reverse proxy moderni molto usati, come Traefik (amatissimo con Docker e Kubernetes perché si configura «da solo» leggendo le etichette dei container). Ma per un backend FastAPI, nginx e Caddy coprono benissimo la stragrande maggioranza dei casi: il primo come standard consolidato, il secondo come via rapida e serena all'HTTPS.

Sezione 17.6Laboratorio: nginx + certificato

Aggiungiamo nginx al nostro stack e, a grandi linee, il percorso per l'HTTPS. Il laboratorio completo si realizza su un server con un dominio (Cap. 18); qui i pezzi che serviranno.

docker-compose.yml — aggiungi il servizio nginx
services:
  nginx:
    image: nginx
    ports:
      - "80:80"
      - "443:443"              # HTTPS
    volumes:
      - ./nginx.conf:/etc/nginx/conf.d/default.conf   # la tua config
    depends_on: [api]

  api:                          # l'app: NON espone più le porte al mondo,
    build: .                     # è raggiungibile solo da nginx (rete interna)
  # ... db, minio come prima ...
✅ Il quadro completo

Ora nginx è la porta d'ingresso: espone 80/443 al mondo e inoltra all'app api (che non espone più porte esternamente — più sicura). Sul server, dopo aver puntato il dominio, lanci certbot --nginx -d mio-sito.com per l'HTTPS. Il risultato: gli utenti visitano https://mio-sito.com, nginx cifra/decifra e smista, l'app lavora tranquilla dietro. È l'architettura di produzione standard. Manca solo un pezzo: un server vero dove far girare tutto questo, con un dominio — ed è il gran finale del prossimo capitolo.

🔑 Quello che devi portarti dietro dal Capitolo 17
  1. Un reverse proxy sta davanti all'app: punto d'ingresso unico, gestisce HTTPS, smista le richieste, protegge. L'app non è mai esposta direttamente (§17.1).
  2. nginx come reverse proxy: location / { proxy_pass http://api:80; } inoltra le richieste all'app (raggiunta per nome nello stack) (§17.2).
  3. nginx serve i file statici direttamente (veloce) e inoltra il resto all'app: ognuno fa il suo mestiere (§17.3).
  4. HTTPS gratuito con Let's Encrypt + certbot --nginx: certificato e rinnovo automatici. I token/password viaggiano sicuri solo su HTTPS (§17.4).
  5. Caddy è l'alternativa moderna: stesso lavoro di nginx ma con HTTPS automatico (niente certbot) e un Caddyfile di poche righe. nginx = standard maturo e controllo fine; Caddy = semplicità e default sicuri (§17.5).

Abbiamo tutti i pezzi: un'app containerizzata, uno stack orchestrato con Compose, e il portiere nginx con HTTPS davanti. Manca solo il palcoscenico: un server vero, connesso a Internet, con un dominio, dove far vivere tutto questo — e le pratiche per gestirlo nel tempo. È il gran finale: nel Capitolo 18 portiamo il nostro backend in produzione, dal noleggio del server alla messa online, fino alla manutenzione. Il momento in cui ciò che hai costruito diventa, finalmente, raggiungibile dal mondo.

Sezione 17.8Soluzioni

Soluzione 17.1

Un reverse proxy è un server che sta davanti alle applicazioni, riceve tutte le richieste dei client e le inoltra al servizio giusto. Non si espone l'app direttamente perché il proxy fa da scudo (l'app resta isolata, più sicura) e centralizza compiti importanti. Due compiti di nginx: (1) gestire l'HTTPS (cifratura/terminazione TLS); (2) smistare le richieste (per percorso o dominio) e servire i file statici; inoltre può fare cache, limitare gli abusi e bilanciare il carico.

Soluzione 17.2

proxy_pass http://api:80; inoltra la richiesta ricevuta da nginx all'applicazione FastAPI, raggiungibile all'host api sulla porta 80. L'host è api perché è il nome del servizio nel docker-compose.yml (Cap. 16): sulla rete interna dello stack, nginx raggiunge l'app per nome, senza bisogno di indirizzi IP.

Soluzione 17.3

Perché nginx è ottimizzato per servire file statici ed è molto più veloce ed efficiente di un'app Python in quel compito, che così resta libera di occuparsi solo della logica. nginx distingue le richieste tramite le location in base al percorso: es. /static/… viene servito direttamente come file (direttiva root), mentre tutto il resto (/) viene inoltrato all'app con proxy_pass.

Soluzione 17.4

Let's Encrypt è un'autorità che rilascia certificati TLS gratuiti e riconosciuti dai browser; certbot è lo strumento che li ottiene, li configura in nginx e li rinnova automaticamente. L'HTTPS è cruciale per un'app con token perché, senza cifratura, il token di autenticazione (e le password al login) viaggerebbero in chiaro sulla rete: chiunque intercetti il traffico potrebbe rubarli e impersonare l'utente. L'HTTPS cifra tutto, rendendo sicuro il trasporto delle credenziali.

Soluzione 17.5

La differenza principale è che Caddy gestisce l'HTTPS in automatico: ottiene e rinnova i certificati Let's Encrypt da solo, senza certbot e senza comandi aggiuntivi, mentre con nginx bisogna installare e lanciare certbot (e affidarsi al suo rinnovo). Il Caddyfile minimo:

Caddyfile
mio-sito.com {
    reverse_proxy api:80
}

Basta indicare il dominio e Caddy attiva l'HTTPS da sé. Preferirei Caddy quando voglio andare online in fretta, per un progetto piccolo/medio, o quando non voglio gestire i certificati a mano (default sicuri già pronti). Preferirei nginx quando lavoro su un sistema esistente che già lo usa, quando serve un controllo molto fine della configurazione o massime prestazioni su carichi estremi, o quando devo appoggiarmi alla sua vastissima documentazione ed ecosistema.
