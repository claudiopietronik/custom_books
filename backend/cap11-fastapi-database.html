Capitolo 11

FastAPI + database

Ecco il capitolo in cui tutto converge. Abbiamo un'API FastAPI ben fatta (Cap. 4-7) e sappiamo parlare a un database tramite SQLAlchemy (Cap. 10). Ora li saldiamo insieme: riscriviamo la rubrica del Capitolo 7 sostituendo il DB = {} in memoria — che perdeva tutto a ogni riavvio — con un database vero e persistente. Impareremo il meccanismo elegante con cui FastAPI ci procura una sessione del database a ogni richiesta, senza doverla aprire e chiudere a mano: la dependency injection. E chiariremo la convivenza dei due tipi di «modello» che ormai conosci — quelli Pydantic (la forma dei dati dell'API) e quelli SQLAlchemy (le righe del database) — che qui lavorano fianco a fianco, ciascuno col suo compito. Alla fine, la nostra rubrica ricorderà i contatti anche dopo aver spento e riacceso il server: sarà una vera applicazione backend.

🗺 Dove siamo

Mettere insieme i pezzi (§11.1), la dependency injection con get_db (§11.2), il CRUD sul database reale (§11.3), la distinzione schemi Pydantic vs modelli SQLAlchemy (§11.4). Nel laboratorio (§11.5) la rubrica diventa persistente.

🔑 Perché ci tocca: è il backend «completo» in miniatura

Un'API che valida gli input e li salva in un database persistente è, in sostanza, un backend funzionante. Quasi ogni applicazione reale è una variazione, più ricca, di questo schema. Da qui in poi aggiungeremo strati (relazioni, autenticazione, file, deploy), ma il cuore — API + validazione + database — è tutto in questo capitolo. Impararlo bene significa saper costruire il grosso di un servizio vero.

Sezione 11.1Mettere insieme i pezzi

Un progetto backend ordinato divide il codice in file con ruoli chiari. Ecco la struttura tipica — piccola ma già «professionale» — che useremo, con i pezzi già visti nei capitoli scorsi.

la struttura del progetto
mia-api/
├── database.py     # engine, SessionLocal, Base (Cap. 10)
├── models.py       # i modelli SQLAlchemy = le tabelle (Cap. 10)
├── schemas.py      # i modelli Pydantic = input/output API (Cap. 6)
├── main.py         # l'app FastAPI e gli endpoint (Cap. 4-7)
└── requirements.txt
📖 Separare le responsabilità

Ogni file ha un compito: database.py la connessione, models.py le tabelle, schemas.py le forme dell'API, main.py gli endpoint. Questa separazione rende il progetto leggibile e scalabile: sai sempre dove mettere e dove cercare le cose. Man mano che l'app cresce, potresti dividere ulteriormente (una cartella per gli endpoint, una per la logica…), ma questo schema è un ottimo punto di partenza, usato in innumerevoli progetti reali.

Sezione 11.2La dependency injection

A ogni richiesta serve una sessione del database, da aprire prima e chiudere dopo — senza dimenticarsene mai, anche se la funzione va in errore. Scriverlo a mano in ogni endpoint sarebbe noioso e fragile. FastAPI offre un meccanismo pulito: la dependency injection.

database.py — la dipendenza get_db
def get_db():
    db = SessionLocal()      # apri la sessione
    try:
        yield db               # "consegnala" all'endpoint
    finally:
        db.close()             # ...e chiudila SEMPRE, alla fine
main.py — usarla in un endpoint
from fastapi import Depends
from sqlalchemy.orm import Session

@app.get("/contatti")
def elenca(db: Session = Depends(get_db)):   # FastAPI fornisce db!
    return db.query(Contatto).all()
Come funziona l'iniezione Depends · yield · apri/chiudi automatici
def get_db() con yield
una funzione «dipendenza» che apre la sessione, la yield (la consegna), e nel finally la chiude — garantito, anche in caso di errore. Lo scrivi una volta sola.
db: Session = Depends(get_db)
nel parametro dell'endpoint dici: «per db, chiama get_db e passami ciò che produce». FastAPI esegue get_db, ti «inietta» la sessione pronta, e a fine richiesta la chiude per te.
perché è bello
ogni endpoint dichiara semplicemente db: Session = Depends(get_db) e usa db: niente codice ripetuto di apertura/chiusura, niente sessioni dimenticate aperte. Un pattern pulito e a prova di errore.
«Dependency injection» suona complicato, ma l'idea è semplice: invece di procurarti tu ciò che ti serve, lo dichiari e lasci che il framework te lo fornisca. È un concetto potente che FastAPI usa anche per l'autenticazione (Cap. 13): lì una dipendenza get_current_user ti fornirà l'utente loggato, esattamente come get_db ti fornisce la sessione. Imparato una volta, lo riconoscerai ovunque.

Sezione 11.3CRUD sul database reale

Riscriviamo gli endpoint della rubrica (Cap. 7), stavolta contro il database. Nota quanto sono simili a prima: è cambiato solo dove stanno i dati (dal dizionario alla sessione), non la struttura dell'API.

main.py — CRUD persistente
@app.post("/contatti", response_model=schemas.Contatto, status_code=201)
def crea(dati: schemas.ContattoCrea, db: Session = Depends(get_db)):
    contatto = models.Contatto(**dati.model_dump())   # Pydantic -> oggetto ORM
    db.add(contatto)
    db.commit()
    db.refresh(contatto)          # recupera l'id assegnato dal DB
    return contatto

@app.get("/contatti/{id}", response_model=schemas.Contatto)
def leggi(id: int, db: Session = Depends(get_db)):
    contatto = db.query(models.Contatto).get(id)
    if contatto is None:
        raise HTTPException(status_code=404, detail="Non trovato")
    return contatto

@app.delete("/contatti/{id}", status_code=204)
def cancella(id: int, db: Session = Depends(get_db)):
    contatto = db.query(models.Contatto).get(id)
    if contatto is None:
        raise HTTPException(status_code=404, detail="Non trovato")
    db.delete(contatto)
    db.commit()
Cosa è cambiato rispetto al Cap. 7 stessi endpoint, dati persistenti
il parametro db: Session = Depends(get_db)
comparso in ogni endpoint: è la sessione iniettata (§11.2). Da qui operi sul database.
models.Contatto(**dati.model_dump())
il ponte tra i due mondi: prendi i dati validati dal modello Pydantic (dati) e crei un oggetto SQLAlchemy (models.Contatto) da salvare. Pydantic valida, SQLAlchemy persiste.
db.query(...).get(id) invece di DB[id]
la lettura ora interroga il database. Il resto della logica (il controllo None → 404) è identico al Cap. 7.
db.add/commit/delete
sostituiscono le operazioni sul dizionario. Il commit rende i dati permanenti: è ciò che dà alla rubrica la memoria.
Guarda l'essenziale: la forma dell'API è la stessa del Cap. 7 — stessi percorsi, metodi, codici di stato, gestione del 404. È cambiato solo il «dietro»: dal dizionario volatile a un database persistente. Questo conferma quanto anticipato: progettare bene l'API (Cap. 7) ripaga, perché il livello dei dati si può cambiare sotto senza stravolgere gli endpoint.

Sezione 11.4Schemi Pydantic vs modelli ORM

Un punto che confonde molti principianti: perché due tipi di modello? Non basterebbe uno? No — e capire il perché chiarisce l'intera architettura.

Due modelli, due lavori Pydantic (API) vs SQLAlchemy (DB)
Modello SQLAlchemy (models.py)
descrive la tabella del database: colonne, tipi, vincoli, chiavi. Il suo mondo è il database. Es. models.Contatto.
Schema Pydantic (schemas.py)
descrive la forma dei dati dell'API: cosa il client può inviare e cosa riceve. Il suo mondo è l'HTTP/JSON. Es. schemas.ContattoCrea (input), schemas.Contatto (output).
perché separarli
i due mondi hanno esigenze diverse: l'input non deve avere l'id (lo assegna il DB); l'output non deve rivelare campi sensibili (password); il DB ha dettagli (indici, relazioni) che all'API non interessano. Tenerli separati mantiene ognuno pulito e sicuro.
schemas.py — gli schemi dell'API
from pydantic import BaseModel

class ContattoCrea(BaseModel):     # INPUT: senza id (lo dà il DB)
    nome: str
    telefono: str

class Contatto(BaseModel):         # OUTPUT: con id
    id: int
    nome: str
    telefono: str
    class Config:
        from_attributes = True       # legge dagli oggetti ORM
La riga from_attributes = True (in Pydantic v2) è il «traduttore»: permette a FastAPI di costruire la risposta JSON leggendo direttamente da un oggetto SQLAlchemy (i suoi attributi). Così puoi return contatto (un oggetto ORM) e, grazie al response_model=schemas.Contatto, FastAPI lo converte nel JSON giusto. È l'ultimo anello: Pydantic valida l'ingresso → SQLAlchemy persiste → Pydantic formatta l'uscita. Tre modelli familiari (Cap. 6, 8, 10) che finalmente collaborano.

Sezione 11.5Laboratorio: rubrica persistente

La rubrica qui sotto funziona come quella del Cap. 7 — ma con una differenza cruciale: ricorda i dati. Crea qualche contatto, poi premi «riavvia il server» (o ricarica questa pagina): i contatti sono ancora lì. È la persistenza del database, resa visibile.

La rubrica persistente i dati sopravvivono al riavvio
    Rubrica vuota. Crea un contatto.
    🔬 Il contrasto col Capitolo 7

    Ricordi la rubrica del Cap. 7? Era identica da usare, ma «riavviando» perdeva tutto: viveva in un dizionario in memoria. Questa versione salva su un database (qui simulato con l'archivio del browser, ma il concetto è lo stesso di SQLite/Postgres): premi «riavvia» — o ricarica la pagina — e i contatti restano. Questa, e non altro, è la differenza che rende un backend utile nel mondo reale. «Svuota il database» cancella tutto di proposito (come un DELETE FROM contatti).

    🔑 Quello che devi portarti dietro dal Capitolo 11
    1. Struttura ordinata: database.py (connessione), models.py (tabelle), schemas.py (API), main.py (endpoint) (§11.1).
    2. Dependency injection: get_db (con yield) apre e chiude la sessione; l'endpoint la riceve con db: Session = Depends(get_db) (§11.2).
    3. Gli endpoint CRUD usano db.add/commit/query/delete: stessa forma del Cap. 7, ma dati persistenti (§11.3).
    4. Due modelli: Pydantic (schemi API, input/output) e SQLAlchemy (tabelle DB); il ponte è model_dump() in ingresso e from_attributes=True in uscita (§11.4).

    La nostra rubrica è ora una vera applicazione: valida, salva, ricorda. Ma i dati del mondo reale sono collegati tra loro: un utente ha molti post, un ordine appartiene a un cliente, un post ha molti commenti. Come si rappresentano questi legami tra tabelle, e come si navigano con SQLAlchemy? È il tema del Capitolo 12: le relazioni — chiavi esterne, uno-a-molti, relationship — che trasformano tabelle isolate in un vero modello di dati.

    Sezione 11.7Soluzioni

    Soluzione 11.1

    get_db fornisce a ogni endpoint una sessione del database già aperta, tramite Depends(get_db): l'endpoint la dichiara come parametro e FastAPI gliela «inietta». Il try/finally con db.close() garantisce che la sessione venga sempre chiusa al termine della richiesta — anche se la funzione solleva un errore — evitando sessioni «dimenticate» aperte, che a lungo andare esaurirebbero le risorse.

    Soluzione 11.2

    Perché servono a cose diverse: il modello Pydantic definisce la forma dei dati dell'API (input e output), il modello SQLAlchemy definisce la tabella del database. Esempio: il campo id sta nel modello di output (il client deve conoscerlo) ma non in quello di input di creazione, perché è il database ad assegnarlo automaticamente — chiederlo al client sarebbe inutile e rischioso. (Altro esempio inverso: la password sta nell'input ma mai nell'output.)

    Soluzione 11.3

    dati.model_dump() converte l'oggetto Pydantic validato in un dizionario; ** lo «spacchetta» come argomenti con nome per costruire un oggetto SQLAlchemy models.Contatto(nome=..., telefono=...). Rappresenta la traduzione dal mondo dell'API (dati validati da Pydantic) al mondo del database (un oggetto ORM pronto per essere salvato con add+commit).

    Soluzione 11.4

    Dal punto di vista dell'utente dell'API non cambia nulla: stessi endpoint, stessi metodi, stessi codici di stato, stesse risposte JSON. «Dietro le quinte», invece, cambia tutto ciò che conta: i dati non vivono più in un dizionario volatile ma in un database persistente, quindi sopravvivono ai riavvii, possono crescere enormemente, essere cercati velocemente e usati da più istanze del server insieme. È la prova che una buona progettazione dell'API disaccoppia l'interfaccia dall'implementazione.
