Capitolo 3

Preparare l'ambiente Python

Prima di scrivere una riga del nostro server, allestiamo il banco di lavoro. Sembra un passo noioso, ma saltarlo è la causa numero uno di guai per chi inizia: pacchetti che entrano in conflitto, «sul mio computer funzionava», progetti che si rompono a vicenda. La soluzione è un'abitudine semplice e universale: dare a ogni progetto il suo ambiente isolato, una bolla con la sua copia di Python e i suoi pacchetti, separata da tutto il resto. In questo capitolo capisci perché serve (§3.1), crei un ambiente virtuale con venv (§3.2), installi pacchetti con pip (§3.3) e li «congeli» in un requirements.txt (§3.4) — il file che permette a chiunque (incluso il tuo futuro server) di ricreare esattamente lo stesso ambiente. È la disciplina che separa un progetto professionale da uno che si sfalda.

🗺 Dove siamo

Perché isolare (§3.1), creare e attivare un venv (§3.2), installare con pip (§3.3), congelare le dipendenze in requirements.txt (§3.4). Nel laboratorio (§3.5) prepari la cartella del progetto che useremo per tutto il libro.

🔑 Perché ci tocca: le dipendenze sono metà del backend

Un backend moderno non è solo il tuo codice: è il tuo codice più decine di pacchetti (FastAPI, il driver del database, la libreria di sicurezza…). Gestire queste dipendenze in modo pulito e riproducibile è una competenza tanto importante quanto saper programmare. Un ambiente ordinato oggi ti evita ore di debug domani, e rende possibile il grande finale: impacchettare tutto in Docker (Cap. 15) e replicarlo identico in produzione.

Sezione 3.1Perché un ambiente isolato

Immagina di avere due progetti sullo stesso computer: uno vecchio che usa la versione 1 di una libreria, uno nuovo che richiede la versione 2. Se installi i pacchetti «globalmente», per tutto il sistema, i due progetti si contendono la stessa copia — e uno dei due si romperà.

⚠ Il problema dell'installazione globale

Se installi ogni pacchetto nel Python «di sistema», tutti i progetti (e persino strumenti del sistema operativo) pescano dallo stesso mucchio. Basta un progetto che aggiorni una libreria a una versione incompatibile per rompere gli altri. In più, il tuo elenco di pacchetti diventa un caos indistinto: non sai più quale progetto ha bisogno di cosa. Non saprai mai ricreare l'ambiente esatto su un altro computer o su un server — e il deploy diventa un incubo.

📖 La soluzione: una bolla per progetto

Un ambiente virtuale è una cartella, dentro il tuo progetto, che contiene una copia isolata di Python e dei suoi pacchetti. Quando lo «attivi», python e pip lavorano solo lì dentro: ciò che installi resta nella bolla, senza toccare il sistema né gli altri progetti. Ogni progetto ha le sue dipendenze, alle sue versioni, senza interferenze. È lo standard assoluto nello sviluppo Python: praticamente ogni progetto serio ne ha uno.

Sezione 3.2Gli ambienti virtuali (venv)

Python include lo strumento per creare queste bolle: il modulo venv. Sono due comandi — creare e attivare — e poi lavori come sempre, ma dentro l'ambiente.

terminale — creare e attivare (Windows)
# 1) crea l'ambiente: nasce una cartella ".venv" nel progetto
$ python -m venv .venv

# 2) attivalo (Windows, PowerShell)
$ .venv\Scripts\Activate.ps1
(.venv) $                          # il prompt ora mostra (.venv): sei "dentro"
terminale — attivare su macOS / Linux
$ python3 -m venv .venv
$ source .venv/bin/activate
(.venv) $
I comandi smontati creare · attivare · disattivare
python -m venv .venv
esegue il modulo venv per creare un ambiente in una cartella. Il secondo .venv è il nome della cartella (una convenzione diffusa; il punto la nasconde). Lo fai una volta sola per progetto.
…Activate.ps1 / source …/activate
attiva l'ambiente nella sessione del terminale corrente. Da qui, python e pip sono quelli della bolla. Lo rifai ogni volta che apri un nuovo terminale per lavorare al progetto.
il prompt (.venv)
il segnale visivo che l'ambiente è attivo: appare all'inizio della riga. Se non lo vedi, non sei nell'ambiente — errore comunissimo che porta a installare i pacchetti nel posto sbagliato.
deactivate
esce dalla bolla e torni al Python di sistema. Raramente serve: di solito chiudi e basta il terminale.
Windows a volte blocca l'esecuzione di Activate.ps1 per policy di sicurezza: se vedi un errore, esegui una volta Set-ExecutionPolicy -Scope CurrentUser RemoteSigned in PowerShell, oppure usa il Prompt dei comandi con .venv\Scripts\activate.bat. Regola d'oro: la cartella .venv non va mai messa sotto git (si ricrea da requirements.txt) — la aggiungerai al .gitignore.

Sezione 3.3pip e i pacchetti

Con l'ambiente attivo, installi le librerie con pip, il gestore di pacchetti di Python. Attinge a PyPI (il grande archivio pubblico dei pacchetti Python) e mette il pacchetto — con tutto ciò da cui dipende — dentro la tua bolla.

terminale (con (.venv) attivo)
(.venv) $ pip install fastapi uvicorn
Collecting fastapi
  Downloading fastapi-...-py3-none-any.whl
Collecting uvicorn
  ...
Successfully installed fastapi-... uvicorn-... starlette-... pydantic-...

(.venv) $ pip list            # cosa c'è installato nella bolla
Package    Version
---------- -------
fastapi    ...
pydantic   ...
uvicorn    ...
I comandi pip essenziali install · list · uninstall
pip install nome
scarica e installa un pacchetto (e le sue dipendenze) nell'ambiente attivo. Puoi indicarne più d'uno, o fissare una versione: pip install fastapi==0.115.0.
pip list
elenca cosa è installato nella bolla. Nota come le dipendenze «a cascata» (starlette, pydantic…) siano entrate insieme a FastAPI: pip risolve automaticamente ciò che serve.
pip uninstall nome
rimuove un pacchetto. Utile per fare pulizia.
pip install -r requirements.txt
installa tutti i pacchetti elencati in un file (§3.4): il comando che ricrea un ambiente identico altrove. Lo userai in continuazione.
Se dimentichi di attivare il venv, pip install mette i pacchetti nel Python globale — di nuovo il problema del §3.1. Controlla sempre il (.venv) nel prompt prima di installare. È l'errore più comune, e ora sai riconoscerlo.

Sezione 3.4requirements.txt

C'è un ultimo pezzo che rende l'ambiente riproducibile: elencare le dipendenze del progetto in un file di testo, il requirements.txt. È il «foglietto della spesa» delle librerie: chiunque lo abbia può ricreare il tuo ambiente esatto con un comando.

terminale
# "congela" l'elenco esatto (con le versioni) in un file
(.venv) $ pip freeze > requirements.txt
requirements.txt (esempio)
fastapi==0.115.0
uvicorn==0.30.6
pydantic==2.9.2
starlette==0.38.6
📖 Perché è il file più importante del progetto (dopo il codice)

pip freeze stampa ogni pacchetto installato con la sua versione esatta; il > requirements.txt lo salva su file. Chiunque poi, su qualsiasi computer, crea un venv e lancia pip install -r requirements.txt per avere lo stesso ambiente, identico. Questo file va sotto git (a differenza della cartella .venv): è la ricetta, non gli ingredienti già cucinati. È anche ciò che il Dockerfile userà (Cap. 15) per costruire l'immagine del tuo server, ed è ciò che rende il tuo progetto installabile ovunque senza sorprese. Fissare le versioni (==) garantisce che «funziona qui» significhi «funziona anche là».

Sezione 3.5Laboratorio: l'ambiente del progetto

Prepariamo ora la cartella che useremo per costruire il nostro backend nei prossimi capitoli. Segui i passi: alla fine avrai un ambiente pulito, pronto per FastAPI.

terminale — la ricetta completa (Windows / PowerShell)
# 1) crea la cartella del progetto ed entra
$ mkdir mia-api
$ cd mia-api

# 2) crea e attiva l'ambiente virtuale
$ python -m venv .venv
$ .venv\Scripts\Activate.ps1

# 3) installa FastAPI e il server uvicorn
(.venv) $ pip install fastapi uvicorn

# 4) congela le dipendenze
(.venv) $ pip freeze > requirements.txt

# 5) proteggi la cartella .venv da git (creando .gitignore)
(.venv) $ echo ".venv/" > .gitignore
✅ Come sapere che è andato tutto bene

Alla fine, nella cartella mia-api devi avere: la cartella .venv, un file requirements.txt con dentro fastapi e uvicorn, e un .gitignore. Il prompt mostra (.venv). Verifica con pip list: devi vedere fastapi nell'elenco. Se tutto torna, il banco di lavoro è pronto: nel prossimo capitolo scriviamo finalmente il server. Se qualcosa non torna, ricontrolla di aver attivato il venv (il (.venv) nel prompt) prima del pip install.

🔑 Quello che devi portarti dietro dal Capitolo 3
  1. Installare pacchetti globalmente crea conflitti e rende l'ambiente irriproducibile: isola ogni progetto (§3.1).
  2. venv: python -m venv .venv (una volta) + attivazione (ogni terminale). Il prompt (.venv) conferma che sei nella bolla (§3.2).
  3. pip: install, list, uninstall. Installa solo con il venv attivo (§3.3).
  4. requirements.txt (da pip freeze) congela le versioni: si ricrea l'ambiente con pip install -r. Va sotto git; la cartella .venv no (§3.4).

Il banco di lavoro è pronto e pulito: un ambiente isolato con FastAPI installato. Non resta che accendere il motore. Nel Capitolo 4 scriviamo il primo server con FastAPI — poche righe di Python, l'avvio con uvicorn, e la meraviglia della documentazione interattiva che FastAPI genera da sola. Dal prossimo capitolo, il backend che finora abbiamo solo descritto inizia davvero a rispondere.

Sezione 3.7Soluzioni

Soluzione 3.1

Esempio: il progetto A funziona con libreria 1.0, il progetto B richiede libreria 2.0 (che ha cambiato il modo di chiamare una funzione). Con l'installazione globale c'è una sola copia della libreria per tutto il sistema: installando la 2.0 per B, il progetto A — che si aspetta la 1.0 — smette di funzionare. Gli ambienti virtuali risolvono dando a ciascun progetto la sua versione.

Soluzione 3.2

Creare (python -m venv .venv) genera la cartella dell'ambiente: si fa una volta sola per progetto. Attivare (Activate.ps1 / source .../activate) rende quell'ambiente quello «in uso» nel terminale corrente: si fa ogni volta che apri un nuovo terminale per lavorare al progetto (l'attivazione vale solo per quella sessione).

Soluzione 3.3

Perché requirements.txt è la ricetta (piccolo file di testo, condivisibile, che dice cosa installare): serve a tutti e va versionato. La cartella .venv, invece, sono gli ingredienti già cucinati: è grande, specifica del tuo sistema operativo, e ricreabile in qualsiasi momento da requirements.txt. Metterla sotto git sarebbe inutile e problematico — per questo va nel .gitignore.

Soluzione 3.4
terminale
$ cd progetto-del-collega
$ python -m venv .venv          # crea l'ambiente
$ .venv\Scripts\Activate.ps1       # attivalo
(.venv) $ pip install -r requirements.txt   # installa tutto

In tre passi hai ricreato l'ambiente esatto del collega. È esattamente ciò che farà anche il server in produzione.
