Capitolo 14

File e storage con MinIO

Un'immagine del profilo, un allegato PDF, un video, un documento: prima o poi ogni backend deve gestire file caricati dagli utenti. E qui si nasconde una domanda importante, con una risposta forse controintuitiva: dove li mettiamo? Non nel database (vedremo perché è una cattiva idea), e nemmeno «sparsi» sul disco del server. La soluzione professionale è lo storage di oggetti: un servizio specializzato nel conservare file, a cui si accede tramite un'API standard chiamata S3. E per averlo «in casa», sul nostro server, useremo MinIO, un software open source compatibile con S3. In questo capitolo capisci perché i file non vanno nel database (§14.1), cos'è l'object storage (§14.2), cos'è MinIO (§14.3) e come caricare e scaricare file (§14.4). È l'ultima capacità applicativa del nostro backend prima di impacchettarlo e metterlo online.

🗺 Dove siamo

Perché i file non nel database (§14.1), l'object storage e lo standard S3 (§14.2), MinIO come S3 «in casa» (§14.3), caricare e scaricare file da FastAPI (§14.4). Nel laboratorio (§14.5) gestisci l'upload di un'immagine.

🔑 Perché ci tocca: i file sono ovunque

Avatar, foto, documenti, backup: quasi ogni app gestisce file. Farlo nel modo giusto — separando i file dal database — è una scelta architetturale che influenza prestazioni, costi e scalabilità. E imparare l'API S3 è particolarmente prezioso: è lo stesso standard usato da Amazon S3 e da innumerevoli servizi cloud, quindi ciò che impari qui con MinIO vale identico «lassù», nel cloud.

Sezione 14.1Perché non i file nel database

Sarebbe tecnicamente possibile salvare un file (i suoi byte) dentro una colonna del database. Ma è quasi sempre una cattiva idea, per ragioni concrete.

Perché i file stanno fuori dal database prestazioni · costi · ruoli
appesantiscono il database
i database sono ottimizzati per dati strutturati e piccoli (numeri, testo), non per grossi blocchi binari. Riempirli di immagini li rende lenti e i backup enormi.
costano di più e scalano peggio
lo spazio del database è prezioso e costoso; lo storage di file è economico e pensato per crescere enormemente. Mescolarli spreca risorse.
ruoli diversi
servire un'immagine a migliaia di utenti è un lavoro da server di file / CDN, non da database. Tenerli separati permette a ciascuno di fare bene il proprio mestiere.
la regola pratica
nel database salvi solo un riferimento al file (il suo nome/chiave o URL); il file vero vive nello storage di oggetti. Così la riga utente ha un campo avatar_url, non i megabyte dell'immagine.
Questa separazione — metadati nel database, file nello storage — è uno dei pattern architetturali più importanti del backend. Il database dice «l'avatar di Anna è il file avatars/anna.jpg»; lo storage custodisce quel file. Quando servi il profilo, restituisci l'URL; il browser scarica l'immagine direttamente dallo storage. Semplice, veloce, scalabile.

Sezione 14.2L'object storage (S3)

Lo storage di oggetti (object storage) è un tipo di archivio pensato apposta per conservare file («oggetti») in grande quantità. Il modello reso famoso da Amazon S3 è diventato lo standard di fatto, imitato da tutti.

Il vocabolario dell'object storage bucket · object · key
oggetto (object)
un singolo file, con i suoi dati e alcuni metadati (tipo, dimensione…). L'unità base dello storage.
bucket (secchio)
un «contenitore» di oggetti, come una cartella di primo livello. Organizzi i file in bucket (es. un bucket avatar, uno documenti). Ogni bucket ha un nome unico.
chiave (key)
il «nome» univoco di un oggetto dentro il bucket, spesso con una struttura a percorso: utenti/42/avatar.jpg. Bucket + key identificano un file in modo univoco — è ciò che salvi nel database.
l'API S3
un insieme standard di operazioni: put_object (carica), get_object (scarica), delete_object (cancella), list_objects (elenca). Uno standard universale: impararlo una volta serve con S3, MinIO e decine di altri servizi.
Perché è così diffuso? Perché l'object storage è economico, enormemente scalabile (miliardi di file), affidabile (i dati sono replicati) e accessibile via HTTP (ogni oggetto può avere un URL). È il modo standard di conservare file nel cloud, da Netflix alle piccole startup. Noi lo useremo in versione «self-hosted» con MinIO — stessa API, sul nostro server.

Sezione 14.3MinIO: S3 in casa

MinIO è un software open source che fornisce uno storage di oggetti compatibile con l'API S3, ma che puoi far girare sul tuo server (o sul tuo computer). In pratica: tutta la comodità di S3, senza dipendere da un provider cloud e senza costi per il traffico.

🪣 Perché MinIO

MinIO parla la stessa API di Amazon S3: il codice che scrivi per MinIO funziona identico con S3 (e viceversa) cambiando solo l'indirizzo. Questo dà due vantaggi: (1) sviluppi e testi in locale con MinIO, gratis; (2) se un domani vuoi passare al cloud, non riscrivi nulla. Lo avvii facilmente con Docker (Cap. 15):

terminale — MinIO in un container
$ docker run -p 9000:9000 -p 9001:9001 \
       -e MINIO_ROOT_USER=admin \
       -e MINIO_ROOT_PASSWORD=segreta123 \
       minio/minio server /data --console-address ":9001"
... API su :9000 · console web su :9001

La porta 9000 è l'API (per il codice), la 9001 è una comoda console web dove vedere e gestire bucket e file dal browser. User e password root vanno tenuti come segreti (variabili d'ambiente, Cap. 9), come sempre.

Sezione 14.4Caricare e scaricare file

Da FastAPI, ricevi il file caricato dall'utente e lo passi allo storage. Servono due pezzi: FastAPI per ricevere il file (UploadFile), e un client S3 (per Python, boto3 o minio) per metterlo nello storage.

main.py — endpoint di upload
from fastapi import FastAPI, UploadFile, Depends

@app.post("/utenti/{id}/avatar")
def carica_avatar(id: int, file: UploadFile, db: Session = Depends(get_db)):
    key = f"avatar/{id}/{file.filename}"            # la "chiave" nello storage

    # carica il file nel bucket "media" via client S3 (MinIO)
    s3.put_object("media", key, file.file, length=-1, part_size=10*1024*1024)

    # nel DATABASE salva SOLO il riferimento (la key), non il file!
    utente = db.query(Utente).get(id)
    utente.avatar_key = key
    db.commit()
    return {"avatar": key}
L'anatomia dell'upload UploadFile · put_object · salva la key
file: UploadFile
il modo di FastAPI per ricevere un file caricato. Ti dà file.filename (il nome) e file.file (i dati). FastAPI gestisce lo streaming, così anche file grandi non saturano la memoria.
la key
costruisci un nome univoco per l'oggetto, spesso includendo l'id utente per organizzare: avatar/42/foto.jpg.
s3.put_object("media", key, ...)
carica i dati nel bucket media con quella key. È l'operazione S3 «metti un oggetto». Il file ora vive nello storage.
utente.avatar_key = key; db.commit()
nel database salvi solo la key (il riferimento), non il file. È il pattern del §14.1 realizzato: metadati nel DB, file nello storage.
Per scaricare/servire il file, hai due strade: leggerlo dallo storage e restituirlo (get_object), oppure — meglio per file pubblici — generare un URL che punta direttamente allo storage e darlo al client, che scarica da lì senza passare dal tuo server. Quest'ultima è la via scalabile: il tuo backend non fa da «tramite» per ogni download, alleggerendosi. Per i file privati si usano URL temporanei firmati (presigned URL), che scadono dopo un po'.

    Esercizio 14.2
    
    
    
      Definisci, nell'object storage, i termini bucket, oggetto e key. Cosa identifica in modo
      univoco un file?
    
    
  
    

  
    
    
    Esercizio 14.3
    
    
    
      Cos'è MinIO e qual è il vantaggio del fatto che sia «compatibile con l'API S3»?
    
    
  
    

  
    
    
    Esercizio 14.4
    
    
    
      Nell'endpoint di upload, quale riga carica il file nello storage e quale salva il riferimento nel database?
      Perché sono due operazioni separate?
    
    
  
    


  
  
    Sezione 14.7Soluzioni
    

  
    
    
    Soluzione 14.1
    
    
    
      Due motivi: (1) i file grossi appesantiscono il database, rendendolo lento e gonfiando i backup
      (i database sono ottimizzati per dati piccoli e strutturati); (2) lo spazio del database costa di
      più e scala peggio rispetto allo storage di oggetti, pensato per crescere. Nel database si salva solo un
      riferimento al file — la sua key (o URL) — mentre il file vero vive nello storage.
    
    
  
    

  
    
    
    Soluzione 14.2
    
    
    
      Un oggetto è un singolo file conservato nello storage. Un bucket è un contenitore di oggetti (come
      una cartella di primo livello). La key è il nome univoco di un oggetto dentro il bucket (es.
      avatar/42/foto.jpg). Un file è identificato in modo univoco dalla combinazione bucket +
      key.
    
    
  
    

  
    
    
    Soluzione 14.3
    
    
    
      MinIO è un software open source di storage di oggetti che puoi far girare sul tuo server. Essendo
      compatibile con l'API S3, il codice scritto per MinIO funziona identico con Amazon S3 (e altri
      servizi S3-compatibili) cambiando solo l'indirizzo: puoi sviluppare in locale gratis con MinIO e, se serve,
      passare al cloud senza riscrivere nulla. Impari uno standard, non uno strumento singolo.
    
    
  
    

  
    
    
    Soluzione 14.4
    
    
    
      La riga s3.put_object("media", key, ...) carica il file nello storage; la riga
      utente.avatar_key = key; db.commit() salva il riferimento nel database. Sono separate perché
      i due sistemi hanno ruoli diversi: lo storage custodisce i byte del file, il database custodisce i metadati e i
      collegamenti. Il file «vero» sta nello storage; il database sa solo dove trovarlo — la separazione che
      rende il sistema efficiente e scalabile.
