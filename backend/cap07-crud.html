Capitolo 7

Un'API CRUD completa

Abbiamo tutti i mattoni: rotte, parametri, modelli, validazione. È ora di metterli insieme e costruire la nostra prima vera applicazione: un'API che gestisce una risorsa in tutte le sue operazioni. Il pattern si chiama CRUD — Create, Read, Update, Delete — ed è lo scheletro di quasi ogni backend del mondo: un social fa CRUD sui post, un negozio sui prodotti, una banca sui movimenti. Costruiremo l'API di una rubrica di contatti, con tutti e cinque gli endpoint classici, e impareremo a gestire gli errori con eleganza: cosa rispondere quando qualcuno chiede un contatto che non esiste? (Un bel 404, fatto bene.) Per ora terremo i dati in memoria, in un semplice dizionario Python; ma la forma di questa API è esattamente quella che, dal Capitolo 11, collegheremo a un database vero — cambierà solo dove stanno i dati, non come è fatta l'API.

🗺 Dove siamo

Cosa significa CRUD (§7.1), come progettare le risorse (§7.2), i cinque endpoint in pratica (§7.3), la gestione degli errori con HTTPException (§7.4). Nel laboratorio (§7.5) scrivi l'API completa di una rubrica.

🔑 Perché ci tocca: è lo scheletro di ogni backend

Il 90% del lavoro di backend è, in fondo, «gestire delle risorse»: crearle, leggerle, aggiornarle, cancellarle. Padroneggiare il pattern CRUD significa saper costruire la struttura portante di quasi qualsiasi applicazione. Tutto il resto (database, auth, file) sono strati che si appoggiano su questa ossatura. Imparala bene una volta, e la riuserai per sempre.

Sezione 7.1Cosa significa CRUD

CRUD è l'acronimo delle quattro operazioni fondamentali sui dati. Ciascuna corrisponde, quasi perfettamente, a un metodo HTTP — il che rende naturale tradurle in endpoint REST.

CRUD ↔ HTTP ↔ SQL le quattro operazioni sui dati
Create — POST
creare una nuova risorsa. POST /contatti con i dati nel corpo. (In SQL sarà INSERT, Cap. 8.)
Read — GET
leggere: l'intera collezione (GET /contatti) o un singolo elemento (GET /contatti/42). (In SQL SELECT.)
Update — PUT / PATCH
modificare una risorsa esistente. PUT /contatti/42 coi nuovi dati. (In SQL UPDATE.)
Delete — DELETE
cancellare. DELETE /contatti/42. (In SQL DELETE.)
Nota la corrispondenza quasi magica: le operazioni sui dati, i metodi HTTP e i comandi SQL «parlano» lo stesso linguaggio a quattro voci. Non è un caso: REST è stato pensato proprio per esporre risorse manipolabili in questo modo. Questa coerenza è ciò che rende il pattern così naturale — e ciò che ti farà sentire «a casa» quando, tra pochi capitoli, sotto gli endpoint metteremo un database SQL.

Sezione 7.2Progettare le risorse

Prima di scrivere codice, decidiamo la forma dell'API: quale risorsa, quali campi, quali endpoint. Una buona progettazione REST rende l'implementazione quasi meccanica.

📐 La rubrica: risorsa «contatto»

La nostra risorsa è il contatto, con campi: nome, telefono, email. Ogni contatto avrà un id assegnato dal server. Ecco la tabella degli endpoint — il «contratto» dell'API:

il contratto REST della rubrica
GET    /contatti       # elenca tutti i contatti          -> 200
POST   /contatti       # crea un contatto (dati nel corpo) -> 201
GET    /contatti/{id}  # leggi un contatto                 -> 200 / 404
PUT    /contatti/{id}  # aggiorna un contatto              -> 200 / 404
DELETE /contatti/{id}  # cancella un contatto              -> 204 / 404

Definire prima questa tabella è metà del lavoro: dice esattamente quali funzioni scrivere, cosa ricevono e cosa restituiscono (compresi i codici di stato). Nota i due esiti possibili degli endpoint su {id}: successo, oppure 404 se quell'id non esiste — la gestione degli errori del §7.4.

Sezione 7.3GET/POST/PUT/DELETE in pratica

Scriviamo il codice, endpoint per endpoint. I dati stanno in un dizionario Python (DB) che fa da «finto database»: id → contatto. Ogni funzione è breve, perché rotte, modelli e validazione fanno il lavoro pesante.

main.py — preparazione
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel

app = FastAPI()

class Contatto(BaseModel):     # la forma dei dati in ingresso
    nome: str
    telefono: str
    email: str = ""

DB = {}              # il "database" in memoria: id -> dati
prossimo_id = 1       # contatore per i nuovi id
main.py — i cinque endpoint CRUD
@app.get("/contatti")                    # READ (tutti)
def elenca():
    return DB

@app.post("/contatti", status_code=201)   # CREATE
def crea(contatto: Contatto):
    global prossimo_id
    DB[prossimo_id] = contatto.model_dump()   # salva come dizionario
    nuovo = {"id": prossimo_id, **DB[prossimo_id]}
    prossimo_id += 1
    return nuovo

@app.get("/contatti/{id}")               # READ (uno)
def leggi(id: int):
    if id not in DB:
        raise HTTPException(status_code=404, detail="Contatto non trovato")
    return {"id": id, **DB[id]}

@app.put("/contatti/{id}")               # UPDATE
def aggiorna(id: int, contatto: Contatto):
    if id not in DB:
        raise HTTPException(status_code=404, detail="Contatto non trovato")
    DB[id] = contatto.model_dump()
    return {"id": id, **DB[id]}

@app.delete("/contatti/{id}", status_code=204)  # DELETE
def cancella(id: int):
    if id not in DB:
        raise HTTPException(status_code=404, detail="Contatto non trovato")
    del DB[id]
I dettagli che contano status_code · model_dump · {id, **dati}
status_code=201 nel decoratore
imposti il codice di successo giusto: 201 Created per la creazione, 204 No Content per la cancellazione (Cap. 2). Il default è 200; qui scegli quello semanticamente corretto.
contatto.model_dump()
converte l'oggetto Pydantic in un normale dizionario Python (per salvarlo nel nostro DB finto). Dal Cap. 11 questo passo diventerà «salva nel database».
{"id": id, **DB[id]}
costruisce la risposta unendo l'id ai dati del contatto (** «spacchetta» il dizionario). Restituisce al client la risorsa completa, id incluso.
raise HTTPException(...)
il modo di segnalare un errore — il tema della prossima sezione.
Guarda la regolarità: ogni endpoint è la traduzione diretta di una riga della tabella del §7.2. La validazione dell'input (è un Contatto valido?) e la conversione dei tipi (id: int) avvengono prima, gratis. Restano solo poche righe di logica pura per endpoint. Questa è la densità di FastAPI: tanto risultato, poco codice.

Sezione 7.4Gestire gli errori

Cosa deve succedere se qualcuno chiede GET /contatti/999 ma quel contatto non esiste? Restituire dati vuoti sarebbe confuso e scorretto. La risposta giusta è un 404 Not Found, e FastAPI lo rende semplice con HTTPException.

segnalare un errore
from fastapi import HTTPException

if id not in DB:
    raise HTTPException(status_code=404, detail="Contatto non trovato")
HTTPException: l'errore fatto bene raise · status_code · detail
raise HTTPException(...)
raise «solleva» l'eccezione: interrompe subito la funzione e FastAPI la trasforma in una risposta HTTP di errore. Non serve return: appena sollevata, la funzione finisce lì.
status_code=404
il codice giusto per «non trovato» (Cap. 2). Puoi usarne altri: 403 (vietato), 400 (richiesta errata), 409 (conflitto)… a seconda del problema.
detail="Contatto non trovato"
il messaggio, che finisce nel JSON di risposta: {"detail": "Contatto non trovato"}. Spiega al client cosa è andato storto, in modo utile ma senza rivelare dettagli interni sensibili.
La differenza tra un'API dilettante e una professionale è spesso proprio qui: nella gestione onesta e coerente degli errori. Codice di stato corretto + messaggio chiaro = un'API con cui è un piacere lavorare. Ricorda i due mondi di errore (Cap. 2): usa 4xx quando il client ha sbagliato (id inesistente, dati non validi); i 5xx li genera FastAPI da solo quando c'è un bug nel tuo codice — e sono il segnale che c'è qualcosa da correggere.

    1. CRUD = Create/Read/Update/Delete, che mappano su POST/GET/PUT/DELETE (e su INSERT/SELECT/ UPDATE/DELETE in SQL) (§7.1).

Un backend vero deve ricordare per sempre: è il momento di dare alla nostra rubrica una memoria permanente. Nei prossimi capitoli entriamo nel mondo dei database: dal Capitolo 8, cosa sono le tabelle e come si parla loro con il linguaggio SQL — le fondamenta su cui poi (Cap. 10–11) sostituiremo il nostro DB = {} con un vero archivio persistente.

    Sezione 7.6Esercizi

    Esercizio 7.1

    Scrivi la tabella CRUD (metodo + percorso + codice di successo) per una risorsa libri, sul modello di quella dei contatti (§7.2).

    Esercizio 7.2

    Perché per la creazione si imposta status_code=201 e per la cancellazione 204, invece di lasciare il default 200? Cosa comunicano questi codici?

    Esercizio 7.3

    Cosa fa raise HTTPException(status_code=404, detail="...")? In cosa differisce da un normale return?

    Esercizio 7.4

    Nell'endpoint DELETE /contatti/{id}, cosa succede (con questo codice) se chiami due volte di seguito lo stesso id valido? Con quali codici di stato risponde la prima e la seconda volta?

    Sezione 7.7Soluzioni

    Soluzione 7.1
    CRUD di /libri
    GET    /libri       # elenca            -> 200
POST   /libri       # crea              -> 201
GET    /libri/{id}  # leggi uno         -> 200 / 404
PUT    /libri/{id}  # aggiorna          -> 200 / 404
DELETE /libri/{id}  # cancella          -> 204 / 404
    Soluzione 7.2

    Perché comunicano l'esito in modo più preciso (Cap. 2): 201 Created dice «ho creato una nuova risorsa» (spesso restituendola), più informativo di un generico 200; 204 No Content dice «operazione riuscita, e non c'è nulla da restituire» — perfetto per una cancellazione, dopo la quale la risorsa non esiste più. Usare il codice giusto rende l'API chiara e conforme alle convenzioni che i client si aspettano.

    Soluzione 7.3

    raise HTTPException(...) interrompe immediatamente la funzione e fa sì che FastAPI invii una risposta di errore con il codice indicato (es. 404) e il messaggio detail. Un normale return, invece, produce una risposta di successo (200 di default) con i dati restituiti. In sintesi: return = «è andata bene, ecco il risultato»; raise HTTPException = «c'è un problema, ecco il codice d'errore e la spiegazione».

    Soluzione 7.4

    La prima volta il contatto esiste: viene cancellato e la risposta è 204 No Content. La seconda volta quell'id non è più nel DB, quindi scatta il controllo if id not in DB e la risposta è 404 Not Found. È un comportamento corretto e prevedibile: la cancellazione è «andata a buon fine» una sola volta, e riprovarci segnala onestamente che non c'è più nulla da cancellare.
