Capitolo 8

I database relazionali

La nostra rubrica funziona, ma ha un difetto fatale: i dati vivono in un dizionario Python e svaniscono a ogni riavvio del server. Un backend serio deve ricordare per sempre, gestire milioni di record, cercarli in un lampo e non perderli mai. Per questo esistono i database. La famiglia più diffusa, e quella che useremo, è quella dei database relazionali: organizzano i dati in tabelle — proprio come fogli di calcolo, con righe e colonne — collegabili tra loro. E hanno un linguaggio dedicato per interrogarli e modificarli, lo SQL, uno dei linguaggi più longevi e universali dell'informatica. In questo capitolo capisci perché servono (§8.1), cosa sono tabelle, righe e colonne (§8.2), le chiavi e i tipi (§8.3), e i quattro comandi SQL fondamentali (§8.4). Sono le fondamenta su cui, dal Capitolo 10, poggeremo la persistenza vera del nostro backend.

🗺 Dove siamo

Perché un database (§8.1), tabelle, righe, colonne (§8.2), chiavi primarie e tipi (§8.3), il linguaggio SQL — SELECT, INSERT, UPDATE, DELETE (§8.4). Nel laboratorio (§8.5) esegui SQL vero con SQLite, incluso nel tuo Python.

🔑 Perché ci tocca: i dati sono il cuore del backend

Un backend, in fondo, è una macchina per custodire e servire dati. Il database è il luogo dove quei dati vivono davvero: saper ragionare in termini di tabelle e SQL è una competenza fondamentale e duratura — l'SQL ha più di quarant'anni ed è ancora ovunque. Anche quando (dal Cap. 10) useremo SQLAlchemy per non scrivere SQL a mano, capire cosa succede «sotto» ti renderà infinitamente più efficace.

Sezione 8.1Perché un database

Potremmo salvare i dati in un file (un JSON, un CSV). Funziona per pochi dati e un solo utente, ma crolla in fretta. Un database risolve una lista di problemi che, altrimenti, dovresti affrontare da solo — e male.

Cosa ti dà un database (e un file no) i problemi che risolve
Persistenza affidabile
i dati restano su disco, in sicurezza, anche se il server si riavvia o va in crash a metà di un'operazione. Un database garantisce di non lasciare i dati a metà (le «transazioni»).
Ricerca veloce
trovare «l'utente con email X» tra milioni di righe è istantaneo, grazie agli indici. In un file dovresti scorrere tutto ogni volta.
Accessi simultanei
tanti utenti (e tante richieste) leggono e scrivono nello stesso momento senza corrompere i dati. Gestire questo a mano su un file è quasi impossibile.
Integrità dei dati
il database impone regole: questo campo è obbligatorio, quell'id dev'essere unico, questo valore deve esistere in un'altra tabella. Impedisce dati incoerenti.
Interrogazioni potenti
con SQL fai domande complesse («i 10 clienti che hanno speso di più questo mese») in una riga. Con un file scriveresti codice su codice.
Per questo praticamente ogni applicazione seria usa un database. Ne esistono due grandi famiglie: i relazionali (tabelle + SQL: PostgreSQL, MySQL, SQLite…), i più diffusi e quelli di questo libro; e i NoSQL (documenti, chiavi-valore: MongoDB, Redis…), utili in casi specifici. Noi ci concentriamo sui relazionali, che sono la scelta di default per la stragrande maggioranza dei backend.

Sezione 8.2Tabelle, righe, colonne

Il modello relazionale è sorprendentemente intuitivo: se hai mai usato un foglio di calcolo, sei già a metà strada. I dati stanno in tabelle, fatte di righe e colonne.

Una tabella "utenti" righe = record, colonne = campi
tabella: utenti
 id | nome   | email             | eta
----+--------+-------------------+-----
  1 | Anna   | anna@esempio.com  |  30     <- una RIGA (un utente)
  2 | Luca   | luca@esempio.com  |  25
  3 | Sara   | sara@esempio.com  |  41
    ^      ^
   colonna colonna
la tabella
raccoglie tutte le entità dello stesso tipo (qui, tutti gli utenti). Un database ha molte tabelle (utenti, prodotti, ordini…). Una tabella ≈ una «risorsa» REST (Cap. 7).
la riga (record)
una singola entità: un utente, con tutti i suoi dati. Aggiungere un utente = aggiungere una riga. Una riga ≈ un oggetto/dizionario (come i Contatto del Cap. 7).
la colonna (campo)
un attributo, uguale per tutte le righe: nome, email, eta. Ogni colonna ha un tipo fisso (§8.3). Le colonne ≈ i campi di un modello Pydantic.
Riconosci le corrispondenze? Tabella ↔ risorsa/modello, riga ↔ oggetto, colonna ↔ campo. Questa mappatura è la chiave di tutto: è esattamente ciò che, dal Cap. 10, SQLAlchemy renderà automatica, permettendoti di lavorare con oggetti Python che «sono» righe del database. Lo schema di una tabella è la definizione delle sue colonne e dei loro tipi: il progetto della tabella.

Sezione 8.3Chiavi primarie e tipi

Due dettagli rendono una tabella robusta: un modo univoco per identificare ogni riga (la chiave primaria) e un tipo per ogni colonna, che il database fa rispettare.

Chiave primaria e tipi delle colonne identità + disciplina dei dati
la chiave primaria (primary key)
una colonna (di solito id) il cui valore è unico per ogni riga: nessun due righe possono avere lo stesso id. È il modo con cui identifichi e ritrovi una riga precisa — l'id di /contatti/42. Di solito il database la assegna da solo, incrementale (auto-increment): 1, 2, 3…
i tipi delle colonne
ogni colonna dichiara che dati contiene: INTEGER (interi), TEXT/VARCHAR (testo), REAL/FLOAT (decimali), BOOLEAN, DATE/TIMESTAMP (date e orari). Il database rifiuta dati del tipo sbagliato, come Pydantic per le API.
vincoli aggiuntivi
puoi imporre altre regole: NOT NULL (obbligatorio), UNIQUE (nessun duplicato — es. sulle email), DEFAULT (valore predefinito). Sono le «regole di integrità» del §8.1.
Nota il parallelo con Pydantic (Cap. 6): là dichiaravi tipi e vincoli per validare i dati in ingresso dall'API; qui il database dichiara tipi e vincoli per proteggere i dati a riposo. Due linee di difesa: l'API valida ciò che entra, il database garantisce la coerenza di ciò che è salvato. Un backend robusto le usa entrambe.

Sezione 8.4Il linguaggio SQL

Ai database relazionali si parla in SQL (Structured Query Language). Quattro comandi coprono il CRUD del Cap. 7 — e li riconoscerai subito. L'SQL è dichiarativo: dici cosa vuoi, non come ottenerlo.

i quattro comandi fondamentali
-- CREARE una riga (Create):
INSERT INTO utenti (nome, email, eta) VALUES ('Anna', 'anna@x.com', 30);

-- LEGGERE righe (Read): tutte, o filtrate con WHERE
SELECT * FROM utenti;                          -- tutte le colonne, tutte le righe
SELECT nome, eta FROM utenti WHERE eta > 28;    -- solo alcune, filtrate

-- AGGIORNARE righe (Update):
UPDATE utenti SET eta = 31 WHERE id = 1;

-- CANCELLARE righe (Delete):
DELETE FROM utenti WHERE id = 2;
SQL smontato SELECT · INSERT · UPDATE · DELETE · WHERE
SELECT colonne FROM tabella
legge dati. * = «tutte le colonne»; oppure elenchi i nomi. Il comando che userai di più.
WHERE condizione
il filtro: applica il comando solo alle righe che soddisfano la condizione (eta > 28, id = 1, nome = 'Anna'). Vale per SELECT, UPDATE e DELETE.
INSERT INTO tabella (...) VALUES (...)
aggiunge una riga: elenchi le colonne e i valori corrispondenti.
UPDATE ... SET ... WHERE
modifica le righe che soddisfano il WHERE. Dimenticare il WHERE aggiorna tutte le righe! (vedi la trappola sotto).
DELETE FROM tabella WHERE
cancella le righe che soddisfano il WHERE.
⚠ La trappola del WHERE dimenticato

DELETE FROM utenti; (senza WHERE) cancella tutti gli utenti. Allo stesso modo, UPDATE utenti SET eta = 0; azzera l'età di tutti. Il WHERE mancante è uno degli errori più temuti e costosi nel lavoro con i database — c'è chi ha svuotato tabelle di produzione così. Prendi l'abitudine di scrivere prima il WHERE, poi il resto. La buona notizia: dal Cap. 10, lavorando con SQLAlchemy sugli oggetti, questo rischio si riduce moltissimo.

Sezione 8.5Laboratorio: SQL con SQLite


  

    Per esercitarti con SQL vero non serve installare nulla: Python include SQLite, un database
    relazionale completo che vive in un singolo file. È perfetto per imparare e per piccoli progetti (lo useremo anche
    nei primi capitoli con SQLAlchemy).
  


  

    
terminale — SQLite dal modulo Python

    
# apri una console SQL su un file di database (lo crea se non esiste)
$ python -m sqlite3 rubrica.db
SQLite version 3...  Enter ".help" for usage hints.
sqlite>

  


  

    
alla console sqlite> — crea, inserisci, interroga

    
sqlite> CREATE TABLE utenti (
   ...>   id INTEGER PRIMARY KEY AUTOINCREMENT,
   ...>   nome TEXT NOT NULL,
   ...>   eta INTEGER
   ...> );
sqlite> INSERT INTO utenti (nome, eta) VALUES ('Anna', 30);
sqlite> INSERT INTO utenti (nome, eta) VALUES ('Luca', 25);
sqlite> SELECT * FROM utenti;
1|Anna|30
2|Luca|25
sqlite> SELECT nome FROM utenti WHERE eta > 28;
Anna
sqlite> .quit                        -- esci

  


  

    
✅ Cosa hai fatto

    

      Hai creato una tabella (con CREATE TABLE) definendo colonne, tipi, chiave primaria
      auto-incrementale e un vincolo NOT NULL; hai inserito righe; le hai interrogate, anche
      con un filtro. Nota che gli id (1, 2) sono stati assegnati dal database da solo, grazie a
      AUTOINCREMENT. Tutto è salvato nel file rubrica.db: chiudi, riapri con
      python -m sqlite3 rubrica.db, rifai la SELECT — i dati sono ancora lì. Questa è la
      persistenza che mancava alla nostra rubrica. Sperimenta con UPDATE e DELETE: sono
      i comandi che, tra due capitoli, azioneremo dal codice Python.

Sezione 8.7Soluzioni


  

    
Soluzione 8.1

    

      Tra i vantaggi: (1) ricerca veloce con gli indici (trovare una riga tra milioni è istantaneo);
      (2) accessi simultanei sicuri (tanti utenti scrivono insieme senza corrompere i dati); (3)
      integrità (vincoli come UNIQUE/NOT NULL impediscono dati incoerenti); inoltre persistenza affidabile con
      transazioni e interrogazioni complesse in SQL. Un file JSON non offre nulla di tutto ciò senza enorme lavoro
      manuale.
    

  


  

    
Soluzione 8.2

    

      Una tabella corrisponde a una risorsa (o a un modello): es. la tabella contatti ↔ la
      risorsa /contatti. Una riga corrisponde a un singolo oggetto (un contatto, come le
      istanze del modello Contatto). Una colonna corrisponde a un campo (nome, telefono,
      email — come i campi del modello Pydantic).
    

  


  

    
Soluzione 8.3

    

      
SQL

      
-- (a)
SELECT nome, email FROM utenti WHERE eta >= 18;
-- (b)
DELETE FROM utenti WHERE id = 5;

    

  


  

    
Soluzione 8.4

    

      Senza WHERE, UPDATE utenti SET eta = 40; imposta l'età a 40 per tutte le righe
      della tabella — un disastro se non voluto. È pericoloso proprio perché sembra un comando innocuo ma agisce su
      tutto. Correzione: aggiungere il filtro →
      UPDATE utenti SET eta = 40 WHERE id = 3;. Regola: con UPDATE e
      DELETE, scrivi sempre il WHERE (a meno che tu non voglia davvero agire su tutte
      le righe).
