Capitolo 10

SQLAlchemy: l'ORM

Nel Capitolo 8 hai scritto SQL a mano. Funziona, ma nel codice di un backend diventa scomodo e rischioso: stringhe SQL infilate nelle funzioni, risultati da «cucire» a mano dentro oggetti Python, dialetti diversi tra Postgres e MySQL, e — pericolo serio — la porta aperta a un tipo di attacco chiamato SQL injection. La soluzione elegante è un ORM (Object-Relational Mapping): uno strato che fa corrispondere classi Python ↔ tabelle e oggetti ↔ righe, generando l'SQL per te. Scrivi Python normale — crei oggetti, li aggiungi, li interroghi con metodi — e l'ORM traduce tutto in SQL, nel dialetto giusto per il tuo database. In Python lo strumento standard è SQLAlchemy. In questo capitolo capisci cos'è un ORM (§10.1), come aprire la connessione con l'engine (§10.2), come definire i modelli (§10.3) e come leggere e scrivere tramite la sessione (§10.4). È il ponte che, nel prossimo capitolo, unirà la nostra API FastAPI a un database vero e persistente.

🗺 Dove siamo

Cos'è un ORM (§10.1), l'engine e la connessione (§10.2), i modelli come classi (§10.3), la sessione per leggere/scrivere (§10.4). Nel laboratorio (§10.5) definisci un modello e fai le tue prime query dagli oggetti.

🔑 Perché ci tocca: è come lavoreremo coi dati d'ora in poi

Da qui in avanti non scriveremo quasi più SQL a mano: lavoreremo con oggetti Python, e SQLAlchemy si occuperà del database. È il modo standard di costruire backend Python: più leggibile, più sicuro (niente SQL injection), e portabile (lo stesso codice su SQLite in sviluppo e Postgres in produzione). Investire qui rende semplici e piacevoli tutti i capitoli sui dati che seguono.

Sezione 10.1Cos'è un ORM

ORM sta per Object-Relational Mapping: «mappatura oggetti-relazionale». È lo strato che collega due mondi — gli oggetti del tuo codice Python e le tabelle del database relazionale — traducendo automaticamente tra loro.

La mappatura dell'ORM i due mondi che si corrispondono
classe Python ↔ tabella
una classe (es. Utente) rappresenta una tabella (utenti). La definisci una volta, in Python.
oggetto (istanza) ↔ riga
un oggetto Utente(nome="Anna") rappresenta una riga della tabella. Crearne uno e «salvarlo» = una INSERT.
attributo ↔ colonna
utente.nome corrisponde alla colonna nome. Leggere/cambiare l'attributo = leggere/aggiornare la colonna.
le query ↔ SELECT
chiedi i dati con metodi Python (.filter(...), .all()) e l'ORM genera la SELECT giusta, ti restituisce oggetti Python già pronti.
Riconosci lo schema? È lo stesso parallelo del Cap. 8 (tabella/riga/colonna ↔ risorsa/oggetto/campo), ma ora automatizzato. I vantaggi concreti: (1) scrivi Python, non stringhe SQL; (2) l'ORM usa il dialetto giusto per il tuo database (SQLite, Postgres, MySQL) — codice portabile; (3) protegge da SQL injection, perché non concateni mai input dell'utente in stringhe SQL. Lo svantaggio: uno strato in più da capire — ma i benefici lo ripagano ampiamente, e per i casi rari puoi sempre scendere a SQL grezzo.
Da oggetto Python a SQL l'ORM traduce, tu scrivi Python

Sezione 10.2Engine e connessione


  

    Il primo pezzo di SQLAlchemy è l'engine: l'oggetto che gestisce la connessione al database. Lo crei
    una volta, passandogli la stringa di connessione del Cap. 9.
  


  

    
database.py

    
from sqlalchemy import create_engine
from sqlalchemy.orm import sessionmaker, declarative_base

# la stringa di connessione (qui SQLite, un file locale)
DATABASE_URL = "sqlite:///./rubrica.db"

engine = create_engine(DATABASE_URL)         # la "connessione" al database
SessionLocal = sessionmaker(bind=engine)  # fabbrica di sessioni (§10.4)
Base = declarative_base()                   # base per i modelli (§10.3)

  


  

    
I tre oggetti fondamentali engine · SessionLocal · Base

    

      

        
engine = create_engine(URL)

        
l'oggetto che sa come raggiungere il database, a partire dalla stringa di connessione.
            Cambiando solo l'URL passi da SQLite a Postgres senza toccare altro — la portabilità
            promessa. Per Postgres: "postgresql://utente:pw@host:5432/db".

        
SessionLocal = sessionmaker(bind=engine)

        
una fabbrica di sessioni: la userai per aprire una «conversazione» col database a ogni operazione
            (§10.4). Legata all'engine.

        
Base = declarative_base()

        
la classe base da cui erediteranno tutti i tuoi modelli (§10.3). Tiene il «registro» delle tabelle
            che definisci.

      

    

    

      Questo file database.py è l'infrastruttura: lo scrivi una volta e lo importi ovunque serva.
      La riga DATABASE_URL è l'unico punto che cambia tra sviluppo e produzione — e in un
      progetto vero non la scriveresti «in chiaro», ma la leggeresti da una variabile d'ambiente (Cap. 9), per non
      mettere la password su git.
    

  



  
  
Sezione 10.3I modelli: classi = tabelle


  

    Un modello SQLAlchemy è una classe che descrive una tabella: eredita da Base, dichiara il nome
    della tabella e le sue colonne. È il «progetto» della tabella, scritto in Python.
  


  

    
models.py

    
from sqlalchemy import Column, Integer, String
from database import Base

class Contatto(Base):
    __tablename__ = "contatti"          # nome della tabella nel database

    id = Column(Integer, primary_key=True, index=True)
    nome = Column(String, nullable=False)   # NOT NULL
    telefono = Column(String)
    email = Column(String, unique=True)     # UNIQUE

  


  

    
Anatomia di un modello __tablename__ · Column · vincoli

    

      

        
class Contatto(Base)

        
eredita da Base: così SQLAlchemy sa che è una tabella. Il nome della classe è la
            convenzione singolare (Contatto), il nome della tabella il plurale.

        
__tablename__ = "contatti"

        
il nome reale della tabella nel database. Obbligatorio.

        
id = Column(Integer, primary_key=True)

        
una colonna: tipo Integer, chiave primaria. SQLAlchemy la rende auto-incrementale
            (come SERIAL/AUTOINCREMENT del Cap. 8-9, ma senza pensarci). index=True
            la rende veloce da cercare.

        
i tipi e i vincoli

        
Integer, String, Float, Boolean, DateTime…
            più vincoli come nullable=False (obbligatorio), unique=True (senza duplicati).
            Sono i tipi e i vincoli SQL del Cap. 8, dichiarati in Python.

      

    

    

      Nota le tre facce ormai familiari: il modello Pydantic (Cap. 6) valida i dati dell'API; il modello
      SQLAlchemy (questo) descrive la tabella del database; e la tabella SQL è ciò che ne risulta. Nel
      prossimo capitolo li useremo insieme: Pydantic per l'input/output dell'API, SQLAlchemy per la
      persistenza. Una volta definiti i modelli, questa riga crea le tabelle nel database:
      Base.metadata.create_all(bind=engine).
    

  



  
  
Sezione 10.4La sessione


  

    La sessione è la «conversazione» con il database: la apri, fai le tue operazioni (aggiungi, interroga,
    modifica), e la chiudi. È attraverso la sessione che oggetti Python e righe del database si scambiano.
  


  

    
esempi d'uso della sessione

    
from database import SessionLocal
from models import Contatto

db = SessionLocal()          # apri la sessione

# CREARE: costruisci un oggetto, aggiungilo, conferma
c = Contatto(nome="Anna", telefono="333-111")
db.add(c)                   # "prepara" l'inserimento
db.commit()                 # CONFERMA: scrive davvero nel database (INSERT)
db.refresh(c)               # ricarica c: ora c.id è valorizzato dal DB

# LEGGERE uno per id:
uno = db.query(Contatto).filter(Contatto.id == 1).first()

# LEGGERE con filtro, tutti i risultati:
lista = db.query(Contatto).filter(Contatto.nome == "Anna").all()

db.close()                  # chiudi la sessione

  


  

    
Il vocabolario della sessione add · commit · query · filter · first/all

    

      

        
db.add(oggetto)

        
«prepara» l'inserimento di un oggetto (una nuova riga). Non scrive ancora nel database: lo mette in coda.

        
db.commit()

        
conferma e rende permanenti le modifiche (esegue davvero le INSERT/UPDATE/DELETE in
            coda). Fondamentale: senza commit, le modifiche vanno perse. È la «transazione».

        
db.query(Modello)

        
inizia una interrogazione su quella tabella. Ci concateni i filtri.

        
.filter(Contatto.nome == "Anna")

        
il WHERE dell'ORM: condizioni scritte come espressioni Python. Componibili:
            .filter(...).filter(...).

        
.first() / .all()

        
eseguono la query: first() restituisce il primo oggetto (o None),
            all() la lista di tutti. Ottieni oggetti Contatto, non righe grezze.

      

    

    

      Il ritmo è sempre lo stesso: apri la sessione, operi, commit se hai modificato, chiudi.
      Modificare è naturale: uno.telefono = "999"; db.commit() genera l'UPDATE; cancellare:
      db.delete(uno); db.commit(). Aprire e chiudere una sessione a ogni richiesta è un lavoro
      ripetitivo — e nel prossimo capitolo vedremo come FastAPI lo automatizza elegantemente con la
      dependency injection, così non dovrai scriverlo ogni volta.
    

  



  
  
Sezione 10.5Laboratorio: primo modello e query


  

    Mettiamo insieme i tre file e facciamo un giro completo: crea, leggi, filtra — tutto da Python, con SQLite (nessun
    server da avviare). Installa prima SQLAlchemy: pip install sqlalchemy.
  


  

    
prova.py — un giro completo

    
from sqlalchemy import create_engine, Column, Integer, String
from sqlalchemy.orm import sessionmaker, declarative_base

engine = create_engine("sqlite:///./prova.db")
SessionLocal = sessionmaker(bind=engine)
Base = declarative_base()

class Contatto(Base):
    __tablename__ = "contatti"
    id = Column(Integer, primary_key=True)
    nome = Column(String, nullable=False)

Base.metadata.create_all(bind=engine)   # crea la tabella se non esiste

db = SessionLocal()
db.add(Contatto(nome="Anna"))
db.add(Contatto(nome="Luca"))
db.commit()

for c in db.query(Contatto).all():
    print(c.id, c.nome)
db.close()

  


  

    
terminale — eseguilo

    
(.venv) $ python prova.py
1 Anna
2 Luca

  


  

    
✅ Cosa è successo (e cosa non hai scritto)

    

      Hai definito una tabella senza scrivere CREATE TABLE, inserito righe senza scrivere
      INSERT, e letto i dati senza scrivere SELECT: SQLAlchemy ha generato tutto
      l'SQL per te. Gli id (1, 2) sono stati assegnati dal database. E il file
      prova.db ora contiene i dati in modo permanente: rilancia lo script (togliendo gli
      add) e i contatti sono ancora lì. Prova a cambiare l'URL in un Postgres (Cap. 9): lo stesso codice
      funzionerà, scrivendo lì invece che nel file. Questa è la potenza dell'ORM.

Sezione 10.7Soluzioni


  

    
Soluzione 10.1

    

      Una classe corrisponde a una tabella (es. classe Contatto ↔ tabella
      contatti); un oggetto (istanza della classe) corrisponde a una riga (un contatto
      specifico); un attributo dell'oggetto corrisponde a una colonna (contatto.nome ↔
      colonna nome).
    

  


  

    
Soluzione 10.2

    

      Tra i vantaggi: (1) scrivi Python leggibile invece di stringhe SQL sparse nel codice, con
      l'autocompletamento e il controllo dell'editor; (2) portabilità: lo stesso codice funziona su
      SQLite, Postgres o MySQL cambiando solo l'URL, perché l'ORM genera il dialetto giusto; inoltre protezione da
      SQL injection, perché non concateni mai input in stringhe SQL.
    

  


  

    
Soluzione 10.3

    

      
models.py

      
from sqlalchemy import Column, Integer, String, Float
from database import Base

class Prodotto(Base):
    __tablename__ = "prodotti"
    id = Column(Integer, primary_key=True)
    nome = Column(String, nullable=False)
    prezzo = Column(Float)

    

  


  

    
Soluzione 10.4

    

      Senza db.commit(), l'inserimento resta «in sospeso» nella sessione e non viene scritto nel
      database: chiudendo la sessione (o riavviando) la modifica va persa. Il commit è ciò che
      rende permanenti le modifiche. Quanto al filtro: .filter(Prodotto.prezzo < 10) è preferibile
      perché è Python controllabile dall'editor, portabile tra database e sicuro (l'ORM gestisce i
      valori senza concatenarli nella stringa SQL, evitando la SQL injection).
