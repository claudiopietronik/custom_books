Capitolo 13

Autenticazione e sicurezza

Finora la nostra API è stata «aperta»: chiunque può creare, leggere e cancellare qualsiasi dato. Va bene per imparare, ma nessuna applicazione reale funziona così. Serve sapere chi sta facendo una richiesta (autenticazione) e cosa gli è permesso (autorizzazione). È il tema più delicato del backend, perché qui si custodiscono le identità e i segreti degli utenti: sbagliare significa mettere a rischio persone reali. In questo capitolo affrontiamo i pilastri, in modo difensivo e corretto: perché l'identità è un problema (§13.1), come conservare le password senza mai salvarle in chiaro — l'hashing (§13.2); come riconoscere un utente tra una richiesta e l'altra con i token (§13.3); come FastAPI struttura tutto questo con OAuth2 (§13.4); e come proteggere le rotte perché solo gli utenti autenticati vi accedano (§13.5). Sono le pratiche che distinguono un giocattolo da un'applicazione di cui ci si può fidare.

🗺 Dove siamo

Il problema dell'identità (§13.1), l'hashing delle password (§13.2), i token e JWT (§13.3), OAuth2 in FastAPI (§13.4), proteggere le rotte (§13.5). Nel laboratorio (§13.6) costruisci un flusso di login con token.

🔑 Perché ci tocca: qui si proteggono persone reali

La sicurezza non è una funzionalità «in più»: è una responsabilità. I dati che custodisci — password, messaggi, informazioni personali — appartengono a persone che si fidano di te. Un errore qui non è un bug qualsiasi: può esporre gli account e la privacy degli utenti. Per questo esistono pratiche consolidate, da seguire sempre — mai improvvisare la sicurezza. Impararle è un dovere, oltre che una competenza molto richiesta.

Sezione 13.1Il problema dell'identità

Ricordi che HTTP è stateless (Cap. 2): ogni richiesta è indipendente, il server non «ricorda» le precedenti. Ma allora, se un utente fa il login e poi chiede la sua pagina, come fa il server a sapere che è ancora lui? Questo è il problema dell'identità.

📖 Autenticazione vs autorizzazione

Due parole simili, due concetti distinti: autenticazione = «chi sei?» (verificare l'identità, es. con email + password); autorizzazione = «cosa puoi fare?» (verificare i permessi, es. «solo l'autore può cancellare il proprio post»). Prima ti autentichi (dimostri chi sei), poi il server autorizza (o nega) le tue azioni. Il flusso tipico: l'utente fa login una volta; il server gli dà una «prova d'identità» (un token, §13.3); l'utente allega quella prova a ogni richiesta successiva, così il server lo riconosce senza richiedere la password ogni volta.

Sezione 13.2Password: mai in chiaro

La regola d'oro assoluta, non negoziabile: non salvare mai le password in chiaro nel database. Se lo facessi, chiunque riesca a leggere il database (un attacco, un errore, un dipendente disonesto) avrebbe subito tutte le password dei tuoi utenti — che spesso le riutilizzano altrove.

🔒 L'hashing: una direzione sola

La soluzione è l'hashing: si applica alla password una funzione matematica speciale che la trasforma in una stringa apparentemente casuale (l'hash), con una proprietà cruciale: è a senso unico — dalla password ottieni l'hash, ma dall'hash non puoi risalire alla password. Nel database salvi solo l'hash, mai la password.

Come si verifica il login, allora? Quando l'utente inserisce la password, la si hasha di nuovo e si confronta il risultato con l'hash salvato: se combaciano, la password è giusta. Il server verifica la password senza mai conservarla. Anche se qualcuno rubasse il database, troverebbe solo hash inutili.

⚠ Non tutti gli hash sono adatti — usa quelli giusti

Per le password si usano algoritmi lenti e con «sale», pensati apposta, come bcrypt o Argon2 — non algoritmi veloci come MD5 o SHA-1 (inadatti alle password). Il «sale» (salt) è un valore casuale aggiunto a ogni password prima dell'hash, così due utenti con la stessa password ottengono hash diversi, e si vanificano gli attacchi con tabelle precalcolate. Non implementare l'hashing a mano: usa una libreria collaudata. In Python, passlib con bcrypt: pip install "passlib[bcrypt]".

hashing e verifica con passlib
from passlib.context import CryptContext

pwd = CryptContext(schemes=["bcrypt"], deprecated="auto")

# al momento della registrazione: salva SOLO l'hash
hash_salvato = pwd.hash("password_segreta")
# -> "$2b$12$Kx9...."  (una stringa che NON rivela la password)

# al login: verifica senza mai conservare la password
pwd.verify("password_segreta", hash_salvato)   # -> True
pwd.verify("sbagliata", hash_salvato)          # -> False
Hashing di una password (dimostrativo) una direzione sola: password → hash

Sezione 13.3Token e JWT

Dopo il login, il server deve riconoscere l'utente nelle richieste successive senza richiedere ogni volta la password. Lo fa consegnandogli un token: una «tessera» temporanea che l'utente allega a ogni richiesta.

Il flusso a token login → token → richieste autenticate
1 · login
l'utente invia email + password una volta. Il server verifica (con l'hash, §13.2) e, se è corretto, genera un token e glielo restituisce.
2 · il token
una stringa firmata che «certifica» l'identità e ha una scadenza. Il formato standard è il JWT (JSON Web Token): contiene dati (es. l'id utente) e una firma del server. La firma impedisce di falsificarlo: solo il server, con la sua chiave segreta, può crearne di validi.
3 · richieste autenticate
a ogni richiesta successiva, l'utente allega il token nell'header Authorization: Bearer <token>. Il server verifica la firma e sa chi è, senza rivedere la password. Alla scadenza, si rifà il login.
Perché un token e non «ricordare» il login lato server? Perché così il server resta stateless (Cap. 2): non deve tenere in memoria chi è loggato — tutta l'informazione è nel token firmato, che il client porta con sé. Questo rende il backend più semplice da scalare. Attenzione: il contenuto di un JWT è leggibile (è solo firmato, non cifrato), quindi non metterci mai dati sensibili; e va trasmesso solo su HTTPS (Cap. 17), per non farlo intercettare.

Sezione 13.4OAuth2 in FastAPI

FastAPI offre strumenti già pronti per l'autenticazione a token, seguendo lo standard OAuth2 (uno schema diffuso per gestire login e permessi). Non devi inventare nulla: usi i suoi mattoni.

auth.py — l'endpoint di login (semplificato)
from fastapi import Depends, HTTPException
from fastapi.security import OAuth2PasswordBearer, OAuth2PasswordRequestForm

oauth2 = OAuth2PasswordBearer(tokenUrl="login")   # dove si ottiene il token

@app.post("/login")
def login(form: OAuth2PasswordRequestForm = Depends()):
    utente = trova_utente(form.username)
    if not utente or not pwd.verify(form.password, utente.hash):
        raise HTTPException(status_code=401, detail="Credenziali errate")
    token = crea_token({"sub": utente.email})       # genera il JWT firmato
    return {"access_token": token, "token_type": "bearer"}
I mattoni OAuth2 di FastAPI OAuth2PasswordBearer · RequestForm
OAuth2PasswordBearer(tokenUrl="login")
configura lo schema: dice a FastAPI (e a /docs) che i token si ottengono da /login e viaggiano nell'header Authorization: Bearer. Lo useremo come dipendenza per proteggere le rotte (§13.5).
OAuth2PasswordRequestForm = Depends()
riceve username e password dal form di login, in modo standard. FastAPI ti consegna un oggetto form con quei campi.
la verifica
trovi l'utente, controlli la password con pwd.verify (§13.2); se fallisce, 401 Unauthorized (Cap. 2). Se va bene, generi e restituisci il token.
Il token JWT si crea e si verifica con una libreria dedicata (es. python-jose o pyjwt): funzioni come crea_token firmano un dizionario con una chiave segreta del server e una scadenza. Quella chiave segreta è il cuore della sicurezza: va tenuta fuori dal codice (variabile d'ambiente, come le password del database, Cap. 9). Non serve memorizzare ogni dettaglio: la cosa importante è capire il flusso e sapere che FastAPI fornisce i pezzi standard.

Sezione 13.5Proteggere le rotte

Ecco dove tutto si unisce: rendere un endpoint accessibile solo agli utenti autenticati. Si fa con una dipendenza (come get_db, Cap. 11) che estrae l'utente dal token — e blocca chi non ce l'ha.

proteggere un endpoint
def utente_corrente(token: str = Depends(oauth2)):
    dati = verifica_token(token)         # controlla la firma e la scadenza
    if dati is None:
        raise HTTPException(status_code=401, detail="Non autenticato")
    return trova_utente(dati["sub"])

@app.get("/profilo")
def profilo(utente = Depends(utente_corrente)):   # protetto!
    return {"email": utente.email, "nome": utente.nome}
La dipendenza che protegge Depends(utente_corrente)
utente_corrente(token = Depends(oauth2))
una dipendenza che estrae il token dall'header, ne verifica firma e scadenza, e restituisce l'utente. Se il token manca o è invalido, solleva 401 e la richiesta è respinta.
def profilo(utente = Depends(utente_corrente))
basta aggiungere questa dipendenza a un endpoint per proteggerlo: FastAPI eseguirà prima utente_corrente. Se l'utente non è autenticato, la funzione profilo non viene nemmeno eseguita — esattamente come la validazione dei tipi (Cap. 5).
bonus: hai l'utente pronto
dentro l'endpoint, utente è già l'utente loggato: lo usi per l'autorizzazione («questo utente può modificare questo dato?») e per personalizzare la risposta.
Riconosci il pattern? È la stessa dependency injection di get_db: una funzione che «prepara» qualcosa e lo inietta nell'endpoint. Qui prepara l'identità dell'utente. Aggiungere Depends(utente_corrente) a un endpoint è tutto ciò che serve per proteggerlo — pulito, riusabile, difficile da dimenticare. È l'eleganza di FastAPI applicata alla sicurezza.

🔑 Quello che devi portarti dietro dal Capitolo 13
  1. Autenticazione = «chi sei»; autorizzazione = «cosa puoi fare». HTTP è stateless: serve una «prova d'identità» a ogni richiesta (§13.1).
  2. Le password si salvano solo come hash (a senso unico), con algoritmi appositi (bcrypt/Argon2) e «sale». Mai in chiaro. Usa una libreria (§13.2).
  3. Dopo il login, il server dà un token (JWT firmato, con scadenza); il client lo allega come Authorization: Bearer. Non contiene segreti; solo su HTTPS (§13.3).
  4. FastAPI offre i mattoni OAuth2 per login e token (§13.4); si protegge una rotta con una dipendenza Depends(utente_corrente) che dà 401 se il token manca/è invalido (§13.5).

La nostra applicazione ora sa chi sono gli utenti e li protegge. Le manca ancora una capacità pratica molto richiesta: gestire i file. Un'immagine del profilo, un allegato, un documento — dove si mettono? Non nel database (vedremo perché). Nel Capitolo 14 conosciamo lo storage di oggetti e MinIO, il modo giusto di caricare, conservare e servire file in un backend — l'ultimo pezzo applicativo prima di impacchettare e mettere online tutto ciò che abbiamo costruito.

Sezione 13.7Esercizi

Esercizio 13.1

Distingui con un esempio autenticazione e autorizzazione. Quale delle due risponde alla domanda «questo utente può cancellare questo post?».

Esercizio 13.2

Perché non si devono salvare le password in chiaro? Come fa il server a verificare il login se conserva solo l'hash e non la password?

Esercizio 13.3

Nel flusso a token: dopo il login, dove viaggia il token nelle richieste successive? Perché il server non ha bisogno di «ricordare» chi è loggato?

Esercizio 13.4

Perché non si devono mettere dati sensibili (es. una password) dentro un JWT? E perché la chiave segreta con cui si firmano i token non va scritta nel codice?

Sezione 13.8Soluzioni

Soluzione 13.1

Autenticazione: verificare l'identità — es. accedere con email e password dimostra «sono Anna». Autorizzazione: verificare i permessi — es. «Anna può cancellare solo i propri post, non quelli altrui». La domanda «questo utente può cancellare questo post?» riguarda l'autorizzazione (i permessi su una risorsa specifica), e presuppone che l'autenticazione sia già avvenuta.

Soluzione 13.2

Perché se il database venisse letto da un malintenzionato, avrebbe subito tutte le password degli utenti (che spesso le riusano su altri siti): un danno enorme. Salvando solo l'hash (a senso unico), un eventuale furto del database non rivela le password. Al login, il server ri-hasha la password inserita e la confronta con l'hash salvato: se combaciano, la password è corretta — il tutto senza mai conservare la password in chiaro.

Soluzione 13.3

Il token viaggia nell'header Authorization: Bearer <token> di ogni richiesta successiva. Il server non deve ricordare chi è loggato perché tutta l'informazione è nel token firmato che il client porta con sé: il server verifica la firma (con la sua chiave segreta) e sa chi è. Questo mantiene il backend stateless — più semplice e scalabile.

Soluzione 13.4

Perché il contenuto di un JWT è firmato ma non cifrato: chiunque abbia il token può leggerne i dati (sono codificati, non segreti). Quindi non vi si mettono password o informazioni riservate — solo dati non sensibili come l'id o l'email. La chiave segreta di firma non va nel codice/git perché è ciò che permette di creare token validi: se trapelasse, un attaccante potrebbe forgiare token e impersonare qualsiasi utente. Va in una variabile d'ambiente, come tutti i segreti (Cap. 9).
