Capitolo 4

Il primo server con FastAPI

È il momento che aspettavi: scriviamo un vero server. In poche righe di Python — grazie a FastAPI, il framework che useremo per tutto il libro — creeremo un'applicazione che riceve richieste HTTP e risponde con JSON. E la sorpresa più bella: FastAPI, a partire dal tuo codice, genera da sola una pagina di documentazione interattiva dove puoi provare le tue API dal browser, senza scrivere una riga di front-end. In questo capitolo capisci cos'è FastAPI e perché lo abbiamo scelto (§4.1), lo installi (§4.2), scrivi il tuo primo endpoint smontandolo pezzo per pezzo (§4.3), avvii il server con uvicorn (§4.4) e scopri la documentazione automatica (§4.5). Alla fine avrai un backend che gira sul tuo computer e risponde davvero — il cuore attorno a cui, da qui in poi, aggiungeremo tutto il resto.

🗺 Dove siamo

Cos'è FastAPI (§4.1), installazione (§4.2), il primo endpoint smontato (§4.3), avvio con uvicorn (§4.4), la documentazione automatica a /docs (§4.5). Nel laboratorio (§4.6) scrivi ed esegui il tuo «hello API».

🔑 Perché ci tocca: è la spina dorsale del libro

Da questo capitolo, ogni cosa che costruiremo — parametri, validazione, CRUD, database, autenticazione, upload — sarà un'aggiunta a questa app FastAPI. Capire bene queste poche righe iniziali significa avere le fondamenta di tutto. E FastAPI è tra le tecnologie backend più usate e amate del momento: impararlo è una competenza spendibile davvero.

Sezione 4.1Cos'è FastAPI

Potremmo scrivere un server «a mano», gestendo noi ogni dettaglio di HTTP. Sarebbe un lavoro enorme e pieno di errori. Un framework come FastAPI fa quel lavoro pesante al posto nostro, lasciandoci scrivere solo la logica che ci interessa.

📖 Un framework per costruire API in Python

FastAPI è una libreria Python per costruire API web. Il suo compito: ricevere le richieste HTTP, capire quale tua funzione deve rispondere, consegnarle i dati già pronti (convertiti da JSON), e reimpacchettare la risposta. Tu scrivi funzioni Python normali; FastAPI le trasforma in endpoint HTTP.

Perché proprio FastAPI, tra i tanti? Tre motivi che toccherai con mano: (1) è veloce da scrivere — un endpoint sono 3 righe; (2) usa i tipi di Python per validare automaticamente i dati in arrivo (grazie a Pydantic, Cap. 6), risparmiandoti montagne di controlli manuali; (3) genera da solo una documentazione interattiva. È moderno, molto usato in produzione, e ha una community enorme.

🔤 Un po' di vocabolario: ASGI, uvicorn

FastAPI definisce cosa fa la tua app, ma non parla direttamente con la rete: per farlo serve un server ASGI, cioè un programma che riceve le connessioni e le passa a FastAPI. Il più comune è uvicorn (§4.4). Pensa a FastAPI come al cuoco (prepara le risposte) e a uvicorn come al cameriere (riceve gli ordini dalla sala e serve i piatti). Li abbiamo installati entrambi nel Cap. 3.

Sezione 4.3Il primo endpoint

Crea un file main.py nella cartella del progetto e scrivi queste poche righe. È un server completo e funzionante: leggiamolo, poi lo smontiamo parola per parola.

main.py
from fastapi import FastAPI

app = FastAPI()          # crea l'applicazione

@app.get("/")             # quando arriva GET su "/" ...
def radice():             # ... esegui questa funzione
    return {"messaggio": "Ciao, mondo!"}
Il primo endpoint smontato import · app · decoratore · funzione
from fastapi import FastAPI
importa la classe principale della libreria. È il punto di partenza di ogni app.
app = FastAPI()
crea l'oggetto applicazione. Questa variabile app rappresenta il tuo intero server: ci «appenderai» tutti gli endpoint. Il nome app è una convenzione che uvicorn si aspetta.
@app.get("/") — il decoratore
la riga magica. @ è un decoratore Python: «collega la funzione qui sotto a un evento». Qui dice: «quando arriva una richiesta GET al percorso /, esegui la funzione seguente». Cambiando get in post gestiresti le POST (Cap. 7).
def radice(): — la path operation function
la funzione che gestisce quella richiesta. Il nome è libero (radice, home…): conta il decoratore sopra, non il nome. Ciò che return restituisce diventa la risposta.
return {"messaggio": "Ciao, mondo!"}
restituisce un dizionario Python. FastAPI lo converte automaticamente in JSON e lo invia al client con codice 200. Non scrivi tu il JSON né gli header: te ne occupi solo dei dati.
Rileggi il tutto ad alta voce: «crea un'app; quando arriva GET su slash, rispondi con questo dizionario». Tre righe, e hai un endpoint HTTP. Tutto il resto del libro consiste nell'aggiungere altri decoratori (@app.post, @app.delete…) e altre funzioni, via via più ricche. La forma base, però, è sempre questa.

Sezione 4.4uvicorn: avviare il server

Il file main.py definisce l'app, ma da solo non «gira». Serve uvicorn per accenderlo e metterlo in ascolto sulla rete.

terminale (con (.venv) attivo)
(.venv) $ uvicorn main:app --reload
INFO:     Uvicorn running on http://127.0.0.1:8000 (Press CTRL+C to quit)
INFO:     Started reloader process ...
INFO:     Application startup complete.
Il comando uvicorn smontato uvicorn main:app --reload
uvicorn
il server ASGI che esegue la tua app e la espone sulla rete.
main:app
dove trovare l'applicazione: «nel file main.py, l'oggetto app». La sintassi è nomefile:nomevariabile. Ecco perché la variabile si chiama app: è la seconda parte di questo indirizzo.
--reload
«riavvia automaticamente a ogni modifica del codice». Comodissimo in sviluppo: salvi il file e il server si aggiorna da solo, senza riavviarlo a mano. Solo per lo sviluppo, però: in produzione (Cap. 18) si usano configurazioni diverse.
l'indirizzo http://127.0.0.1:8000
dove il server è in ascolto. 127.0.0.1 (alias localhost) = «questo stesso computer»; 8000 = la porta. Apri quell'indirizzo nel browser e parli col tuo server.
Con il server acceso, apri http://127.0.0.1:8000 nel browser: vedrai {"messaggio":"Ciao, mondo!"}. Il tuo primo endpoint risponde! Per fermare il server: Ctrl + C. E se qualcosa non va, guarda il terminale: uvicorn stampa lì ogni richiesta ricevuta e ogni eventuale errore — proprio come il log del Cap. 1.

Sezione 4.5La documentazione automatica

Ora il regalo di FastAPI. Con il server acceso, apri nel browser http://127.0.0.1:8000/docs. Troverai una pagina interattiva — che non hai scritto tu — con l'elenco dei tuoi endpoint e un pulsante per provarli dal vivo.

📖 /docs: la tua API, documentata e provabile

FastAPI ispeziona il tuo codice (i decoratori, i nomi, i tipi) e genera automaticamente una documentazione conforme allo standard OpenAPI, mostrata con un'interfaccia chiamata Swagger UI. Per ogni endpoint vedi metodo, percorso, parametri attesi e forma della risposta. Il pulsante «Try it out» ti lascia chiamare l'endpoint direttamente dalla pagina e vedere la risposta reale — senza curl, senza scrivere un front-end.

Perché è importante? Perché la documentazione è di solito la parte più noiosa e trascurata di un backend. Qui è gratis e sempre aggiornata (nasce dal codice, quindi non può «scordarsi» di un endpoint). È uno strumento di sviluppo formidabile per te, e un regalo per chiunque userà la tua API. (Esiste anche /redoc, una seconda documentazione dallo stile diverso.)

La tua prima API, dal vivo come il «Try it out» di /docs
Endpoint disponibili (dal tuo main.py):

Il secondo endpoint del widget — /saluta/{nome} — anticipa i parametri di percorso del prossimo capitolo. Ecco com'è scritto:

main.py — l'endpoint con parametro
@app.get("/saluta/{nome}")      # {nome} è un "buco" nel percorso
def saluta(nome: str):        # FastAPI ci passa quel valore
    return {"saluto": f"Ciao, {nome}!"}

Sezione 4.6Laboratorio: hello API

Mettiamo insieme tutto in un esercizio completo, da fare ora sul tuo computer. In cinque minuti avrai il tuo primo server FastAPI attivo, con documentazione interattiva.

terminale + file — la ricetta completa
# nella cartella "mia-api" del Cap. 3, con (.venv) attivo:
# 1) crea main.py con questo contenuto:
main.py
from fastapi import FastAPI

app = FastAPI()

@app.get("/")
def radice():
    return {"messaggio": "Ciao, mondo!"}

@app.get("/saluta/{nome}")
def saluta(nome: str):
    return {"saluto": f"Ciao, {nome}!"}
terminale
# 2) avvia il server
(.venv) $ uvicorn main:app --reload

# 3) nel browser, prova questi indirizzi:
#    http://127.0.0.1:8000/            -> {"messaggio":"Ciao, mondo!"}
#    http://127.0.0.1:8000/saluta/Luca -> {"saluto":"Ciao, Luca!"}
#    http://127.0.0.1:8000/docs        -> la documentazione interattiva!
✅ Cosa dovresti vedere

Aprendo /: il messaggio JSON. Aprendo /saluta/Luca: il saluto personalizzato — cambia «Luca» con il tuo nome e guarda la risposta cambiare. Aprendo /docs: due endpoint elencati, con «Try it out» funzionante. Se vedi tutto questo, hai un backend che funziona. Congratulazioni: da qui in poi non partiamo più da zero, ma arricchiamo questa app.

🔑 Quello che devi portarti dietro dal Capitolo 4
  1. FastAPI è un framework che trasforma funzioni Python in endpoint HTTP, con validazione e docs automatiche (§4.1).
  2. Un endpoint = app = FastAPI() + un decoratore @app.get("/percorso") sopra una funzione che return un dizionario (→ JSON) (§4.3).
  3. Si avvia con uvicorn main:app --reload: file:variabile, --reload per lo sviluppo. In ascolto su 127.0.0.1:8000 (§4.4).
  4. /docs: documentazione interattiva generata da sola, dove provi le API dal browser (§4.5).

Hai un server che risponde — ma per ora dice sempre più o meno la stessa cosa. Un backend utile deve reagire ai dettagli della richiesta: «dammi il prodotto numero 42», «cerca i risultati che contengono questa parola». Nel Capitolo 5 impari a ricevere parametri nelle tue rotte — quelli nel percorso (come il {nome} che hai già intravisto) e quelli nella query — con la validazione automatica dei tipi che è il vero superpotere di FastAPI.

Sezione 4.7Esercizi

Esercizio 4.1

Nel comando uvicorn main:app --reload, cosa indicano main e app? Cosa succederebbe se rinominassi il file in server.py?

Esercizio 4.2

A cosa serve il decoratore @app.get("/")? Cosa cambieresti per gestire invece le richieste POST allo stesso percorso?

Esercizio 4.3

Aggiungi al tuo main.py un nuovo endpoint GET /ping che restituisce {"stato": "ok"}. Scrivi il codice.

Esercizio 4.4

Perché la documentazione automatica di FastAPI è «sempre aggiornata», al contrario di una documentazione scritta a mano?

Sezione 4.8Soluzioni

Soluzione 4.1

main è il file (main.py), app è la variabile dell'applicazione dentro quel file (app = FastAPI()). Se rinominassi il file in server.py, dovresti lanciare uvicorn server:app --reload: la prima parte segue sempre il nome del file (senza .py).

Soluzione 4.2

@app.get("/") collega la funzione sottostante alle richieste GET sul percorso /: quando ne arriva una, FastAPI esegue quella funzione e ne invia il risultato come risposta. Per gestire le POST allo stesso percorso useresti @app.post("/") sopra un'altra funzione. (Stesso percorso, metodo diverso, funzione diversa: è lo schema REST del Cap. 2.)

Soluzione 4.3
main.py (aggiunta)
@app.get("/ping")
def ping():
    return {"stato": "ok"}

Con --reload attivo, salvando il file il server si aggiorna da solo: visita /ping (o guardalo comparire in /docs) per verificarlo.

Soluzione 4.4

Perché è generata dal codice stesso: FastAPI ispeziona i tuoi decoratori, i nomi delle funzioni e i tipi dei parametri per costruirla. Se aggiungi, togli o modifichi un endpoint, la documentazione cambia automaticamente di conseguenza. Una documentazione scritta a mano, invece, va aggiornata manualmente a ogni modifica del codice — e quasi sempre finisce per «restare indietro» e diventare inesatta.
