Capitolo 5

Rotte e parametri

Un endpoint che risponde sempre allo stesso modo serve a poco. Il bello inizia quando la risposta dipende dalla richiesta: «dammi il prodotto numero 42», «cerca gli articoli che contengono questa parola», «mostrami solo i primi 10 risultati». Questi «dettagli» viaggiano nell'URL come parametri, di due tipi: quelli di percorso (parte dell'indirizzo, per identificare quale risorsa) e quelli di query (dopo il ?, per filtrare o modulare la richiesta). FastAPI li riceve per te e — qui sta la magia — usa i tipi di Python che dichiari per validarli automaticamente: se qualcuno chiede il prodotto «banana» invece di un numero, FastAPI rifiuta la richiesta con un errore chiaro, prima ancora che il tuo codice giri. È il superpotere che ti fa scrivere meno controlli e più logica.

🗺 Dove siamo

Le path operation (§5.1), i parametri di percorso (§5.2), i parametri di query (§5.3), e la validazione automatica dei tipi (§5.4). Nel laboratorio (§5.5) costruisci un'API che accetta e valida parametri.

🔑 Perché ci tocca: qui l'API diventa utile

Ricevere input dal client è ciò che rende un backend interattivo: ricerche, filtri, paginazione, dettagli di una risorsa. E la validazione automatica dei tipi è, forse, la ragione numero uno per cui si sceglie FastAPI: elimina un'intera categoria di bug e di codice difensivo noioso. Impararla bene ti farà risparmiare tempo per tutto il resto del libro.

Sezione 5.1Le path operation

Ricapitoliamo il vocabolario, ora con un nome preciso. Ogni coppia «decoratore + funzione» che gestisce una richiesta si chiama, in gergo FastAPI, path operation: un'operazione (il metodo: get, post…) su un percorso (path).

Anatomia di una path operation operazione + percorso + funzione
main.py
@app.get("/prodotti")      # operazione GET, percorso /prodotti
def elenca_prodotti():
    return [{"id": 1}, {"id": 2}]
l'operazione: @app.get, .post, .put, .delete
il metodo HTTP (Cap. 2). Determina quando la funzione scatta.
il percorso: "/prodotti"
l'indirizzo, sempre a partire da /. Può contenere segnaposto tra graffe (§5.2).
la funzione sottostante
la logica che produce la risposta. I suoi parametri (tra le parentesi) sono la porta d'ingresso dei dati della richiesta — è il tema di tutto il capitolo.
L'idea chiave che sblocca il capitolo: i parametri che dichiari nella funzione Python dicono a FastAPI quali dati aspettarsi dalla richiesta, e da dove prenderli. FastAPI li «inietta» al posto giusto. Vediamo i due tipi principali di parametro.

Sezione 5.2Parametri di percorso

Un parametro di percorso è un pezzo variabile dell'URL, usato per identificare una risorsa specifica: il 42 in /prodotti/42. Lo dichiari mettendo un segnaposto {…} nel percorso e un parametro con lo stesso nome nella funzione.

main.py
@app.get("/prodotti/{id_prodotto}")     # {id_prodotto} è il "buco"
def leggi_prodotto(id_prodotto: int):  # stesso nome, tipo int
    return {"id": id_prodotto, "nome": "Tazza"}
Come funziona {segnaposto} → parametro della funzione
il segnaposto {id_prodotto} nel percorso
marca la parte variabile dell'URL. Chiamando /prodotti/42, il 42 finisce in id_prodotto. Il nome tra graffe deve combaciare col nome del parametro nella funzione.
il parametro id_prodotto: int
FastAPI ci mette dentro il valore preso dall'URL. Il : int è l'annotazione di tipo: dice «questo dev'essere un numero intero». FastAPI lo converte e lo valida (§5.4).
dentro la funzione
id_prodotto è una normale variabile Python con quel valore. Nel libro, presto, lo useremo per cercare la riga giusta nel database.
Nota il salto rispetto al Cap. 4: là il {nome} era una stringa; qui id_prodotto: int dice a FastAPI di aspettarsi (e restituire) un numero. Chiamando /prodotti/42, id_prodotto arriva come l'intero 42, non la stringa "42" — la conversione è automatica. Usa i parametri di percorso per identificare una risorsa precisa (il «quale»); per tutto il resto ci sono i parametri di query.

Sezione 5.3Parametri di query

Un parametro di query è un'informazione aggiuntiva che viaggia dopo il ? nell'URL, nella forma chiave=valore, separati da &. Serve a filtrare, cercare, paginare: non identifica quale risorsa, ma come vuoi i dati.

main.py
@app.get("/prodotti")
def cerca_prodotti(q: str = None, limite: int = 10):
    return {"cerca": q, "limite": limite}

Con questa funzione, l'URL /prodotti?q=tazza&limite=5 produce la risposta {"cerca": "tazza", "limite": 5}.

Query param: la regola del valore di default parametro = default
parametri non nel percorso
Ecco la regola d'oro di FastAPI: se un parametro della funzione non compare tra le graffe del percorso, è automaticamente un parametro di query. Qui q e limite non sono nel path /prodotti, quindi sono query param.
q: str = None
il = None gli dà un valore di default: rende il parametro facoltativo. Chiamare /prodotti senza q è valido (q vale None).
limite: int = 10
facoltativo con default 10: se non lo passi, vale 10. Se passi ?limite=5, vale 5 (convertito a intero). I default rendono le API comode: il chiamante specifica solo ciò che gli serve.
obbligatorio vs facoltativo
un query param senza default (es. q: str) diventa obbligatorio: ometterlo dà errore. Con il default, è opzionale. Sta a te scegliere.
Riassumendo la distinzione: percorso = «quale risorsa» (/prodotti/42), fa parte dell'identità dell'indirizzo, di solito obbligatorio. Query = «come filtrare/modulare» (?q=…&limite=…), spesso facoltativo. Puoi combinarli: una funzione può avere sia un parametro di percorso sia dei query param, e FastAPI smista ciascuno alla fonte giusta guardando se il nome è tra le graffe.

Sezione 5.4Tipi e validazione automatica

Arriviamo al superpotere. Le annotazioni di tipo (: int, : str…) non sono decorazioni: FastAPI le usa per validare e convertire ogni parametro in arrivo. Se il dato non rispetta il tipo, la richiesta viene respinta automaticamente con un errore chiaro — senza che tu scriva un solo controllo.

cosa succede se il tipo non torna
# l'endpoint: def leggi_prodotto(id_prodotto: int)

$ curl http://127.0.0.1:8000/prodotti/42
{ "id": 42, "nome": "Tazza" }          # ok: 42 è un intero valido

$ curl -i http://127.0.0.1:8000/prodotti/banana
HTTP/1.1 422 Unprocessable Entity       # respinto! "banana" non è un int
{ "detail": [ { "msg": "Input should be a valid integer", ... } ] }
📖 Validazione «gratis»: cosa fa FastAPI per te

Dichiarando id_prodotto: int, hai detto a FastAPI «questo dev'essere un intero». Da quel momento, per ogni richiesta, FastAPI: (1) prende il valore grezzo dall'URL (sempre testo); (2) prova a convertirlo nel tipo dichiarato; (3) se ci riesce, passa alla tua funzione il valore già del tipo giusto; (4) se non ci riesce (come «banana» → int), blocca tutto e risponde 422 con un messaggio d'errore preciso, senza mai eseguire la tua funzione. Il tuo codice riceve solo dati già validi: niente più if not isinstance(...) sparsi ovunque. Questa garanzia, moltiplicata per ogni parametro di ogni endpoint, è una quantità enorme di lavoro e di bug risparmiati. Funziona con int, float, bool, str, e (dal Cap. 6, con Pydantic) con interi oggetti complessi.

💜 Il codice di stato giusto: 422

Nota il 422 Unprocessable Entity: è un errore 4xx, cioè «colpa del client» (Cap. 2) — ed è corretto, perché è il client ad aver mandato dati malformati. FastAPI sceglie il codice giusto da solo e spiega cosa non andava nel campo detail. Comunicazione impeccabile verso chi usa la tua API, senza che tu abbia scritto nulla.

Sezione 5.5Laboratorio: un'API con parametri

Costruiamo un piccolo «catalogo» che usa entrambi i tipi di parametro. Aggiungi questo al tuo main.py e provalo — soprattutto, prova a rompere la validazione, per vederla in azione.

main.py (aggiunte)
# un piccolo "catalogo" finto in memoria
CATALOGO = {
    1: {"nome": "Tazza", "prezzo": 9.9},
    2: {"nome": "Quaderno", "prezzo": 4.5},
}

@app.get("/prodotti/{id_prodotto}")    # parametro di percorso
def leggi_prodotto(id_prodotto: int):
    return CATALOGO.get(id_prodotto, {"errore": "non trovato"})

@app.get("/prodotti")                 # parametri di query
def cerca(prezzo_max: float = 100):
    return [p for p in CATALOGO.values() if p["prezzo"] <= prezzo_max]
terminale — provalo (e provalo a rompere)
$ curl http://127.0.0.1:8000/prodotti/1
{ "nome": "Tazza", "prezzo": 9.9 }

$ curl "http://127.0.0.1:8000/prodotti?prezzo_max=5"
[ { "nome": "Quaderno", "prezzo": 4.5 } ]      # solo i prodotti <= 5

$ curl -i "http://127.0.0.1:8000/prodotti?prezzo_max=caro"
HTTP/1.1 422 Unprocessable Entity            # "caro" non è un float!
🔬 Cosa osservare

/prodotti/1 usa il percorso per identificare il prodotto; /prodotti?prezzo_max=5 usa la query per filtrare. Cambiando prezzo_max vedi la lista restringersi. E chiedendo un prezzo_max non numerico, FastAPI risponde 422 da solo. Nota le virgolette attorno all'URL con il ?: nel terminale servono a evitare che la shell interpreti caratteri come ? e &. Apri anche /docs: vedrai i campi dei parametri già pronti da compilare, con l'indicazione del tipo.

🔑 Quello che devi portarti dietro dal Capitolo 5
  1. Una path operation = operazione (metodo) + percorso + funzione; i parametri della funzione sono la porta dei dati della richiesta (§5.1).
  2. Parametro di percorso: segnaposto {…} nel path + parametro omonimo nella funzione; identifica quale risorsa (§5.2).
  3. Parametro di query: ogni parametro della funzione non nel percorso; dopo il ?; con un = default diventa facoltativo (§5.3).
  4. Le annotazioni di tipo attivano la validazione automatica: dato del tipo giusto → convertito e passato; tipo sbagliato → 422 automatico, funzione non eseguita (§5.4).

Sai ricevere singoli valori — un id, una parola, un numero. Ma per creare una risorsa (un nuovo utente, un ordine) serve inviare un intero oggetto strutturato: nome, email, età, indirizzo… tutti insieme, nel corpo della richiesta. Come si riceve e si valida un dato così ricco? Con i modelli Pydantic, il cuore del Capitolo 6 — dove la validazione automatica che hai appena assaggiato diventa uno strumento potentissimo per definire e proteggere la forma dei tuoi dati.

Sezione 5.7Soluzioni

Soluzione 5.1

(a) /utenti/7: il 7 è un parametro di percorso — identifica quale utente (una risorsa precisa). (b) /utenti?attivi=true: attivi=true è un parametro di query — filtra la collezione (mostra solo gli utenti attivi). Percorso = quale; query = come filtrare/modulare.

Soluzione 5.2
main.py
@app.get("/citta/{nome}")
def leggi_citta(nome: str):
    return {"citta": nome}
Soluzione 5.3

È un parametro di query, perché pagina non compare tra graffe in un percorso (la regola del §5.3). Ed è facoltativo, perché ha un valore di default (= 1): se il client non passa ?pagina=…, vale 1. È il pattern tipico della paginazione.

Soluzione 5.4

FastAPI prova a convertire "abc" in int, non ci riesce, e respinge la richiesta con un errore 422 Unprocessable Entity, indicando nel dettaglio che quantita doveva essere un intero. La tua funzione non viene eseguita affatto: la validazione avviene prima. È esattamente la protezione «gratuita» che ti evita di controllare i tipi a mano.
