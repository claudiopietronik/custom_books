Capitolo 15

Docker: impacchettare l'app

Il nostro backend funziona… sul nostro computer. Ma quando proverai a metterlo su un server, o a passarlo a un collega, incontrerai il tormentone più famoso dell'informatica: «sul mio computer funzionava!». Versioni di Python diverse, librerie mancanti, sistemi operativi differenti: mille dettagli che rendono il software fragile e difficile da spostare. La soluzione moderna è Docker: uno strumento che chiude la tua applicazione — codice, dipendenze, configurazione, persino una fetta di sistema operativo — in un contenitore isolato e portabile, che gira identico ovunque. In questo capitolo capisci il problema che Docker risolve (§15.1), la differenza tra immagini e container (§15.2), come descrivere il tuo pacchetto con un Dockerfile (§15.3) e come costruirlo ed eseguirlo (§15.4). È il passo che trasforma «funziona da me» in «funziona ovunque», e apre la strada al deploy vero.

🗺 Dove siamo

Il problema «sul mio PC funziona» (§15.1), immagini e container (§15.2), il Dockerfile (§15.3), costruire ed eseguire (§15.4). Nel laboratorio (§15.5) impacchetti la tua app FastAPI in un container.

🔑 Perché ci tocca: dal codice al software spedibile

Un backend che gira solo sul tuo portatile non serve a nessuno: deve poter essere spostato — su un server, nel cloud, sulla macchina di un collega — e funzionare allo stesso modo. Docker è lo standard con cui oggi si distribuisce e si esegue il software. È anche il fondamento di tutto ciò che resta: mettere insieme più servizi (Cap. 16) e andare in produzione (Cap. 18). Impararlo è indispensabile per chiunque porti codice nel mondo reale.

Sezione 15.1Il problema «sul mio PC funziona»

Il tuo codice non gira nel vuoto: dipende da un ambiente. La versione di Python, le librerie installate (Cap. 3), variabili di sistema, persino il sistema operativo. Se anche uno solo di questi dettagli è diverso altrove, l'app può rompersi.

📖 L'ambiente è invisibile finché non cambia

Sul tuo computer «tutto funziona» perché l'ambiente è quello giusto — ma spesso non te ne rendi conto: è invisibile. Sposta il codice su un server con una versione di Python diversa, o dove manca una libreria di sistema, e all'improvviso crolla. Il problema non è il codice, è l'ambiente che non combacia. Gli ambienti virtuali (Cap. 3) risolvono una parte (le librerie Python), ma non tutto: restano la versione di Python, le librerie di sistema, la configurazione. Docker risolve il problema alla radice: impacchetta l'intero ambiente insieme al codice, così ciò che gira da te gira identico ovunque.

📦 Container vs macchina virtuale

Forse conosci le macchine virtuali (VM): simulano un intero computer, sistema operativo compreso — potenti ma pesanti (lente, tanta RAM e disco). I container di Docker sono più leggeri: condividono il kernel del sistema ospite e isolano solo ciò che serve (processi, file, rete). Risultato: partono in secondi, occupano poco, e ne puoi far girare tanti insieme. Per distribuire applicazioni, i container hanno largamente sostituito le VM.

Sezione 15.2Immagini e container

Due parole al cuore di Docker, spesso confuse: immagine e container. La distinzione è semplice e fondamentale — è come quella tra una classe e un oggetto, o tra una ricetta e il piatto.

Immagine vs container il modello vs l'istanza in esecuzione
l'immagine (image)
il pacchetto immutabile: contiene il tuo codice, le dipendenze, il runtime, tutto l'ambiente. È il «modello» congelato, che puoi salvare, condividere e scaricare. Non «gira»: è un file (a strati).
il container
un'istanza in esecuzione di un'immagine: l'app effettivamente avviata e viva. Da una sola immagine puoi avviare tanti container identici — utile per scalare (Cap. 18).
l'analogia
immagine : container = classe : oggetto (Cap. Python/C++), o ricetta : piatto cucinato. L'immagine descrive; il container esegue.
il registry (accenno)
le immagini si condividono tramite «registri» come Docker Hub: da lì hai scaricato postgres e minio nei capitoli scorsi. Immagini pronte per migliaia di software.
Nei capitoli 9 e 14 hai già usato immagini pronte (docker run postgres, docker run minio/minio): hai scaricato un'immagine e avviato un container da essa. Ora impari a creare l'immagine della tua app. Il flusso è: scrivi un Dockerfile → costruisci l'immagine → avvii container da essa.

Sezione 15.3Il Dockerfile

Il Dockerfile è un file di testo che descrive, passo per passo, come costruire l'immagine della tua app: da quale base partire, cosa copiare, cosa installare, come avviarla. È la «ricetta» dell'immagine.

Dockerfile — per la nostra app FastAPI
FROM python:3.12-slim          # 1) parti da un'immagine con Python

WORKDIR /app                     # 2) cartella di lavoro dentro il container

COPY requirements.txt .        # 3) copia l'elenco delle dipendenze
RUN pip install -r requirements.txt   # 4) installale

COPY . .                       # 5) copia tutto il resto del codice

CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "80"]  # 6) comando d'avvio
Le istruzioni del Dockerfile FROM · WORKDIR · COPY · RUN · CMD
FROM python:3.12-slim
la base da cui partire: un'immagine ufficiale con Python già pronto. Non parti da zero: costruisci sopra un'immagine esistente (slim = versione leggera).
WORKDIR /app
imposta la cartella di lavoro dentro il container: i comandi successivi girano lì.
COPY + RUN
COPY porta file dal tuo PC dentro l'immagine; RUN esegue un comando durante la costruzione (qui pip install). Nota l'ordine furbo: copiare e installare le dipendenze prima del resto del codice sfrutta la cache (se il codice cambia ma le dipendenze no, non le reinstalla).
CMD [...]
il comando che parte quando il container si avvia: qui uvicorn che lancia la nostra app. Nota --host 0.0.0.0: dentro un container serve per accettare connessioni dall'esterno del container.
Ogni istruzione crea uno strato (layer) dell'immagine, impilato sul precedente — l'hai visto nel widget. Il Dockerfile rende l'ambiente codice: versionabile, riproducibile, condivisibile. Chiunque abbia il tuo Dockerfile può ricostruire esattamente lo stesso ambiente. È la fine del «sul mio PC funziona». Aggiungi anche un file .dockerignore (come .gitignore) per escludere dal pacchetto cose come .venv e i file temporanei.

Sezione 15.4Costruire ed eseguire

Con il Dockerfile pronto, due comandi: build per creare l'immagine, run per avviare un container da essa.

terminale — build ed esecuzione
# costruisci l'immagine, dandole un nome ("tag") con -t
$ docker build -t mia-api .
... Successfully built · Successfully tagged mia-api:latest

# avvia un container dall'immagine, mappando la porta 80 del container sulla 8000 del PC
$ docker run -p 8000:80 mia-api
INFO:  Uvicorn running on http://0.0.0.0:80  (dentro il container)
# ora apri http://localhost:8000 -> la tua API risponde, dal container!
I due comandi smontati build -t · run -p
docker build -t mia-api .
costruisce l'immagine seguendo il Dockerfile. -t mia-api le dà un nome (tag); il . finale è la cartella dove cercare il Dockerfile (quella corrente). Fai questo dopo ogni modifica al codice.
docker run -p 8000:80 mia-api
avvia un container dall'immagine. -p 8000:80 collega la porta 80 del container alla 8000 del tuo PC: senza questa mappatura, il container sarebbe irraggiungibile (è isolato). L'hai già usata per Postgres e MinIO.
comandi utili
docker ps (container attivi), docker stop <id> (fermane uno), docker images (le immagini), docker logs <id> (i log di un container).
Nota la doppia porta 8000:80: «esterna:interna». Dentro il container uvicorn ascolta sulla 80; tu, fuori, la raggiungi sulla 8000. È lo stesso schema di Postgres (5432:5432) e MinIO (9000:9000) dei capitoli scorsi. Il container è una scatola isolata: le porte sono le sue «finestre» verso il mondo, che apri esplicitamente.

Sezione 15.5Laboratorio: l'app in un container

Impacchettiamo davvero la nostra API FastAPI. Ti serve Docker installato (docker.com) e la cartella del progetto con main.py e requirements.txt.

terminale — i passi
# 1) nella cartella del progetto, crea il file "Dockerfile" (§15.3)
#    e un ".dockerignore" con dentro: .venv

# 2) assicurati che requirements.txt includa uvicorn:
$ pip freeze > requirements.txt

# 3) costruisci l'immagine
$ docker build -t mia-api .

# 4) avvia il container
$ docker run -p 8000:80 mia-api

# 5) nel browser: http://localhost:8000/docs  -> la tua API, dal container!
✅ Cosa hai ottenuto

La tua applicazione ora vive in un'immagine autosufficiente: contiene Python, le dipendenze e il codice, pronta a girare su qualsiasi macchina con Docker — il tuo PC, quello di un collega, un server nel cloud — sempre allo stesso modo. Puoi fermarla, riavviarla, avviarne più copie. Hai trasformato il tuo codice in software spedibile. Un difetto resta però evidente: la nostra app ha bisogno anche di Postgres e MinIO, che finora avviamo a mano in container separati. Coordinare tutti questi pezzi con comandi sciolti è scomodo — ed è esattamente il problema che risolve il prossimo capitolo.

🔑 Quello che devi portarti dietro dal Capitolo 15
  1. Docker risolve «sul mio PC funziona» impacchettando codice + ambiente insieme. I container sono più leggeri delle macchine virtuali (§15.1).
  2. Immagine = pacchetto immutabile (il modello); container = istanza in esecuzione. Da un'immagine, tanti container (§15.2).
  3. Il Dockerfile descrive come costruire l'immagine: FROM, WORKDIR, COPY, RUN, CMD (§15.3).
  4. docker build -t nome . crea l'immagine; docker run -p esterna:interna nome avvia un container e ne espone la porta (§15.4).

Sai impacchettare un servizio. Ma il nostro backend è fatto di più servizi che devono lavorare insieme: l'app FastAPI, il database Postgres, lo storage MinIO. Avviarli e collegarli uno per uno, a mano, con tanti docker run, è scomodo e fragile. Nel Capitolo 16 conosciamo Docker Compose: uno strumento per descrivere tutto lo stack — app, database, storage e le loro connessioni — in un unico file, e avviarlo con un solo comando. Il nostro backend diventerà un sistema coordinato, pronto per il deploy finale.

Esercizio 15.3

Nel comando docker run -p 8000:80 mia-api, cosa significano i due numeri 8000 e 80? Cosa succederebbe se omettessi del tutto l'opzione -p?

Sezione 15.7Soluzioni

Soluzione 15.1

Docker risolve il problema del «funziona da me ma non altrove»: impacchetta l'applicazione insieme a tutto il suo ambiente (versione di Python, librerie, configurazione, sistema di base), così gira identica ovunque. Gli ambienti virtuali (venv) isolano solo le librerie Python, ma non la versione di Python stessa, le librerie di sistema o la configurazione del sistema operativo: Docker copre tutto l'ambiente, non solo una parte.

Soluzione 15.2

Un'immagine è come una ricetta (o una classe): un modello immutabile, che descrive l'app e il suo ambiente. Un container è come il piatto cucinato (o un oggetto): un'istanza in esecuzione creata a partire dall'immagine. Da una sola immagine puoi avviare quanti container vuoi, tutti identici — utile, per esempio, per gestire più traffico avviando più copie dell'app.

Soluzione 15.3

FROM indica l'immagine di base da cui partire (es. Python già pronto); COPY porta file dal tuo PC dentro l'immagine; CMD definisce il comando d'avvio del container. Si copiano/installano le dipendenze prima del resto del codice per sfruttare la cache di Docker: se il codice cambia ma le dipendenze no, Docker non le reinstalla, rendendo le ricostruzioni molto più veloci.

Soluzione 15.4

In -p 8000:80, 8000 è la porta sul tuo computer (esterna) e 80 è la porta dentro il container (interna, dove ascolta uvicorn): il comando le collega, così visitando localhost:8000 raggiungi l'app nel container. Se omettessi -p, il container girerebbe ma la sua porta non sarebbe esposta: l'app risulterebbe irraggiungibile dall'esterno del container.
