Sezione 16.1L'idea: uno script di backup

Prima di scrivere, definiamo cosa deve fare lo script. Un buon progetto parte da requisiti chiari.

📋 Cosa deve fare il nostro backup

Lo script deve:

  • comprimere una cartella importante (es. ~/documenti) in un unico archivio;
  • datare l'archivio: il nome include la data del giorno (es. backup-2026-07-19.tar.gz), così i backup non si sovrascrivono;
  • salvarlo in una cartella dedicata (es. ~/backups);
  • fare pulizia: tenere solo i backup recenti (es. degli ultimi 7 giorni), cancellando i più vecchi;
  • registrare cosa ha fatto in un file di log;
  • girare da solo, ogni notte, con cron.

Riconosci gli strumenti dietro ognuno? Comprimere = tar; datare = la sostituzione di comando $(date) (Cap. 12); condizioni e cicli per la pulizia (Cap. 13); log con >> (Cap. 9); automazione con cron (Cap. 14). Nessuno strumento nuovo: solo composizione — più un comando che incontriamo ora, tar, per gli archivi.

Sezione 16.2Costruire lo script pezzo per pezzo

Costruiamo lo script un pezzo alla volta, così ogni riga ha un senso chiaro. Prima cosa nuova: tar, il comando per creare archivi compressi.

📦 tar: creare un archivio compresso
tar per comprimere una cartella
$ tar -czf archivio.tar.gz documenti/
# -c = crea, -z = comprimi (gzip), -f = nome file

tar -czf crea un archivio compresso (un .tar.gz) di una cartella: -c «create», -z «zip/comprimi», -f «file» (il nome). Un unico file, più piccolo, che contiene tutta la cartella — perfetto per un backup. (Per estrarlo: tar -xzf archivio.tar.gz, con -x «extract».)

backup.sh — lo script, costruito pezzo per pezzo
#!/bin/bash

# 1) VARIABILI: cosa salvare, dove, e il nome con la data (Cap. 12)
SORGENTE="/home/anna/documenti"
DESTINAZIONE="/home/anna/backups"
DATA=$(date +%Y-%m-%d)              # es. 2026-07-19
ARCHIVIO="$DESTINAZIONE/backup-$DATA.tar.gz"

# 2) assicurati che la cartella di destinazione esista (Cap. 4)
mkdir -p "$DESTINAZIONE"

# 3) crea l'archivio compresso (tar)
tar -czf "$ARCHIVIO" "$SORGENTE"

# 4) registra nel log cosa è stato fatto (Cap. 9)
echo "[$(date)] Backup creato: $ARCHIVIO" >> $DESTINAZIONE/backup.log
Lo script letto riga per riga tutto il libro in un file
le variabili in cima (Cap. 12)
SORGENTE, DESTINAZIONE, e il nome datato con $(date +%Y-%m-%d). Definite una volta, usate ovunque: per cambiare cosa salvare, modifichi solo una riga.
mkdir -p (Cap. 4)
crea la cartella dei backup se non esiste, senza errori se già c'è. Una piccola robustezza.
tar -czf
il cuore: comprime la cartella nell'archivio datato. Un file per ogni giorno, che non si sovrascrivono grazie alla data nel nome.
il log con >> (Cap. 9)
aggiunge una riga al file di log con data e nome dell'archivio. Così puoi verificare che il backup (che gira di notte, invisibile) sia stato fatto.
Nota le virgolette attorno alle variabili ("$SORGENTE"): buona pratica (Cap. 13), evita problemi con spazi nei percorsi. E la data nel nome è l'idea chiave: backup-2026-07-19.tar.gz, backup-2026-07-20.tar.gz… ogni giorno il suo, senza cancellare i precedenti. Questo script già funziona: comprime e registra. Ora lo rendiamo più robusto.

Sezione 16.3Renderlo robusto

Uno script che gira da solo dev'essere robusto: controllare che le cose vadano bene e non accumulare backup all'infinito (riempirebbe il disco!). Aggiungiamo un controllo e una pulizia.

backup.sh — aggiunte per la robustezza
# dopo aver creato l'archivio, CONTROLLA che sia andato bene (Cap. 13)
if [ -f "$ARCHIVIO" ]; then
    echo "[$(date)] OK: $ARCHIVIO" >> $DESTINAZIONE/backup.log
else
    echo "[$(date)] ERRORE: backup fallito!" >> $DESTINAZIONE/backup.log
fi

# PULIZIA: cancella i backup più vecchi di 7 giorni (find, Cap. 5)
find "$DESTINAZIONE" -name "backup-*.tar.gz" -mtime +7 -delete
Le due robustezze controllo · pulizia automatica
il controllo con if [ -f ] (Cap. 13)
verifica che l'archivio sia stato davvero creato (-f = il file esiste). Se sì, log di successo; se no, log di errore. Così, guardando il log, sai se il backup ha funzionato — essenziale per un task automatico che non vedi girare.
la pulizia con find (Cap. 5)
find ... -mtime +7 -delete trova i backup più vecchi di 7 giorni e li cancella. Senza questo, i backup si accumulerebbero all'infinito riempiendo il disco. La «rotazione»: tieni i recenti, elimina i vecchi.
perché è importante
un backup senza controllo è pericoloso (potrebbe fallire in silenzio e scoprirlo troppo tardi); un backup senza pulizia è una bomba a orologeria per lo spazio. La robustezza fa la differenza tra uno script «di prova» e uno «di produzione».
Guarda quante competenze del libro convergono: find con criterio temporale (Cap. 5), condizione if (Cap. 13), log con redirezione (Cap. 9), -mtime per la data (Cap. 5). Ogni pezzo l'hai imparato separatamente; qui collaborano in uno strumento coeso. Questa è la ricompensa dello scripting: le piccole competenze si moltiplicano quando si combinano.

Sezione 16.4Automatizzarlo con cron


  

    L'ultimo tocco: far girare lo script da solo, ogni notte, con cron (Cap. 14). Così non devi ricordarti di
    lanciarlo — il backup avviene mentre dormi.
  


  

    
terminale — schedulare il backup

    
# rendi lo script eseguibile (una volta)
$ chmod +x ~/backup.sh

# apri il crontab e aggiungi il task: ogni notte alle 2:00
$ crontab -e
#   aggiungi questa riga (nota il PERCORSO ASSOLUTO, Cap. 14!):
  0 2 * * *   /home/anna/backup.sh

# verifica
$ crontab -l
0 2 * * *   /home/anna/backup.sh

  


  

    
🌙 Il backup che si prende cura di sé

    

      Con questa riga di cron (0 2 * * * = ogni giorno alle 2:00, Cap. 14), lo script gira ogni notte
      da solo: crea l'archivio datato, controlla che sia riuscito, ruota i vecchi, e registra tutto nel log.
      Da domani, i tuoi documenti sono protetti automaticamente, senza che tu debba fare nulla. Nota il
      dettaglio importante: il percorso assoluto dello script (e delle variabili dentro), essenziale perché cron
      lo esegue «fuori» dalla tua cartella (Cap. 14). Ogni mattina, se vuoi, controlli il backup.log per
      conferma. Questo è il senso profondo di Linux e dell'automazione: definisci il lavoro una volta, e il sistema
      lo esegue per sempre, all'orario giusto, prendendosi cura di sé.
    

  



  
  
Sezione 16.5Laboratorio: il backup automatico


  

    Ecco lo script completo, da provare (in sicurezza, su una cartella di test). È tutto ciò che hai imparato, in un
    unico file.
  


  

    
backup.sh — completo

    
#!/bin/bash
SORGENTE="$HOME/documenti"
DESTINAZIONE="$HOME/backups"
DATA=$(date +%Y-%m-%d)
ARCHIVIO="$DESTINAZIONE/backup-$DATA.tar.gz"

mkdir -p "$DESTINAZIONE"
tar -czf "$ARCHIVIO" "$SORGENTE"

if [ -f "$ARCHIVIO" ]; then
    echo "[$(date)] OK: $ARCHIVIO" >> "$DESTINAZIONE/backup.log"
else
    echo "[$(date)] ERRORE: backup fallito!" >> "$DESTINAZIONE/backup.log"
fi

find "$DESTINAZIONE" -name "backup-*.tar.gz" -mtime +7 -delete

Sezione 16.8Soluzioni


  

    
Soluzione 16.1

    

      Includere la data nel nome (es. backup-2026-07-19.tar.gz) fa sì che ogni giorno crei un
      archivio con nome diverso. Così i backup non si sovrascrivono: ne conservi uno per ogni giorno,
      potendo tornare a una versione precedente. Senza la data, ogni esecuzione userebbe lo stesso nome e
      sovrascriverebbe il backup del giorno prima — perdendo la storia, il senso stesso di un backup ricorrente.
    

  


  

    
Soluzione 16.2

    

      Quella riga cancella i backup più vecchi di 7 giorni: find cerca nella cartella dei backup i
      file col nome backup-*.tar.gz (-name) modificati più di 7 giorni fa (-mtime
      +7) e li elimina (-delete). È importante perché, senza pulizia, i backup si
      accumulerebbero all'infinito, finendo per riempire il disco. È la «rotazione»: si tengono solo i backup
      recenti.
    

  


  

    
Soluzione 16.3

    

      Il controllo if [ -f "$ARCHIVIO" ] verifica che l'archivio di backup sia stato davvero
      creato (il file esiste): in caso positivo scrive un log di successo, altrimenti un log di errore. È utile
      soprattutto per uno script automatico e notturno perché nessuno lo vede girare: se il backup fallisse in
      silenzio, potresti scoprirlo troppo tardi (quando ti serve e non c'è). Il controllo, registrato nel log, ti
      permette di accorgerti dei problemi semplicemente leggendo il log.
    

  


  

    
Soluzione 16.4

    

      
crontab -e

      
0 2 * * *   /home/anna/backup.sh

    

    

      0 2 * * * significa «al minuto 0 dell'ora 2, ogni giorno» = ogni notte alle 2:00. Il percorso dello
      script (e dei file dentro) deve essere assoluto perché cron esegue lo script non dalla tua cartella:
      con un percorso relativo, cron non troverebbe lo script (o i file che usa). Il percorso assoluto lo individua in
      modo univoco, indipendentemente da dove cron opera.
