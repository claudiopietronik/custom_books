Capitolo 13

Bash scripting: logica

I tuoi script sanno eseguire comandi e usare variabili, ma finora vanno «dritti», dall'alto in basso, sempre uguali. Per renderli davvero utili serve dar loro la capacità di ragionare: decidere cosa fare in base a una condizione («se il file esiste, aggiornalo; altrimenti crealo») e ripetere un'azione più volte («per ogni file nella cartella, comprimilo»). Sono le stesse due idee di ogni linguaggio di programmazione — condizioni e cicli — che se hai fatto Python o C++ già conosci; qui le vediamo nella sintassi di bash, che ha le sue peculiarità (a volte un po' bizzarre, ma con uno schema regolare). Con queste, uno script smette di essere una lista di comandi e diventa un piccolo programma intelligente. In questo capitolo: le condizioni con if (§13.1), i cicli for (§13.2) e while (§13.3), e come combinarli in logica utile (§13.4). È il capitolo che sblocca il vero potere dell'automazione: script che si adattano e ripetono.

🗺 Dove siamo

Le condizioni con if e test (§13.1), il ciclo for (§13.2), il ciclo while (§13.3), combinare la logica (§13.4). Nel laboratorio (§13.5) scrivi uno script utile.

🔑 Perché ci tocca: script che si adattano

Uno script che fa sempre la stessa cosa, ciecamente, è fragile. Con condizioni e cicli, i tuoi script si adattano: controllano se qualcosa esiste, reagiscono agli errori, elaborano tanti file uno per uno. È ciò che serve per l'automazione vera — un backup che verifica, una pulizia che scorre tutti i file, un controllo che decide. Le fondamenta di ogni script utile.

Sezione 13.1Condizioni: if, test

L'if permette allo script di scegliere: esegue un blocco di comandi solo se una condizione è vera. La sintassi di bash è un po' particolare, ma segue uno schema fisso.

controlla.sh — un if
#!/bin/bash
FILE="dati.txt"

if [ -f "$FILE" ]; then          # se il file esiste...
    echo "Il file esiste!"
else                              # ...altrimenti
    echo "Il file non c'è."
fi                                # fine dell'if (fi = if al contrario!)
Anatomia di un if if [ condizione ]; then ... else ... fi
if [ condizione ]; then
«se la condizione è vera, allora…». Attenzione agli spazi dentro le parentesi quadre: [ -f "$FILE" ] con spazi attorno (è una trappola classica). Le [ ] sono in realtà il comando test travestito.
else e fi
else = «altrimenti» (facoltativo); fi chiude l'if (è «if» scritto al contrario — una peculiarità di bash!). Ogni if va chiuso con fi.
i test più comuni
-f file (il file esiste?), -d cartella (la cartella esiste?), -z "$VAR" (la variabile è vuota?). Per i numeri: -eq (uguale), -gt (maggiore), -lt (minore). Per il testo: = (uguale), != (diverso).
La sintassi di bash per le condizioni è famosa per le sue stranezze: le parentesi quadre con gli spazi, il fi per chiudere, i test con lettere (-gt invece di >). Non è la più elegante, ma ha uno schema regolare che, una volta imparato, si ripete uguale. Un consiglio: metti sempre le variabili tra virgolette nei test ("$FILE") — evita errori quando la variabile è vuota o contiene spazi. Prova le condizioni nel widget qui sotto.

Sezione 13.2Cicli: for


  

    Il ciclo for ripete un blocco di comandi per ogni elemento di una lista. È perfetto per
    elaborare tanti file, o ripetere un'azione un numero noto di volte.
  


  

    
esempi di for

    
#!/bin/bash
# per ogni parola nella lista
for FRUTTO in mela pera kiwi; do
    echo "Mi piace la $FRUTTO"
done

# per ogni file .txt della cartella (con i jolly del Cap. 4!)
for FILE in *.txt; do
    echo "Elaboro $FILE"
done

# per un intervallo di numeri
for N in {1..5}; do
    echo "Giro numero $N"
done

  


  

    
Il ciclo for for X in lista; do ... done

    

      

        
for X in lista; do ... done

        
ripete il blocco (tra do e done) una volta per ogni elemento della lista,
            mettendolo di volta in volta nella variabile X. done chiude il ciclo
            (come fi per l'if).

        
la lista: parole, file, numeri

        
può essere un elenco di parole (mela pera kiwi), un pattern di file coi jolly
            (*.txt), o un intervallo ({1..5}). Il caso più potente: iterare sui
            file di una cartella.

        
l'uso killer: elaborare tanti file

        
for FILE in *.log; do ...; done = «fai qualcosa a ogni file .log». Ecco perché lo
            scripting fa risparmiare tempo: un'azione ripetuta su cento file, scritta una volta.

      

    

    

      Il ciclo for è dove lo scripting mostra i muscoli: ciò che faresti a mano su un file, lo fai
      automaticamente su tutti. «Rinomina tutte le foto», «comprimi tutti i log vecchi», «controlla ogni
      cartella» — tutto diventa un for di poche righe. Combinato coi jolly (Cap. 4) e coi comandi che già
      conosci, è uno strumento enorme. Nota i «chiuditori» di bash: if→fi,
      for→done: ogni blocco ha il suo.
    

  



  
  
Sezione 13.3Cicli: while


  

    Il ciclo while ripete un blocco finché una condizione resta vera. Si usa quando non sai in
    anticipo quante ripetizioni serviranno (a differenza del for, che scorre una lista definita).
  


  

    
conta.sh — un while

    
#!/bin/bash
N=1
while [ $N -le 3 ]; do       # finché N è <= 3
    echo "Conteggio: $N"
    N=$((N + 1))            # $(( )) fa i calcoli: aumenta N di 1
done

  


  

    
Il ciclo while while [ condizione ]; do ... done

    

      

        
while [ condizione ]; do ... done

        
ripete il blocco finché la condizione è vera; appena diventa falsa, si ferma. Usa gli stessi
            test dell'if (-le, -gt…).

        
attenzione: aggiorna la condizione!

        
dentro il ciclo devi modificare qualcosa che alla fine renda la condizione falsa (qui, incrementi
            N). Se dimentichi, il ciclo va all'infinito (loop infinito): un errore
            classico. Si interrompe con Ctrl+C.

        
i calcoli: $(( ))

        
per fare aritmetica in bash si usa $(( espressione )): $((N + 1)) = «N più 1».
            Necessario perché bash tratta tutto come testo di default; le doppie parentesi dicono «questo è
            un calcolo».

        
for vs while

        
for quando sai su cosa iterare (una lista, dei file); while quando ripeti finché
            una condizione regge (numero di giri non noto in anticipo). Due strumenti per due situazioni.

      

    

    

      Il while è meno usato del for nell'automazione quotidiana (di solito iteri su file, e lì
      il for è naturale), ma è indispensabile quando la fine dipende da una condizione: «finché il server
      non risponde, riprova», «finché ci sono righe da leggere». Ricorda l'incremento per non cadere nel
      loop infinito — l'errore più comune con while.
    

  



  
  
Sezione 13.4Mettere insieme la logica


  

    La vera potenza emerge combinando condizioni e cicli: uno script che scorre dei file e, per ciascuno,
    decide cosa fare. È lo schema di quasi ogni script di automazione utile.
  


  

    
organizza.sh — for + if insieme

    
#!/bin/bash
# per ogni file nella cartella corrente...
for FILE in *; do
    if [ -d "$FILE" ]; then            # ...se è una cartella
        echo "[CARTELLA] $FILE"
    else                                # ...altrimenti è un file
        echo "[file]     $FILE"
    fi
done

  


  

    
🔗 Lo schema «scorri e decidi»

    

      Guarda il pattern: un ciclo (for) scorre gli elementi, e dentro un if decide cosa fare
      per ciascuno. Questo «scorri e decidi» è la struttura di innumerevoli script: «per ogni file, se è
      vecchio comprimilo»; «per ogni utente, se è inattivo avvisa»; «per ogni server, se non risponde segnala». È la
      stessa logica dei programmi in Python/C++ (cicli + condizioni), applicata ai file e ai comandi del sistema. Con
      questi due mattoni — for e if — più i comandi che già conosci, puoi automatizzare quasi
      qualsiasi compito ripetitivo. Nel prossimo capitolo (e nel progetto finale) vedrai script sempre più completi.
    

  



  
  
Sezione 13.5Laboratorio: uno script utile


  

    Scriviamo uno script che fa qualcosa di concreto: conta i file .txt in una cartella e avvisa se non
    ce ne sono. Unisce variabili, comandi, condizioni.
  


  

    
conta-txt.sh

    
#!/bin/bash
# conta i file .txt (con la sostituzione di comando del Cap. 12)
CONTA=$(ls *.txt 2>/dev/null | wc -l)

if [ "$CONTA" -eq 0 ]; then
    echo "Nessun file .txt qui."
else
    echo "Ci sono $CONTA file .txt:"
    for F in *.txt; do
        echo "  - $F"
    done
fi

  


  

    
terminale — eseguirlo

    
$ chmod +x conta-txt.sh
$ ./conta-txt.sh
Ci sono 3 file .txt:
  - note.txt
  - todo.txt
  - lista.txt

  


  

    
✅ Cosa hai costruito

    

      Uno script che ragiona: conta i file (con $(...) e la pipe), decide con un
      if (nessuno? mostra un avviso; altrimenti…), e scorre con un for per elencarli.
      Hai unito tutto il libro: comandi, pipe, variabili, condizioni, cicli. (La 2>/dev/null
      è un trucco: manda gli eventuali errori «nel nulla», per non sporcare l'output — usa la redirezione del canale
      errore del Cap. 9.) Questo è uno script vero, utile, che potresti tenere e riusare. Nel prossimo capitolo
      impariamo a rendere gli script ancora più comodi (alias, variabili d'ambiente) e — la ciliegina — a farli girare
      da soli a orari prestabiliti, con cron.

Sezione 13.7Soluzioni


  

    
Soluzione 13.1

    

      
bash

      
if [ -d "backup" ]; then
    echo "esiste"
else
    echo "manca"
fi

    

    
Nota gli spazi dentro le [ ] e il fi che chiude l'if.

  


  

    
Soluzione 13.2

    

      Il ciclo scorre tutti i file .jpg della cartella corrente (il jolly *.jpg li
      seleziona) e, per ciascuno, stampa il suo nome (echo "$F"). È particolarmente utile quando devi fare
      la stessa operazione su molti file: qui solo li elenca, ma potresti ridimensionarli, spostarli,
      rinominarli — l'azione su un file, applicata automaticamente a tutti.
    

  


  

    
Soluzione 13.3

    

      Il for ripete il blocco per ogni elemento di una lista definita (parole, file, numeri): sai
      «su cosa» iterare. Il while ripete finché una condizione è vera: non sai in anticipo quante
      ripetizioni serviranno. Il rischio tipico di un while scritto male è il loop infinito: se
      dentro il ciclo non modifichi nulla che renda la condizione falsa, il ciclo non finisce mai (si interrompe solo
      con Ctrl+C).
    

  


  

    
Soluzione 13.4

    

      Un if si chiude con fi («if» scritto al contrario); un for (come
      pure un while) si chiude con done. Ogni blocco di controllo in bash ha il suo
      «chiuditore»: dimenticarlo è un errore di sintassi comune.
