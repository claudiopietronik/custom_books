Capitolo 15

Sistema e rete da terminale

Prima del progetto finale, completiamo la cassetta degli attrezzi con gli strumenti per ispezionare il sistema e la rete dal terminale: le domande pratiche che ti farai spesso, soprattutto amministrando un server. Quanto spazio è rimasto sul disco? Cosa lo sta riempiendo? La rete funziona, il sito è raggiungibile? Come mi collego a un server remoto? È attivo quel servizio? Sono comandi diagnostici che userai per capire lo stato di una macchina e per risolvere problemi — dal «il disco è pieno» al «il server non risponde». Molti li hai già sfiorati nei libri «Reti» e «Backend»: qui li mettiamo a fuoco. In questo capitolo: lo spazio su disco con df e du (§15.1), la rete con ip, ping, curl (§15.2), l'accesso remoto con ssh (§15.3), e i servizi con systemctl (§15.4). Sono gli strumenti che trasformano «il sistema non va» in «ecco cosa non va».

🗺 Dove siamo

Spazio su disco: df, du (§15.1), rete: ip, ping, curl (§15.2), accesso remoto: ssh (§15.3), servizi: systemctl (§15.4). Nel laboratorio (§15.5) ispezioni il sistema.

🔑 Perché ci tocca: diagnosticare è metà del risolvere

«Il server è lento», «il disco è pieno», «il sito non si apre»: dietro ogni problema c'è una diagnosi da fare. Questi comandi sono i tuoi «strumenti di misura»: ti dicono lo stato reale del sistema, così puoi agire con cognizione invece che a tentoni. Sono indispensabili per chiunque gestisca un server (l'hai visto nel deploy del libro «Backend»).

Sezione 15.1Spazio su disco: df, du

Due comandi rispondono a «quanto spazio ho?» e «cosa lo sta occupando?»: df (spazio dei dischi nel complesso) e du (spazio usato da cartelle e file specifici).

terminale — spazio su disco
# df -h: spazio libero/usato sui dischi (-h = "human", leggibile: GB invece di byte)
$ df -h
Filesystem   Size  Used  Avail  Use%  Mounted on
/dev/sda1     50G   38G    12G   76%  /

# du -sh: quanto occupa una cartella (-s = totale, -h = leggibile)
$ du -sh ~/documenti
2.3G  /home/anna/documenti

# le sottocartelle più pesanti (per trovare cosa occupa spazio)
$ du -h ~ | sort -rh | head
df e du i dischi vs le cartelle
df -h — «disk free»
mostra lo spazio totale, usato e libero di ogni disco/partizione. La colonna Use% è la più importante: se è vicina al 100%, il disco è quasi pieno (guai in arrivo). Il -h rende i numeri leggibili (GB/MB).
du -sh cartella — «disk usage»
mostra quanto occupa una cartella specifica (-s = solo il totale, -h = leggibile). Per capire cosa occupa spazio, non solo quanto ne resta.
trovare i «divoratori» di spazio
la pipe du -h | sort -rh | head (Cap. 9!) elenca le cartelle più pesanti. Il modo classico per scoprire cosa riempie il disco quando df ti avvisa che è pieno.
La coppia diagnostica tipica: df -h ti dice «il disco è pieno al 95%», poi du (con la pipe) ti dice «è quella cartella di log da 30 GB». Prima il quadro d'insieme, poi il dettaglio. È una delle situazioni più comuni su un server: il disco si riempie (spesso di log, in /var/log), il sistema inizia a fare i capricci, e questi comandi ti portano dritto al colpevole. Sezione 15.2Rete: ip, ping, curl


  

    Tre strumenti per la rete, che riprendono concetti del libro «Reti»: ip (il tuo indirizzo),
    ping (raggiungo quel computer?), curl (scarica/interroga un URL).
  


  

    
terminale — strumenti di rete

    
# ip a: mostra i tuoi indirizzi di rete (il tuo IP)
$ ip a
... inet 192.168.1.42/24 ...          # il tuo indirizzo locale

# ping: verifica se un computer è raggiungibile (invia pacchetti di prova)
$ ping google.com
64 bytes from ...: time=12.3 ms          (Ctrl+C per fermare)

# curl: scarica il contenuto di un URL (o interroga un'API, come nel libro Backend)
$ curl https://esempio.com

  


  

    
Gli strumenti di rete ip · ping · curl

    

      

        
ip a

        
mostra le tue interfacce di rete e i loro indirizzi IP (Cap. «Reti»). Per sapere «qual è il mio
            indirizzo» sulla rete locale. (Il vecchio comando era ifconfig.)

        
ping host

        
invia piccoli pacchetti di prova a un computer e misura se (e quanto veloce) risponde. «Quel server è
            raggiungibile?» Se il ping funziona, la rete verso quella destinazione va. Si ferma con
            Ctrl+C.

        
curl url

        
scarica il contenuto di un URL o interroga un'API (l'hai usato nel libro «Backend»!). Per testare che
            un sito/servizio risponda, o scaricare file. Comodissimo per il debug dei servizi web.

      

    

    

      Questi comandi sono la tua «prima diagnosi di rete»: ping per capire se arrivi a una macchina,
      curl per capire se un servizio web risponde, ip per sapere il tuo indirizzo. Se
      hai fatto «Reti», riconoscerai i concetti (IP, pacchetti, HTTP); qui sono gli strumenti pratici per verificarli
      dal terminale. Un altro utile: ping non funziona ma il sito sì? Potrebbe essere il DNS o un firewall
      — la diagnosi continua da lì.
    

  



  
  
Sezione 15.3Accesso remoto: ssh


  

    ssh è il modo di collegarti al terminale di un computer remoto, in modo sicuro. È così che si
    amministra un server: apri un terminale sul tuo PC e «entri» nel server, dando comandi come se fossi lì.
  


  

    
terminale — collegarsi a un server con ssh

    
# collegati a un server: ssh utente@indirizzo
$ ssh anna@203.0.113.5
anna@server:~$          # ora il prompt è del SERVER: sei "dentro"!

# da qui, ogni comando gira sul server remoto
anna@server:~$ df -h          # controlli il disco DEL SERVER

# per tornare al tuo computer: exit (o Ctrl+D)
anna@server:~$ exit

  


  

    
ssh smontato ssh utente@host · exit

    

      

        
ssh utente@indirizzo

        
apre una sessione remota sicura (cifrata) sul computer indicato, come quell'utente. Da quel
            momento il tuo terminale «è» quello del server: ogni comando gira lì. Il prompt cambia per
            ricordartelo (Cap. 2: ecco perché il nome host nel prompt è utile!).

        
l'autenticazione

        
ssh ti chiede la password (o usa una chiave, più sicura — Cap. «Backend»). Il traffico è
            cifrato: nessuno può spiare i comandi o i dati.

        
exit

        
chiude la sessione remota e torni al tuo computer. «Esci» dal server.

      

    

    

      ssh è il ponte verso i server: tutto ciò che hai imparato in questo libro — navigare, gestire
      file, processi, script, cron — lo applichi sui server attraverso ssh. È il motivo per cui saper
      usare Linux da terminale è così importante: i server non hanno grafica, ci si arriva solo così. Lo hai visto nel
      libro «Backend», per il deploy. Ora sai cos'è davvero: una finestra sicura sul terminale di una macchina lontana.
    

  



  
  
Sezione 15.4Servizi: systemctl (cenni)


  

    Molti programmi «di sistema» (un web server, un database, ssh stesso) girano come servizi: processi che
    partono all'avvio e restano attivi in background. Si gestiscono con systemctl. Ne diamo i cenni: ti
    serviranno amministrando server.
  


  

    
terminale — gestire i servizi (con sudo)

    
# stato di un servizio (es. nginx, il web server del libro Backend)
$ systemctl status nginx
● nginx.service - ... active (running)          # attivo!

# avviare / fermare / riavviare un servizio
$ sudo systemctl start nginx
$ sudo systemctl restart nginx

# fare in modo che parta automaticamente all'avvio del sistema
$ sudo systemctl enable nginx

  


  

    
systemctl: le azioni principali status · start/stop/restart · enable

    

      

        
systemctl status servizio

        
mostra se un servizio è attivo («active/running») o fermo, con gli ultimi messaggi. La prima
            cosa da controllare quando un servizio «non funziona».

        
start / stop / restart

        
avvia, ferma o riavvia un servizio (con sudo: sono azioni di sistema). Il
            restart è il classico «spegni e riaccendi» quando qualcosa si è impuntato.

        
enable / disable

        
decide se il servizio parte automaticamente all'avvio del sistema (enable) o no (disable).
            Perché un server, riavviato, deve far ripartire i suoi servizi da solo (come il
            restart: always di Docker nel libro «Backend»).

      

    

    

      Non serve approfondire ora: ti basti riconoscere systemctl come «il telecomando dei
      servizi» di sistema. Quando amministrerai un server e un servizio (web, database) non risponderà, il primo
      gesto sarà systemctl status nome per vedere se è attivo, e magari restart per
      riavviarlo. È il completamento naturale della gestione dei processi (Cap. 8), applicato ai programmi «di sistema»
      che girano stabilmente.
    

  



  
  
Sezione 15.5Laboratorio: ispezionare il sistema


  

    Facciamo un «check-up» del sistema con i comandi diagnostici — un giro che replicherai spesso, su PC e server.
  


  

    
terminale — un check-up del sistema

    
# quanto spazio ho sui dischi?
$ df -h

# cosa occupa spazio nella mia home? (le 5 cartelle più pesanti)
$ du -h ~ | sort -rh | head -n 5

# la rete funziona? (raggiungo internet?)
$ ping -c 3 google.com          # -c 3 = solo 3 pacchetti, poi si ferma

# qual è il mio indirizzo IP?
$ ip a | grep inet          # filtra le righe con l'IP (pipe!)

Sezione 15.7Soluzioni


  

    
Soluzione 15.1

    

      df mostra lo spazio dei dischi nel complesso (totale/usato/libero per partizione);
      du mostra lo spazio occupato da cartelle e file specifici. Per scoprire che il disco è pieno
      al 95% usi df -h (guardando la colonna Use%); per capire cosa lo sta riempiendo
      usi du (spesso con la pipe du -h | sort -rh | head, per trovare le cartelle più
      pesanti).
    

  


  

    
Soluzione 15.2

    

      ping verifica se un computer è raggiungibile in rete (invia pacchetti di prova e misura la
      risposta); curl scarica/interroga il contenuto di un URL. Per «un sito non si apre»: con
      ping verifichi se il server è raggiungibile a livello di rete; con curl se il servizio
      web risponde effettivamente (magari il server è su ma il sito dà errore). Insieme, restringono il problema (rete
      vs servizio).
    

  


  

    
Soluzione 15.3

    

      ssh anna@203.0.113.5 apre una sessione remota sicura (cifrata) sul server a quell'indirizzo,
      come utente anna: da quel momento il tuo terminale «è» quello del server e ogni comando gira lì. Lo capisci dal
      prompt, che cambia mostrando il nome dell'host del server (es. anna@server:~$) invece del tuo
      computer. Per tornare indietro (chiudere la sessione remota) usi exit (o Ctrl+D).
    

  


  

    
Soluzione 15.4

    

      Per primo, systemctl status nginx: mostra se il servizio è attivo («active/running») o fermo, con
      gli ultimi messaggi utili a capire il problema. Per provare a rimetterlo in funzione:
      sudo systemctl restart nginx (lo riavvia) — il classico «spegni e riaccendi» che risolve molti
      blocchi; se non basta, i messaggi dello status (e i log) indicano cosa non va.
