Capitolo 7

Utenti e sudo

Nel capitolo scorso hai visto che i file appartengono a utenti e hanno permessi. Ma c'è un utente speciale, al di sopra di tutti, che può fare qualsiasi cosa: l'amministratore, chiamato root. Root può leggere e modificare ogni file, installare software, cambiare le configurazioni di sistema, creare altri utenti. È onnipotente — e proprio per questo pericoloso: un errore da root può distruggere il sistema. La soluzione elegante che Linux adotta è non lavorare mai come root, ma «prendere in prestito» i suoi poteri solo per il singolo comando che lo richiede, con sudo. In questo capitolo capiamo la gerarchia degli utenti e cos'è root (§7.1), come usare sudo in sicurezza (§7.2), come gestire utenti e gruppi (§7.3), e il principio d'oro della sicurezza — il minimo privilegio (§7.4). È il capitolo che ti insegna a usare il «potere» quando serve, senza farti male.

🗺 Dove siamo

root e gli utenti normali (§7.1), sudo per i poteri temporanei (§7.2), gestire utenti e gruppi (§7.3), il minimo privilegio (§7.4). Nel laboratorio (§7.5) usi sudo.

🔑 Perché ci tocca: potere e responsabilità

Tante operazioni di sistema (installare programmi, configurare servizi, gestire altri utenti) richiedono i poteri di amministratore. Saper usare sudo in modo corretto e sicuro è indispensabile per amministrare qualsiasi sistema Linux, dal tuo PC a un server (l'hai già incontrato nel libro «Backend», nel deploy). E capire perché non si lavora sempre da root ti protegge dagli errori più gravi.

Sezione 7.1root e gli utenti normali

In Linux ci sono due «livelli» di utente: gli utenti normali (come te, con permessi limitati ai propri file) e l'utente root (l'amministratore, che può tutto). La distinzione è alla base della sicurezza del sistema.

Utente normale vs root limitato vs onnipotente
l'utente normale (tu)
può gestire i propri file (nella sua home), eseguire programmi, ma non toccare il sistema: non può installare software per tutti, modificare le configurazioni globali, o toccare i file altrui. I suoi errori restano circoscritti.
l'utente root (l'amministratore)
può fare tutto: leggere/modificare ogni file, installare software, gestire utenti, cambiare il sistema. Nessun permesso lo ferma. Il suo nome è «root» perché sta alla radice del controllo.
il segnale nel prompt
ricordi dal Cap. 2? Un prompt che finisce con $ = utente normale; con # = sei root. Il # è un promemoria visivo: «attenzione, hai pieni poteri».
perché non lavorare da root
da root, un comando sbagliato non ha protezioni: un rm nel posto sbagliato può cancellare file di sistema e rendere il computer inutilizzabile. Da utente normale, lo stesso errore fallirebbe con «Permission denied» — il sistema ti salva.
Per questo la pratica moderna è non accedere mai direttamente come root per il lavoro quotidiano. Si lavora come utente normale — protetti — e si «diventa» root solo per il singolo comando che lo richiede, con sudo (prossima sezione). È la differenza tra tenere sempre in mano un coltello affilato e prenderlo solo quando devi tagliare qualcosa.

Sezione 7.2sudo: poteri temporanei

sudo («superuser do») è il comando che esegue un comando con i poteri di root, chiedendoti la tua password. È il modo pulito e sicuro di fare operazioni da amministratore, quando servono.

terminale — sudo in azione
# senza sudo: un'operazione di sistema fallisce
$ apt install htop
E: Could not open lock file ... (Permission denied)

# con sudo: esegui lo stesso comando con i poteri di root
$ sudo apt install htop
[sudo] password for anna:           # chiede la TUA password
... htop installato con successo
Come funziona sudo un comando, poteri di root
sudo comando
esegue quel singolo comando con i poteri di root, poi torni a essere te stesso. I poteri sono temporanei: valgono solo per quel comando, non per la sessione.
chiede la tua password
non la password di root, ma la tua (per confermare che sei davvero tu). Non vedi i caratteri mentre digiti: è normale, per sicurezza — scrivi e premi Invio. La ricorda per qualche minuto, così non te la richiede a ogni comando.
chi può usare sudo
solo gli utenti autorizzati (di solito il primo utente creato, membro del gruppo sudo). Non chiunque può fare sudo: è un privilegio concesso.
quando serve
installare/rimuovere software, modificare file di sistema (in /etc), gestire servizi e utenti, usare chown (Cap. 6). Regola: se un comando tocca «il sistema» e non i tuoi file, serve sudo.
⚠ sudo è potere: pensa prima di premere Invio

Con sudo i «freni» dei permessi sono disattivati per quel comando: un errore può danneggiare il sistema. Il famigerato sudo rm -rf / cancellerebbe l'intero sistema (i moderni Linux lo bloccano, ma il concetto resta). Regola d'oro: usa sudo solo quando sai perché ti serve, e rileggi il comando prima di eseguirlo. Non prendere l'abitudine di «mettere sudo davanti a tutto» per far sparire gli errori: se un comando fallisce senza sudo, chiediti prima se doveva davvero girare da amministratore.

Sezione 7.3Gestire utenti e gruppi


  

    Su un sistema con più persone (o un server), si creano e gestiscono utenti e gruppi. Sono operazioni da
    amministratore (quindi con sudo); vediamo le principali, senza entrare troppo nel dettaglio.
  


  

    
terminale — gestione utenti (con sudo)

    
# crea un nuovo utente (e la sua home)
$ sudo adduser luca

# aggiungi un utente a un gruppo (es. sudo, per dargli poteri di admin)
$ sudo usermod -aG sudo luca

# chi sono io e a quali gruppi appartengo?
$ whoami
anna
$ groups
anna sudo sviluppo

  


  

    
I comandi principali adduser · usermod · groups

    

      

        
sudo adduser nome

        
crea un nuovo utente con la sua cartella home e chiede di impostare una password. Il modo standard
            di aggiungere una persona al sistema.

        
sudo usermod -aG gruppo utente

        
aggiunge un utente a un gruppo (-aG = «append to Group»). Aggiungere qualcuno al
            gruppo sudo gli dà i poteri di amministratore. I gruppi servono a condividere permessi
            (Cap. 6).

        
whoami · groups

        
whoami dice chi sei; groups elenca i gruppi a cui appartieni. Utile per
            capire cosa puoi fare: se sei nel gruppo sudo, puoi usare sudo.

      

    

    

      Non serve memorizzare tutto: l'importante è capire il modello — ci sono utenti, ogni utente sta in
      uno o più gruppi, e i gruppi (insieme ai permessi del Cap. 6) determinano chi accede a cosa. Su un server, ad
      esempio, potresti creare un utente apposito per far girare un'applicazione (senza poteri di admin), seguendo il
      principio del minimo privilegio (§7.4). Sono operazioni che ritroverai amministrando sistemi reali.
    

  



  
  
Sezione 7.4Il principio del minimo privilegio


  

    C'è un'idea di sicurezza che attraversa tutto questo capitolo e vale la pena esplicitare, perché guida ogni buona
    pratica su Linux (e non solo): il principio del minimo privilegio.
  


  

    
🛡 Solo i poteri necessari, solo quando servono

    

      Il principio dice: ogni utente (e ogni programma) dovrebbe avere solo i permessi strettamente
      necessari per il suo compito, e nulla di più. Da qui discendono tutte le pratiche viste: non lavorare da
      root (usi sudo solo per i singoli comandi che lo richiedono); dare a un'applicazione un utente
      dedicato con permessi limitati; concedere i poteri di admin solo a chi serve. Perché? Perché meno potere è
      in circolazione, meno danni può fare un errore o un attacco. Se un programma con permessi minimi viene bucato, il
      danno è contenuto; se aveva i poteri di root, il disastro è totale. È lo stesso principio della sicurezza
      del libro «Backend» (non fidarsi, limitare gli accessi): meno privilegi = meno superficie di rischio.
    

  


  

    
💜 Da ricordare: la disciplina del sudo

    

      In pratica, il minimo privilegio si traduce in un'abitudine semplice: lavora sempre come utente normale, e
      usa sudo solo per i comandi che davvero lo richiedono, dopo aver capito perché. Resisti alla
      tentazione di «diventare root e restarci» per comodità: è comodo finché non fai l'errore che cancella tutto. La
      protezione dei permessi (Cap. 6) e la disciplina del sudo sono i tuoi angeli custodi — non disattivarli per
      pigrizia.
    

  



  
  
Sezione 7.5Laboratorio: usare sudo


  

    Vediamo sudo all'opera e la differenza tra ciò che puoi fare come utente normale e ciò che richiede i
    poteri di admin.
  


  

    
terminale — utente normale vs sudo

    
# chi sei e a quali gruppi appartieni
$ whoami
anna
$ groups          # se vedi "sudo", puoi usare sudo

# operazione sui TUOI file: nessun sudo necessario
$ touch ~/prova.txt          # funziona: è casa tua

# operazione sul SISTEMA: senza sudo fallisce...
$ touch /etc/prova.txt
touch: cannot touch '/etc/prova.txt': Permission denied

# ...con sudo funziona (ma chiediti: mi serve DAVVERO scrivere in /etc?)
$ sudo touch /etc/prova.txt
[sudo] password for anna:          # la tua password (non si vede)

  


  

    
✅ Cosa hai imparato a fare

    

      Hai visto la linea di confine: sui tuoi file (la home) agisci liberamente; per toccare il
      sistema (/etc, installazioni…) serve sudo. E hai visto che «Permission
      denied» non è un ostacolo da aggirare a caso con sudo, ma un segnale da capire: doveva quel comando
      girare da amministratore? Questa consapevolezza — usare il potere solo quando serve — è ciò che distingue chi
      amministra bene un sistema. Con la comprensione di utenti, permessi e sudo, hai le basi della «gestione» di
      Linux. Nel prossimo capitolo torniamo all'uso quotidiano con un tema affascinante: i processi, cioè i
      programmi in esecuzione, e come vederli e controllarli.

Sezione 7.7Soluzioni


  

    
Soluzione 7.1

    

      root è l'utente amministratore, che può fare qualsiasi cosa sul sistema: leggere/modificare ogni
      file, installare software, gestire utenti, cambiare configurazioni — nessun permesso lo ferma. Un utente
      normale, invece, gestisce solo i propri file e non può toccare il sistema. Dal prompt lo capisci dal
      simbolo finale: $ = utente normale, # = root (pieni poteri).
    

  


  

    
Soluzione 7.2

    

      sudo esegue un singolo comando con i poteri di root. Ti chiede la tua password (non
      quella di root), per confermare la tua identità. I poteri sono temporanei: valgono solo per quel comando,
      dopodiché torni a essere un utente normale (sudo ricorda la password per qualche minuto, per non richiederla a
      ogni comando ravvicinato).
    

  


  

    
Soluzione 7.3

    

      (a) creare un file nella tua home: no sudo (sono i tuoi file). (b) installare con apt: sì
      sudo (modifica il sistema per tutti). (c) modificare un file in /etc: sì sudo (file di
      sistema). (d) elencare i tuoi documenti: no sudo (sono tuoi). Regola: serve sudo quando l'operazione tocca
      «il sistema» o file di altri, non i tuoi.
    

  


  

    
Soluzione 7.4

    

      Il principio del minimo privilegio dice che ogni utente e programma dovrebbe avere solo i permessi
      strettamente necessari al proprio compito, e nulla di più. È meglio usare sudo solo per i singoli
      comandi che lo richiedono, invece di lavorare sempre come root, perché così si limitano i danni: la
      maggior parte del tempo operi senza poteri speciali (quindi un errore è circoscritto), e attivi il «potere» solo
      per l'istante necessario, con piena consapevolezza. Meno privilegi in circolazione = meno danni possibili da
      errori o attacchi.
