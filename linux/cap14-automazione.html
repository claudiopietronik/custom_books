Capitolo 14

Automazione e ambiente

Sai scrivere script che ragionano e ripetono. Manca l'ultimo tocco per renderli davvero comodi e autonomi: integrarli nel tuo ambiente e — la parte più magica — farli girare da soli, a orari prestabiliti, senza che tu li debba lanciare. In questo capitolo vediamo come «personalizzare» il tuo terminale: le variabili d'ambiente (impostazioni condivise, come il PATH che dice dove trovare i comandi), gli alias (scorciatoie per comandi lunghi) e il file .bashrc che li rende permanenti. E poi il pezzo forte: cron, lo schedulatore di Linux, che esegue i tuoi script automaticamente — ogni notte, ogni ora, ogni lunedì — mentre tu dormi. È il coronamento dell'automazione: non solo «il computer fa il lavoro al posto tuo», ma «lo fa quando deve, senza che tu ci pensi». In questo capitolo: variabili d'ambiente e PATH (§14.1), alias e .bashrc (§14.2), pianificare con cron (§14.3), e rendere gli script comodi (§14.4). È dove Linux inizia davvero a lavorare per te.

Variabili d'ambiente e PATH (§14.1), alias e .bashrc (§14.2), pianificare con cron (§14.3), rendere gli script comodi (§14.4). Nel laboratorio (§14.5) automatizzi un compito.

🔑 Perché ci tocca: dal fare al far fare

Un backup che parte ogni notte, un rapporto generato ogni lunedì, una pulizia periodica: l'automazione schedulata è ciò che fa lavorare il sistema mentre tu fai altro. È il cuore dell'amministrazione dei server (dove cron fa girare manutenzioni e controlli in continuazione) e una comodità enorme anche sul PC. E gli alias/`.bashrc` rendono il terminale «tuo», adattato al tuo modo di lavorare.

Sezione 14.1Variabili d'ambiente e PATH

Le variabili d'ambiente sono variabili «globali» del sistema, disponibili a tutti i programmi. Contengono impostazioni importanti — la più famosa è il PATH, che dice al sistema dove cercare i comandi.

terminale — le variabili d'ambiente
# mostra tutte le variabili d'ambiente
$ env

# mostra una variabile specifica (es. la tua home, o il PATH)
$ echo $HOME
/home/anna
$ echo $PATH
/usr/local/bin:/usr/bin:/bin:...

# imposta una variabile d'ambiente (per questa sessione)
$ export NOME_APP="MioProgetto"
Variabili d'ambiente e PATH env · $HOME · $PATH · export
variabili come $HOME, $USER
impostazioni utili disponibili ovunque: $HOME (la tua cartella), $USER (il tuo nome). I programmi le leggono per adattarsi a te. Nei libri «Backend» hai visto i «segreti» messi in variabili d'ambiente: stesso concetto.
il PATH: dove trovare i comandi
una lista di cartelle (separate da :) dove il sistema cerca i comandi che digiti. Quando scrivi ls, il sistema lo cerca in queste cartelle e lo trova (in /bin). Se un comando «non si trova», spesso è perché non è nel PATH.
export VAR=valore
imposta una variabile d'ambiente. Vale solo per la sessione corrente (e i programmi che avvii da lì); per renderla permanente, va messa nel .bashrc (§14.2).
Il PATH spiega un mistero: perché puoi digitare ls e funziona, ma per uno script tuo devi scrivere ./script.sh? Perché ls è in una cartella del PATH, il tuo script no (la cartella corrente non è nel PATH, per sicurezza — Cap. 12). Se aggiungessi la cartella dei tuoi script al PATH, potresti lanciarli col solo nome. Le variabili d'ambiente sono il «pannello di controllo» condiviso del tuo ambiente Linux.

Sezione 14.2Alias e .bashrc

Un alias è una scorciatoia: un nome breve per un comando lungo che usi spesso. E il file .bashrc è dove metti alias e impostazioni perché siano permanenti, caricati a ogni apertura del terminale.

terminale — alias
# crea un alias: "ll" al posto del più lungo "ls -la"
$ alias ll="ls -la"

# ora "ll" fa "ls -la"
$ ll
... elenco dettagliato ...

# un altro esempio comodo
$ alias ..="cd .."          # ".." per salire di un livello
Alias e il file .bashrc scorciatoie permanenti
alias nome="comando"
crea una scorciatoia: da lì in poi, digitando nome esegui quel comando. Comodo per comandi lunghi che ripeti (opzioni complesse, percorsi lunghi).
il file ~/.bashrc
un file nascosto nella tua home (nota il punto: file nascosto, Cap. 3) che bash esegue a ogni apertura del terminale. Ciò che metti qui è permanente: alias, variabili (export), personalizzazioni del prompt.
rendere permanente un alias
un alias creato al volo sparisce chiudendo il terminale. Per tenerlo, aggiungi la riga alias ll="ls -la" al .bashrc (con nano, Cap. 10), poi source ~/.bashrc (o riapri il terminale) per applicarlo.
Il .bashrc è dove il terminale diventa «tuo»: ci accumuli le scorciatoie e le impostazioni che rendono il tuo flusso di lavoro più veloce. Molti sviluppatori curano il proprio .bashrc con decine di alias personali. È anche il posto dove aggiungere una cartella al PATH (export PATH="$HOME/scripts:$PATH") per lanciare i tuoi script col solo nome. Piccole comodità che, moltiplicate per anni d'uso, fanno una grande differenza.

Sezione 14.3Pianificare con cron

Ecco il pezzo forte: cron è il sistema che esegue comandi e script automaticamente, a orari prestabiliti. Definisci «cosa» eseguire e «quando», e cron ci pensa — anche se il terminale è chiuso, anche mentre dormi.

terminale — programmare un task con cron
# apri la tua tabella di cron (crontab) per modificarla
$ crontab -e

# dentro, aggiungi righe come questa (5 campi di tempo + il comando):
# ┌── minuto (0-59)
# │ ┌── ora (0-23)
# │ │ ┌── giorno del mese (1-31)
# │ │ │ ┌── mese (1-12)
# │ │ │ │ ┌── giorno settimana (0-6, 0=domenica)
# │ │ │ │ │
  0 3 * * *   /home/anna/backup.sh   # ogni giorno alle 3:00
La sintassi di cron min ora giorno mese settimana comando
i 5 campi di tempo
nell'ordine: minuto, ora, giorno del mese, mese, giorno della settimana. Poi il comando da eseguire. Insieme definiscono quando eseguire.
l'asterisco * = «ogni»
un * in un campo significa «ogni valore». 0 3 * * * = «al minuto 0 dell'ora 3, ogni giorno del mese, ogni mese, ogni giorno della settimana» = ogni giorno alle 3:00.
esempi comuni
0 * * * * = ogni ora (al minuto 0); */15 * * * * = ogni 15 minuti; 0 9 * * 1 = ogni lunedì alle 9:00; 30 2 1 * * = il 1° di ogni mese alle 2:30. Combinazioni per ogni esigenza.
crontab -e · crontab -l
-e modifica la tabella (si apre nell'editor), -l la elenca. Ogni riga è un task programmato.
cron è ciò che rende un server «vivo» anche quando nessuno lo tocca: manutenzioni, backup, controlli, rapporti — tutto girato automaticamente all'orario giusto. La sintassi dei 5 campi confonde all'inizio, ma con un po' di pratica (e strumenti online che la traducono) diventa naturale. Sezione 14.4Rendere gli script comodi

Uniamo i pezzi: come far sì che i tuoi script siano facili da lanciare, ovunque, e girino da soli quando serve. Poche accortezze che fanno la differenza tra uno script «scritto» e uno «integrato».

Da script a strumento del sistema le buone abitudini
usa percorsi assoluti negli script schedulati
quando cron esegue uno script, non è «nella tua cartella»: usa percorsi assoluti (/home/anna/dati, non dati) per i file, o lo script non li troverà. Errore classico degli script cron.
una cartella per i tuoi script
raccogli i tuoi script in una cartella (es. ~/scripts) e, aggiungendola al PATH (§14.1), li lanci col solo nome, da ovunque. Ordine e comodità.
log dei task automatici
uno script schedulato gira «invisibile»: fai in modo che scriva cosa fa in un file di log (con >>, Cap. 9), così puoi controllare se ha funzionato. Un task senza log è cieco.
alias per i comandi frequenti
metti nel .bashrc gli alias per le operazioni che ripeti: rende il lavoro quotidiano più rapido.
Il filo conduttore: uno script è utile quanto è comodo da usare e affidabile quando gira da solo. Percorsi assoluti (per cron), una cartella ordinata nel PATH, log per controllare, alias per velocizzare. Queste piccole cure trasformano uno script «di prova» in uno strumento su cui puoi contare — proprio ciò che serve per il progetto finale del prossimo capitolo: un backup automatico che gira ogni notte.

Sezione 14.5Laboratorio: automatizzare un compito

Mettiamo insieme alias, .bashrc e cron per automatizzare un piccolo compito reale.

terminale — un giro di automazione
# 1) crea un alias comodo e rendilo permanente nel .bashrc
$ echo 'alias ll="ls -la"' >> ~/.bashrc
$ source ~/.bashrc          # applica subito (senza riaprire)
$ ll                       # ora funziona!

# 2) crea uno script che salva la data in un log
$ nano ~/registra.sh
#   dentro:  #!/bin/bash
#            echo "Eseguito il $(date)" >> /home/anna/registro.log
$ chmod +x ~/registra.sh

# 3) programmalo con cron: ogni giorno alle 8:00
$ crontab -e
#   aggiungi la riga:
  0 8 * * *   /home/anna/registra.sh

# 4) verifica che il task sia registrato
$ crontab -l
✅ Cosa hai automatizzato

Hai reso un alias permanente (nel .bashrc), scritto uno script che registra un log (con percorso assoluto, come consigliato per cron), e lo hai schedulato perché giri ogni giorno alle 8:00 — da solo, senza che tu debba fare nulla. Da domani, ogni mattina, il tuo registro.log avrà una nuova riga, aggiunta dal computer al posto tuo. Questo è il senso profondo dell'automazione: definisci il lavoro una volta, e il sistema lo esegue per sempre, all'orario giusto. Con questa competenza sei pronto per il gran finale: nel prossimo capitolo, prima uno sguardo ai comandi di sistema e rete, poi il progetto completo — uno script di backup automatico che unisce tutto ciò che hai imparato.

🔑 Quello che devi portarti dietro dal Capitolo 14
  1. Variabili d'ambiente (impostazioni globali): $HOME, $PATH (dove il sistema cerca i comandi). export per impostarle (§14.1).
  2. Alias: scorciatoie (alias ll="ls -la"). Il file .bashrc le rende permanenti (viene eseguito a ogni terminale) (§14.2).
  3. cron esegue comandi a orari prestabiliti: 5 campi (min ora giorno mese settimana) + comando. * = «ogni». crontab -e per modificare (§14.3).
  4. Script comodi: percorsi assoluti (per cron), una cartella nel PATH, log per controllare, alias per velocizzare (§14.4).

I tuoi script ora girano da soli, quando devono: l'automazione è completa. Prima di chiudere con il progetto finale, uno sguardo agli strumenti per ispezionare il sistema e la rete dal terminale — quanto spazio resta sul disco, se la rete funziona, come collegarsi a un server remoto. Nel Capitolo 15 vediamo df, du, ping, ssh e cenni ai servizi con systemctl: gli strumenti diagnostici che completano la tua cassetta degli attrezzi Linux.

Sezione 14.6Esercizi

Esercizio 14.1

Cos'è il PATH? Perché puoi lanciare ls col solo nome, ma per un tuo script devi scrivere ./script.sh?

Esercizio 14.2

Crei un alias con alias gs="git status", ma dopo aver chiuso e riaperto il terminale non funziona più. Perché? Come lo rendi permanente?

Esercizio 14.3

Cosa significa l'espressione cron 0 3 * * *? E 0 9 * * 1? (Ricorda l'ordine: minuto, ora, giorno, mese, giorno della settimana.)

Esercizio 14.4

Perché in uno script eseguito da cron è importante usare percorsi assoluti per i file, invece di percorsi relativi?

Sezione 14.7Soluzioni

Soluzione 14.1

Il PATH è una variabile d'ambiente che contiene la lista di cartelle dove il sistema cerca i comandi che digiti. Puoi lanciare ls col solo nome perché ls si trova in una cartella del PATH (es. /bin), quindi il sistema lo trova. Un tuo script, invece, sta in una cartella non nel PATH (la cartella corrente non è inclusa, per sicurezza): perciò devi indicargli esplicitamente il percorso con ./script.sh («in questa cartella»).

Soluzione 14.2

Non funziona più perché un alias creato al volo con alias vale solo per la sessione corrente del terminale: chiudendolo, sparisce. Per renderlo permanente, aggiungi la riga alias gs="git status" al file ~/.bashrc (che bash esegue a ogni apertura del terminale); poi source ~/.bashrc o riapri il terminale per applicarlo.

Soluzione 14.3

0 3 * * * = «al minuto 0 dell'ora 3, ogni giorno, ogni mese, ogni giorno della settimana» = ogni giorno alle 3:00. 0 9 * * 1 = «al minuto 0 dell'ora 9, ogni giorno del mese, ogni mese, ma solo il giorno 1 della settimana (lunedì)» = ogni lunedì alle 9:00. Gli * significano «ogni», e il quinto campo (giorno settimana) usa 0=domenica … 6=sabato.

Soluzione 14.4

Perché cron esegue lo script non «dalla tua cartella»: la sua directory di lavoro non è quella che ti aspetti. Un percorso relativo (es. dati.txt) verrebbe cercato nel posto sbagliato e lo script non troverebbe il file (o lo creerebbe altrove). Con un percorso assoluto (es. /home/anna/dati.txt), invece, il file è individuato in modo univoco, indipendentemente da dove cron esegue lo script. È l'errore più comune degli script schedulati.
