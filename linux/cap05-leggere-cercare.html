Capitolo 5

Leggere e cercare nei file

Organizzare i file è metà del lavoro; l'altra metà è ritrovarci dentro le informazioni. In questo capitolo impari a guardare il contenuto di un file senza aprirlo in un editor, e — soprattutto — a cercare: trovare una riga in un file di migliaia, o trovare un file tra migliaia. Sono tra i comandi più potenti e usati di Linux. Per leggere: cat (mostra tutto), less (sfoglia file lunghi), head/tail (inizio e fine). Per cercare, due strumenti da imparare bene: grep, che trova testo dentro i file («in quale riga compare la parola errore?»), e find, che trova i file stessi nell'albero («dov'è finito quel file .pdf?»). Sono strumenti che userai per il debug, per esplorare log, per orientarti in progetti sconosciuti. Padroneggiare grep e find è uno dei salti di qualità più grandi nell'uso del terminale.

🗺 Dove siamo

Vedere il contenuto: cat, less (§5.1), inizio e fine: head, tail (§5.2), cercare testo: grep (§5.3), trovare file: find (§5.4). Nel laboratorio (§5.5) cerchi in file reali.

🔑 Perché ci tocca: trovare l'ago nel pagliaio

I sistemi reali sono pieni di file e di log enormi. Saper cercare — l'errore in un log di 100.000 righe, il file perso in mille cartelle — è ciò che distingue chi «si arrangia» da chi lavora con efficienza. grep in particolare è talmente utile che il suo nome è diventato un verbo tra gli informatici («greppare» un file). Vale ogni minuto imparato.

Sezione 5.1Vedere il contenuto: cat, less

Per guardare cosa c'è dentro un file di testo, senza aprire un editor: cat per i file corti, less per quelli lunghi.

terminale — vedere il contenuto
# cat: stampa TUTTO il file (ottimo per file corti)
$ cat note.txt
Comprare il latte
Chiamare Luca
Finire il progetto

# less: sfoglia file lunghi una schermata alla volta
$ less /var/log/syslog
# dentro less: frecce per scorrere, / per cercare, q per uscire
cat e less tutto in una volta vs sfogliabile
cat file
«concatenate»: stampa l'intero contenuto del file sullo schermo, tutto in una volta. Perfetto per file corti; per file lunghi «scorre via» troppo in fretta.
less file
mostra il file una schermata alla volta, sfogliabile: frecce/PagGiù per scorrere, /parola per cercare dentro, q per uscire. Il modo giusto per file grandi (log, configurazioni). Non carica tutto in memoria: veloce anche su file enormi.
bonus: cat per unire
il nome «concatenate» rivela l'uso originale: cat a.txt b.txt stampa i due file uno dopo l'altro (li «concatena»). Con la redirezione (Cap. 9) puoi unirli in un file solo.
Regola pratica: cat per un'occhiata veloce a un file piccolo, less per esplorare con calma un file grande. Attenzione a non fare cat su un file enorme (o su un file binario): riempirebbe lo schermo di roba illeggibile. Se non sai quanto è grande, parti con less — è sempre sicuro.

Sezione 5.2Inizio e fine: head, tail

Spesso non serve tutto il file: bastano le prime righe (per capire com'è fatto) o le ultime (per vedere le cose più recenti, es. in un log). head e tail fanno esattamente questo.

terminale — inizio e fine
# head: le prime 10 righe (di default)
$ head dati.csv

# head -n 3: solo le prime 3 righe
$ head -n 3 dati.csv

# tail: le ultime righe (utile per i log: le cose più recenti stanno in fondo)
$ tail /var/log/syslog

# tail -f: "segui" un file mentre cresce (log in tempo reale!)
$ tail -f /var/log/syslog
head, tail e il potente -f prime · ultime · in diretta
head file
mostra le prime 10 righe (con -n N, le prime N). Utile per «sbirciare» com'è fatto un file senza aprirlo tutto.
tail file
mostra le ultime 10 righe (con -n N, le ultime N). Perfetto per i log: le righe più recenti sono in fondo, quindi tail te le mostra subito.
tail -f file — «follow»
il gioiello: segue il file in tempo reale, mostrando le nuove righe man mano che vengono aggiunte. Indispensabile per guardare un log «dal vivo» mentre un programma gira (es. il server FastAPI del libro «Backend»). Si ferma con Ctrl+C.
tail -f è uno di quei comandi che, una volta scoperti, usi in continuazione: apri un terminale, lanci tail -f sul log della tua app, e guardi cosa succede in diretta mentre la usi. È il modo standard di «osservare» un sistema in funzione — l'hai visto nominare nei libri «Reti» e «Backend». Un classico strumento di debug.

Sezione 5.3Cercare testo: grep

Ecco lo strumento re: grep cerca un testo dentro uno o più file e stampa le righe che lo contengono. È il modo per trovare un ago in un pagliaio di righe.

terminale — grep in azione
# trova le righe che contengono "errore" nel file log.txt
$ grep errore log.txt
[10:32] errore: connessione fallita
[10:45] errore: timeout del database

# -i: ignora maiuscole/minuscole (trova anche ERRORE, Errore)
$ grep -i errore log.txt

# -r: cerca in TUTTI i file di una cartella (ricorsivo)
$ grep -r "password" progetto/

# -n: mostra anche il NUMERO di riga
$ grep -n TODO app.py
grep smontato grep [opzioni] cosa dove
grep parola file
stampa solo le righe del file che contengono «parola». Il resto viene ignorato: filtra migliaia di righe alle poche che ti interessano.
-i — ignora maiuscole/minuscole
trova «errore», «Errore», «ERRORE» indifferentemente. Utile quando non sei sicuro di come è scritto.
-r — ricorsivo
cerca in tutti i file di una cartella (e sottocartelle). Potentissimo: «in quale file del progetto compare questa funzione?» → grep -r nomeFunzione .
-n — numero di riga
mostra a quale riga si trova ogni risultato. Comodo per aprire poi il file esattamente lì.
grep è ancora più potente combinato con la pipe (Cap. 9): puoi filtrare l'output di qualsiasi comando, non solo i file. Le opzioni si combinano: grep -rin "errore" logs/ = «cerca errore, ignorando le maiuscole (-i), in tutti i file di logs/ (-r), mostrando i numeri di riga (-n)». Prova grep nel widget qui sotto: cerca una parola e guarda comparire solo le righe che la contengono.
grep dal vivo: cerca in un file di log solo le righe che combaciano
$ grep log.txt
risultato di grep

      

    

    

  



  
  
Sezione 5.4Trovare file: find


  

    grep cerca dentro i file; find cerca i file stessi nell'albero delle
    directory. «Dov'è finito quel file? Quali file .pdf ci sono in questa cartella e nelle sottocartelle?» —
    find risponde.
  


  

    
terminale — find in azione

    
# trova tutti i file .txt a partire dalla cartella corrente (.)
$ find . -name "*.txt"
./note.txt
./documenti/lettera.txt
./backup/vecchie/appunti.txt

# trova solo le cartelle di nome "backup"
$ find . -type d -name backup

# trova i file modificati negli ultimi 7 giorni
$ find . -mtime -7

  


  

    
find smontato find dove criteri

    

      

        
find dove

        
il primo argomento è da dove partire a cercare (e in tutte le sottocartelle): . = qui,
            /home = da lì, ecc. find scende in profondità in tutto l'albero indicato.

        
-name "schema"

        
cerca per nome, con i jolly (tra virgolette): -name "*.pdf" = tutti i PDF. Il criterio
            più comune.

        
-type

        
filtra per tipo: -type f (solo file), -type d (solo directory/cartelle). Utile
            per cercare specificamente file o cartelle.

        
altri criteri

        
-mtime -7 (modificati negli ultimi 7 giorni), -size +100M (più grandi di
            100 MB)… find combina criteri per ricerche precise. Utile, ad esempio, per trovare i file
            grandi che riempiono il disco.

      

    

    

      grep vs find, per non confonderli: grep cerca testo dentro i file;
      find cerca i file per nome/tipo/caratteristiche. «Cerca la parola errore» →
      grep. «Cerca il file errore.log» → find. Spesso si usano insieme: find
      per individuare i file, poi grep per cercarci dentro (o viceversa, con la pipe). Sono i due pilastri
      della ricerca in Linux.
    

  



  
  
Sezione 5.5Laboratorio: cercare nei file


  

    Mettiamo alla prova cat, grep e find su file reali del sistema — un buon modo per vedere quanto sono utili.
  


  

    
terminale — un giro di ricerca

    
# crea un piccolo file di prova e guardalo
$ printf "riga uno\nerrore qui\nriga tre\nErrore grave\n" > prova.log
$ cat prova.log

# cerca "errore" (solo minuscolo) e poi ignorando le maiuscole
$ grep errore prova.log
errore qui
$ grep -i errore prova.log
errore qui
Errore grave          # -i trova anche "Errore"

# trova tutti i file .log a partire dalla cartella corrente
$ find . -name "*.log"

# solo le ultime righe di un log di sistema
$ tail -n 5 /var/log/syslog   # (se disponibile sul tuo sistema)

  


  

    
✅ Cosa hai imparato a fare

    

      Hai letto file (cat, tail), cercato testo dentro di essi
      (grep, con e senza -i) e trovato file nell'albero (find).
      Nota la differenza fondamentale: grep -i ha trovato sia «errore» sia «Errore»,
      mostrando il valore dell'opzione -i; e find ha cercato i file, non il loro contenuto.
      Questi comandi sono i tuoi occhi nel sistema: per capire cosa contiene un file, per trovare un errore in un log,
      per orientarti in un progetto che non conosci. Nel prossimo capitolo affrontiamo un tema diverso ma
      fondamentale: i permessi — chi può leggere, scrivere ed eseguire ogni file, e come si controllano.

Sezione 5.7Soluzioni


  

    
Soluzione 5.1

    

      cat per file corti (li stampa tutti in una volta, occhiata veloce); less per
      file lunghi (li sfoglia una schermata alla volta, con ricerca interna). Facendo cat su un
      file enorme, riempi lo schermo di migliaia di righe che «scorrono via» illeggibili (e se è un file binario,
      caratteri incomprensibili): meglio less, che è sempre sicuro e gestibile.
    

  


  

    
Soluzione 5.2

    

      tail -f «segue» un file in tempo reale: mostra le nuove righe man mano che vengono aggiunte.
      È particolarmente utile per osservare un log dal vivo: ad esempio, lanci tail -f sul log del
      tuo server web e vedi comparire le richieste (o gli errori) mentre usi l'applicazione — uno strumento di debug
      fondamentale. Si interrompe con Ctrl+C.
    

  


  

    
Soluzione 5.3

    

      
terminale

      
$ grep -in warning server.log

    

    
Le opzioni si combinano: -i (ignora maiuscole) + -n (numero
      di riga) → -in. Trova «warning», «Warning», «WARNING» e mostra a quale riga.

  


  

    
Soluzione 5.4

    

      grep cerca testo dentro i file (le righe che contengono qualcosa); find cerca
      i file stessi per nome/tipo/caratteristiche nell'albero. Quindi: (a) «trova le righe con la parola bug» →
      grep (grep bug file); (b) «trova i file chiamati config.txt» →
      find (find . -name config.txt).
