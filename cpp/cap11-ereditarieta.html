Capitolo 11

Ereditarietà e polimorfismo

Spesso più tipi hanno molto in comune: un Cane e un Gatto sono entrambi Animale; un Cerchio e un Quadrato sono entrambi Forma. L'ereditarietà permette a una classe di basarsi su un'altra, riusandone dati e metodi. Ma la vera magia è il polimorfismo: la capacità di trattare oggetti diversi attraverso un'interfaccia comune, lasciando che ognuno risponda «a modo suo». Chiami disegna() su una Forma, e se è un cerchio disegna un cerchio, se è un quadrato un quadrato — senza che il tuo codice debba sapere quale forma sia. In C++ questo passa dai metodi virtuali e ha un legame profondo con i puntatori che hai imparato. È una delle idee più potenti (e belle) della programmazione a oggetti.

🗺 Dove siamo

Ereditare da una classe base (§11.1), i metodi virtuali (§11.2), il polimorfismo (§11.3), le classi astratte (§11.4). Nel laboratorio (§11.5) costruisci una gerarchia di forme.

🔑 Perché ci tocca: codice che si estende senza riscriversi

Ereditarietà e polimorfismo sono ciò che rende il software estensibile: puoi aggiungere un nuovo tipo (una nuova forma, un nuovo tipo di utente) senza toccare il codice che li usa. È il fondamento di framework, motori di gioco, interfacce grafiche — ovunque serva trattare «cose diverse in modo uniforme». Il polimorfismo, in particolare, è un salto concettuale importante: una volta capito, cambia il modo in cui progetti i programmi. È il capitolo che completa la tua conoscenza dell'OOP in C++.

Sezione 11.1Ereditare da una classe

Una classe derivata (figlia) può ereditare da una classe base (genitore), ottenendone dati e metodi «gratis», e aggiungendo o modificando ciò che serve.

Ereditarietà class Figlia : public Base
esempio.cpp
class Animale {
public:
    std::string nome;
    void dormi() { std::cout << nome << " dorme\n"; }
};

class Cane : public Animale {   // Cane EREDITA da Animale
public:
    void abbaia() { std::cout << "Bau!\n"; }   // metodo in più
};

Cane c;
c.nome = "Fido";   // campo EREDITATO da Animale
c.dormi();           // metodo ereditato → "Fido dorme"
c.abbaia();          // metodo proprio → "Bau!"
class Cane : public Animale
i due punti : seguiti da public Animale dicono «Cane eredita da Animale». Cane ottiene tutti i membri pubblici di Animale (campo nome, metodo dormi) senza riscriverli.
aggiungere
Cane aggiunge ciò che gli è proprio (abbaia). Un oggetto Cane è anche un Animale: ha tutto ciò che ha un Animale, più il suo.
la relazione «è-un»
l'ereditarietà modella un «è-un»: un Cane è un Animale, un Cerchio è una Forma. Se non regge («un motore è-un auto»? no, ne fa parte), probabilmente non è ereditarietà che ti serve.

Sezione 11.2Metodi virtuali

La classe figlia può ridefinire un metodo del genitore (override). Ma perché la ridefinizione «funzioni» anche attraverso un puntatore alla classe base, il metodo dev'essere dichiarato virtual. È il meccanismo che abilita il polimorfismo.

Metodi virtuali virtual · override
esempio.cpp
class Animale {
public:
    virtual void verso() {          // VIRTUAL: può essere ridefinito polimorficamente
        std::cout << "...\n";
    }
};

class Cane : public Animale {
public:
    void verso() override {           // override: ridefinisce verso()
        std::cout << "Bau!\n";
    }
};
virtual (nella base)
marca un metodo come «ridefinibile in modo polimorfico»: dice al C++ «se qualcuno chiama questo metodo, usa la versione dell'oggetto reale, non quella scritta nel tipo del puntatore». Senza virtual, il polimorfismo NON funziona (§11.3) — è la parola chiave cruciale.
override (nella figlia)
parola opzionale ma fortemente consigliata che segna «questo ridefinisce un metodo virtuale della base». Il compilatore controlla che sia davvero così: se sbagli firma (un typo, un tipo diverso), ti avvisa invece di creare silenziosamente un metodo nuovo. Ti salva da bug sottili.
il distruttore virtuale
regola pratica: se una classe ha metodi virtuali (è pensata per l'ereditarietà), dovrebbe avere anche il distruttore virtual (virtual ~Animale() = default;), per una distruzione corretta attraverso puntatori base. Un dettaglio importante da ricordare.

Sezione 11.3Il polimorfismo

Ecco il cuore. Con i metodi virtuali, puoi tenere oggetti di tipi diversi tramite un puntatore (o riferimento) alla classe base comune, e chiamare un metodo: ognuno risponde a modo suo, automaticamente. Il tuo codice non deve sapere quale tipo concreto sia.

Il polimorfismo in azione un'interfaccia, tanti comportamenti
esempio.cpp
#include <vector>

// un vector di PUNTATORI ad Animale, che contiene Cani e Gatti insieme:
std::vector<Animale*> zoo;
zoo.push_back(new Cane());
zoo.push_back(new Gatto());

for (Animale* a : zoo) {
    a->verso();     // ognuno fa il SUO verso: "Bau!", poi "Miao!"
}
Animale* a — un puntatore alla base
il puntatore è di tipo Animale*, ma può puntare a un Cane o un Gatto (perché «sono» Animali). Il polimorfismo in C++ funziona tramite puntatori (o riferimenti) alla base — ecco perché il Cap. 7 era fondamentale.
a->verso()
la freccia -> è «dereferenzia e chiama il metodo» (equivale a (*a).verso()). Grazie a virtual, viene eseguito il verso() dell'oggetto reale (Cane o Gatto), non quello generico di Animale. Questa è la magia: stesso codice, comportamenti diversi.
perché è potente
il ciclo non sa né gli importa quali animali ci sono: tratta tutti come Animale e ognuno risponde correttamente. Vuoi aggiungere un Uccello? Lo scrivi, e il ciclo funziona senza modifiche. Codice aperto all'estensione, chiuso alle modifiche: un principio d'oro del design.
Nota: qui abbiamo usato new per creare oggetti (memoria dinamica, Cap. 12), necessario per tenere tipi diversi in un vector di puntatori base. Nel C++ moderno si usano gli smart pointer (std::vector<std::unique_ptr<Animale>>) per evitare i rischi del new/ delete — lo vedremo al Capitolo 12. Il concetto di polimorfismo, però, è questo. Prova il widget.

Sezione 11.4Classi astratte

A volte una classe base è solo un concetto: «Forma» in generale non ha un'area — solo le forme concrete ce l'hanno. Una classe astratta definisce l'interfaccia comune senza poter essere istanziata da sola.

Metodi virtuali puri e classi astratte = 0
esempio.cpp
class Forma {
public:
    virtual double area() = 0;   // "= 0": metodo virtuale PURO (senza corpo)
    virtual ~Forma() = default;    // distruttore virtuale (buona pratica)
};

class Cerchio : public Forma {
    double raggio;
public:
    Cerchio(double r) : raggio(r) {}   // init con lista (comune in C++)
    double area() override { return 3.14159 * raggio * raggio; }
};

// Forma f;   →  ERRORE! Forma è astratta, non si può creare "una Forma generica"
Cerchio c(5);   // OK: Cerchio è concreto e fornisce area()
virtual double area() = 0;
il = 0 rende il metodo virtuale puro: dichiara che esiste ma non ha corpo nella base. È un obbligo: «ogni Forma deve fornire un'area, ma io (Forma) non so quale».
classe astratta
una classe con almeno un metodo virtuale puro è astratta: non puoi crearne un'istanza (non ha senso «una Forma generica»). Serve come interfaccia/contratto: definisce cosa devono saper fare tutte le forme.
le figlie devono implementarlo
ogni classe concreta (Cerchio, Quadrato) deve fornire il proprio area(), altrimenti resta astratta anch'essa. Così il compilatore garantisce che ogni forma concreta abbia un'area. È un modo pulito per definire interfacce.
Piccola sintassi nuova: Cerchio(double r) : raggio(r) {} — la lista di inizializzazione del costruttore (dopo i due punti) inizializza i campi direttamente. È lo stile idiomatico e preferito in C++ (più efficiente dell'assegnare nel corpo). Leggilo come «costruisci raggio col valore r».

Sezione 11.5Laboratorio: una gerarchia di forme

Mettiamo tutto insieme nell'esempio classico del polimorfismo: una Forma astratta e forme concrete, trattate in modo uniforme in un ciclo. È il «hello world» dell'OOP polimorfica.

Il programma forme.cpp interfaccia comune, aree diverse
forme.cpp
#include <iostream>
#include <vector>

class Forma {
public:
    virtual double area() = 0;
    virtual ~Forma() = default;
};

class Quadrato : public Forma {
    double lato;
public:
    Quadrato(double l) : lato(l) {}
    double area() override { return lato * lato; }
};

class Cerchio : public Forma {
    double r;
public:
    Cerchio(double raggio) : r(raggio) {}
    double area() override { return 3.14159 * r * r; }
};

int main() {
    std::vector<Forma*> forme = { new Quadrato(4), new Cerchio(3) };
    for (Forma* f : forme) {
        std::cout << "Area: " << f->area() << "\n";   // ognuna calcola la SUA
    }
    for (Forma* f : forme) delete f;   // libera la memoria (Cap. 12)
    return 0;
}
Output: «Area: 16» (quadrato 4×4), «Area: 28.27...» (cerchio r=3). Il ciclo tratta tutto come Forma* e chiama area(): ognuna risponde con la propria formula, grazie a virtual. Aggiungi un Triangolo con il suo area() e mettilo nel vector: il ciclo lo gestisce senza cambiare una riga. Questa è l'estensibilità del polimorfismo. (Il delete finale libera la memoria allocata con new — il tema del prossimo capitolo, dove imparerai a farlo in automatico e in sicurezza.)
🔑 Quello che devi portarti dietro dal Capitolo 11
  1. L'ereditarietà (class Figlia : public Base) riusa dati e metodi della base e modella un «è-un» (§11.1).
  2. I metodi virtuali (virtual nella base, override nella figlia) permettono la ridefinizione polimorfica; senza virtual il polimorfismo non funziona (§11.2).
  3. Il polimorfismo: tramite un puntatore/riferimento alla base, chiami un metodo e ognuno risponde a modo suo — codice estensibile senza modifiche (§11.3).
  4. Una classe astratta (con virtual … = 0;) definisce un'interfaccia e non si istanzia: obbliga le figlie a implementare i metodi (§11.4).

In questo capitolo è tornato più volte un tema che avevamo accantonato: new, delete, la memoria dinamica. Per il polimorfismo abbiamo dovuto allocare oggetti a mano e liberarli. Ma gestire la memoria manualmente è la fonte numero uno di bug del C++ (fughe di memoria, doppie liberazioni). Nel Capitolo 12 affrontiamo la memoria di petto: stack e heap, new e delete, le fughe di memoria — e poi l'idea che rende il C++ moderno sicuro ed elegante: RAII e gli smart pointer, oggetti che gestiscono la memoria per te, automaticamente. È il capitolo che unisce tutto ciò che sai su puntatori, costruttori e distruttori.

Sezione 11.7Soluzioni

Soluzione 11.1

Significa che Cane si basa su Animale, ottenendone «gratis» tutti i membri pubblici (campi e metodi, come nome e dormi) senza riscriverli, e può aggiungere i propri. La relazione è sensata se «un Cane è un Animale» è vero: se invece è un «fa parte di» o «ha un», l'ereditarietà è la scelta sbagliata.

Soluzione 11.2

Perché virtual dice al C++ di usare, alla chiamata, la versione del metodo dell'oggetto reale (Cane) e non quella del tipo del puntatore (Animale). Senza virtual, chiamando verso() su un Animale* verrebbe eseguita la versione di Animale, ignorando l'override. override nella figlia segnala (e fa controllare al compilatore) che quel metodo ridefinisce uno virtuale della base — evitando errori di firma silenziosi.

Soluzione 11.3

Una classe astratta ha almeno un metodo virtuale puro (= 0): dichiara un metodo senza fornirne il corpo. Non puoi istanziarla perché sarebbe «incompleta» — area() non ha implementazione, quindi «una Forma generica» non saprebbe calcolare la propria area. Serve come interfaccia: solo le classi concrete che implementano tutti i metodi puri possono essere istanziate.

Soluzione 11.4

Devi scrivere solo una nuova classe Triangolo : public Forma con il suo double area() override { … }, e aggiungerne un'istanza al vector. Il ciclo non va toccato: tratta ogni elemento come Forma* e chiama area(), che grazie al polimorfismo eseguirà quella del Triangolo. È l'estensibilità: nuovi tipi senza modificare il codice esistente.
