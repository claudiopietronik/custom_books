Capitolo 14

Gestione degli errori

I programmi vanno a male: un input sbagliato, un file che manca, una divisione per zero, un indice fuori limite. Un programma robusto non crolla di fronte all'imprevisto — lo gestisce. Il C++ distingue tra errori del compilatore (colti prima dell'esecuzione, il suo grande vantaggio) ed errori a runtime, e per questi ultimi offre le eccezioni: throw per «lanciare» un problema, try/ catch per «catturarlo» e reagire. Unite a RAII (Cap. 12), le eccezioni permettono al C++ di gestire gli errori in modo pulito, con le risorse sempre liberate correttamente. Ma il C++ ha anche una sua cultura sull'uso delle eccezioni, che vale la pena conoscere.

🗺 Dove siamo

Errori di compilazione vs a runtime (§14.1), le eccezioni con try/catch (§14.2), sollevare eccezioni con throw (§14.3), e quando (non) usarle (§14.4). Nel laboratorio (§14.5) rendi un input robusto.

🔑 Perché ci tocca: programmi che reggono l'imprevisto

Nel mondo reale gli imprevisti sono la norma. Un software professionale gestisce l'errore e prosegue (o fallisce con grazia), invece di crollare. In C++, saper usare le eccezioni — e sapere che, insieme a RAII, non lasciano risorse «appese» — è essenziale per scrivere codice affidabile. Ed è utile capire la posizione, un po' particolare, che il C++ ha verso le eccezioni: non sempre sono lo strumento giusto, e conoscerne i costi ti fa scegliere bene.

Sezione 14.1Errori di compilazione vs a runtime

Due famiglie di errori, in momenti diversi. Il C++, essendo compilato, ha un asso nella manica: ne cattura moltissimi prima ancora di eseguire.

📐 Prima o durante
  • Errori di compilazione: il compilatore rifiuta il codice e non produce l'eseguibile. Sintassi sbagliata, tipi incompatibili, ; mancante, funzione non dichiarata. Sono i «migliori»: li scopri subito, prima che il programma possa fare danni. È il grande vantaggio della tipizzazione statica e della compilazione (Cap. 1-2): tanti bug muoiono qui.
  • Errori a runtime: il codice compila, ma durante l'esecuzione qualcosa va storto — un file non c'è, un indice sfora, una conversione fallisce. Questi il compilatore non può prevederli (dipendono dai dati e dall'ambiente). Sono quelli che vanno gestiti col codice, con le eccezioni.
💡 Leggere gli errori del compilatore

Gli errori del compilatore C++ hanno fama di essere lunghi e criptici (specie con i template!). Il trucco: leggi la prima riga d'errore e il numero di riga — spesso il vero problema è lì, e il resto è «rumore» conseguente. Correggi il primo errore e ricompila: spesso ne spariscono a cascata anche altri. Come col traceback (Cap. 11 di Python), il messaggio è un alleato, non un nemico: dagli riga e causa, e il problema si risolve. Imparare a «dialogare» col compilatore è metà del mestiere.

Sezione 14.2Le eccezioni: try / catch

Per gli errori a runtime prevedibili, il C++ offre le eccezioni: metti il codice «rischioso» in un blocco try, e gestisci l'eventuale errore in un blocco catch.

try / catch prova · e se va male…
esempio.cpp
#include <vector>
#include <stdexcept>

std::vector<int> v = {1, 2, 3};

try {
    int x = v.at(10);      // .at() LANCIA un'eccezione se fuori limite (Cap. 8)
    std::cout << x;
} catch (const std::out_of_range& e) {   // cattura QUEL tipo di errore
    std::cout << "Indice fuori limite: " << e.what();
}
// il programma continua invece di crollare
try { … }
il blocco col codice che potrebbe lanciare un'eccezione. Se tutto va bene, il catch viene saltato.
catch (const std::out_of_range& e)
se dentro il try viene lanciata un'eccezione di quel tipo, l'esecuzione salta qui: il blocco catch la gestisce, e il programma continua invece di crollare. e.what() dà un messaggio descrittivo. Cattura il tipo specifico che ti aspetti.
più catch, o «catch-all»
puoi mettere più catch per tipi d'errore diversi. catch (...) cattura qualsiasi eccezione (usalo con parsimonia, come l'except: muto di Python — meglio essere specifici). Molte eccezioni standard derivano da std::exception.
try / catch: quale percorso prende l'esecuzione scegli l'indice e guarda il flusso

Sezione 14.3Sollevare eccezioni (throw)

Puoi lanciare tu un'eccezione quando rilevi una situazione che il tuo codice non deve accettare, con throw. È il modo di dire «qui c'è un problema che non posso gestire, se ne occupi chi mi ha chiamato».

throw segnalare un problema
esempio.cpp
#include <stdexcept>

double dividi(double a, double b) {
    if (b == 0) {
        throw std::invalid_argument("divisione per zero");   // LANCIA un'eccezione
    }
    return a / b;
}

try {
    std::cout << dividi(10, 0);   // lancia! l'esecuzione salta al catch
} catch (const std::exception& e) {
    std::cout << "Errore: " << e.what();   // "Errore: divisione per zero"
}
throw std::invalid_argument("...")
«lancia» un'eccezione: interrompe subito la funzione e «propaga» l'errore verso l'alto, finché un catch adatto non lo raccoglie (o, se nessuno lo fa, il programma termina). Il messaggio spiega il problema.
le eccezioni standard
il C++ ha tipi pronti in <stdexcept>: std::invalid_argument, std::out_of_range, std::runtime_error… tutti derivati da std::exception (quindi un catch (const std::exception&) li cattura tutti).
la magia con RAII
quando un'eccezione «risale», tutti gli oggetti locali vengono distrutti correttamente (i distruttori partono): grazie a RAII (Cap. 12), file e memoria vengono rilasciati anche in caso d'errore. È l'accoppiata che rende sicura la gestione degli errori in C++.

Sezione 14.4Quando (non) usare le eccezioni

Qui il C++ ha una cultura sua. Le eccezioni sono potenti, ma hanno un costo e non sono sempre la scelta giusta. Conoscere il dibattito ti fa usarle con criterio.

📐 Eccezioni sì, eccezioni no

Le eccezioni sono ideali per errori eccezionali e irrecuperabili sul posto: un file di configurazione mancante all'avvio, memoria esaurita, dati corrotti. Non sono adatte per il controllo di flusso «normale»: se un input sbagliato è previsto e frequente (l'utente che digita male), spesso è meglio un controllo esplicito (un if, o un valore di ritorno che segnala l'esito). Motivi: le eccezioni hanno un piccolo costo quando scattano, e certi ambiti del C++ (giochi, sistemi embedded, dove le prestazioni sono tutto) a volte le disabilitano del tutto, preferendo i codici di ritorno. Il C++ moderno offre anche alternative come std::optional (un valore che «forse c'è») per gli errori «attesi». La regola: eccezioni per l'eccezionale, controlli espliciti per il prevedibile. Non c'è una risposta unica — dipende dal contesto, ed è una scelta di design che imparerai a fare con l'esperienza.

Sezione 14.5Laboratorio: input robusto

Costruiamo un programma che chiede un numero e non crolla se l'utente scrive qualcosa di sbagliato, usando le eccezioni della conversione stringa→numero.

Il programma input.cpp try/catch attorno a stoi
input.cpp
#include <iostream>
#include <string>

int main() {
    std::string testo;
    std::cout << "Scrivi un numero: ";
    std::cin >> testo;

    try {
        int n = std::stoi(testo);      // stoi lancia se "testo" non è un numero
        std::cout << "Il doppio è " << n * 2 << "\n";
    } catch (const std::exception& e) {
        std::cout << "\"" << testo << "\" non è un numero valido!\n";
    }
    return 0;
}
Esecuzione: con «21» stampa «Il doppio è 42»; con «abc» stampa «"abc" non è un numero valido!» — senza crashare. std::stoi (string→int) lancia un'eccezione se la stringa non rappresenta un numero, e il catch la trasforma in un messaggio gentile. È lo schema robusto: metti la conversione (o l'operazione rischiosa) nel try, gestisci il fallimento nel catch. Per insistere finché l'input è valido, metteresti il tutto in un ciclo (come lo schema try-dentro-while del libro di Python). Compila con g++ input.cpp -o input.
🔑 Quello che devi portarti dietro dal Capitolo 14
  1. Errori di compilazione (colti prima, il vantaggio del C++) vs runtime (da gestire); leggi la prima riga dell'errore del compilatore (§14.1).
  2. Le eccezioni: try { … } col codice rischioso, catch (const Tipo& e) per gestirlo; e.what() dà il messaggio. Il programma continua invece di crollare (§14.2).
  3. throw lancia tu un'eccezione (tipi standard in <stdexcept>); con RAII le risorse restano liberate anche in caso d'errore (§14.3).
  4. Eccezioni per l'eccezionale/irrecuperabile; controlli espliciti (if, valori di ritorno, optional) per il prevedibile — hanno un costo e a volte si disabilitano (§14.4).

I tuoi programmi ora reggono gli imprevisti. Manca un ultimo strumento pratico prima del gran finale: la persistenza. Tutto ciò che un programma elabora svanisce alla chiusura, a meno di salvarlo su disco. Nel Capitolo 15 vediamo file e I/O in C++: il concetto di stream (lo stesso di cout/cin, esteso ai file), come scrivere su file con ofstream e leggere con ifstream, riga per riga. Scoprirai che leggere e scrivere file usa la stessa elegante idea degli stream che conosci fin dal Capitolo 1 — e con RAII, i file si chiudono da soli.

Sezione 14.7Soluzioni

Soluzione 14.1

Un errore di compilazione impedisce di creare l'eseguibile (colto prima dell'esecuzione): es. un ; mancante, un tipo incompatibile. Un errore a runtime avviene durante l'esecuzione, su codice che compila: es. accedere a v.at(10) su un vector di 3 elementi, o stoi("abc"). Il primo lo scopri subito, il secondo dipende dai dati e va gestito.

Soluzione 14.2

Il try racchiude il codice che potrebbe lanciare un'eccezione; il catch gestisce l'eventuale errore di quel tipo. Quando dentro il try viene lanciata un'eccezione, l'esecuzione salta immediatamente al catch corrispondente (saltando il resto del try), e il programma continua dopo il catch invece di crollare.

Soluzione 14.3

Perché le eccezioni sono pensate per situazioni eccezionali e hanno un piccolo costo quando scattano: usarle per un evento frequente e previsto (input sbagliato dell'utente) è considerato uso improprio. Meglio un controllo esplicito (un if, o un valore di ritorno/ std::optional che segnala l'esito) per gli errori attesi, riservando le eccezioni all'imprevisto.
