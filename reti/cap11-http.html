Capitolo 11

HTTP

Eccoci al livello che tocchi ogni secondo: il linguaggio del web. Ogni pagina che apri, ogni immagine che carica, ogni «mi piace» che clicchi è, sotto sotto, una conversazione in HTTP — un dialogo di una semplicità sorprendente: il browser fa una richiesta («dammi questa pagina»), il server dà una risposta («eccola», oppure «non c'è», «non ti è permesso», «mi sono rotto»). La cosa straordinaria è che è tutto testo leggibile: alla fine del capitolo scriverai richieste HTTP a mano e capirai davvero cosa succede quando premi Invio nella barra degli indirizzi.

🗺 Dove siamo

HTTP come protocollo richiesta-risposta (§11.1), l'anatomia di una richiesta (§11.2) e di una risposta con i codici di stato (§11.3), i metodi GET/POST/… (§11.4), header e cookie (§11.5). Nel laboratorio (§11.6) parli HTTP a mano con curl.

🔑 Perché ci tocca: è il ponte con la programmazione

HTTP è il confine dove le reti incontrano lo sviluppo web. Ogni sito, ogni app, ogni API parla HTTP: quando costruirai un backend (FastAPI), riceverai richieste HTTP e manderai risposte con questi stessi codici di stato; quando qualcosa non funziona, leggere un 404, un 500 o un 301 ti dice immediatamente di chi è la colpa e cosa è successo. Capire HTTP qui significa avere in mano il vocabolario di tutti i libri di programmazione web che verranno.

Sezione 11.1Il protocollo del web

HTTP (HyperText Transfer Protocol) è il protocollo di livello applicativo del web. Gira sopra TCP (Cap. 9): prima si apre una connessione affidabile alla porta 80 (o 443 per HTTPS), poi si dialoga in HTTP.

📐 Domanda, risposta, e poi silenzio

Il modello è elementare: il client manda una richiesta, il server manda una risposta, e la transazione è finita. Ogni richiesta è indipendente: HTTP di base è «senza stato» (stateless), cioè il server non ricorda nulla della richiesta precedente. Questa semplicità è la sua forza: rende il web scalabile a miliardi di richieste. Ma pone anche un problema — se il server non ricorda nulla, come fa a tenerti «loggato» tra una pagina e l'altra? La risposta sono i cookie (§11.5). Per ora tieni la base: HTTP = una domanda, una risposta, entrambe in testo semplice.

Sezione 11.2Anatomia di una richiesta

Vediamo com'è fatta, letteralmente, una richiesta HTTP. È testo, poche righe, con una struttura fissa che una volta capita non dimentichi più.

Una richiesta HTTP riga di richiesta + header + (corpo)
GET di una pagina
GET /wiki/Pagina_principale HTTP/1.1
Host: it.wikipedia.org
User-Agent: Mozilla/5.0 (Windows…)
Accept: text/html
(riga vuota = fine degli header; una GET non ha corpo)
la riga di richiesta (la prima)
metodo (GET), percorso della risorsa (/wiki/…), versione (HTTP/1.1). È il cuore: «voglio fare GET su questa risorsa».
gli header (le righe successive)
coppie «Nome: valore» che danno contesto. Host dice quale sito (un server può ospitarne molti!); User-Agent dice che browser sei; Accept che formati gradisci. Sono le «istruzioni a margine» della richiesta.
il corpo (opzionale)
i dati che spedisci al server (assente in una GET; presente in una POST, quando invii un modulo — §11.4).

Sezione 11.3Anatomia di una risposta (status code)

La risposta ha la stessa struttura, ma inizia con la cosa più importante di tutto HTTP: il codice di stato, tre cifre che riassumono l'esito. Impararne le famiglie ti fa capire al volo qualsiasi problema web.

Richiesta → risposta: esploratore dei codici di stato scegli uno scenario e leggi la risposta del server
Le cinque famiglie di codici la prima cifra dice tutto
2xx — successo
200 OK: tutto bene, ecco quel che hai chiesto. La risposta «normale».
3xx — redirezione
301/302: «la risorsa è altrove, vai qui». Il browser segue automaticamente. Usato quando un sito si sposta o forza HTTPS.
4xx — errore del client
hai sbagliato tu: 404 (non esiste), 401/403 (non autorizzato/vietato), 400 (richiesta malformata). La colpa è della richiesta.
5xx — errore del server
ha sbagliato il server: 500 (errore interno), 502/503 (server non raggiungibile/sovraccarico). La colpa è dall'altra parte.
1xx — informativi
rari, messaggi intermedi. Puoi ignorarli per ora.
Regola d'oro da ricordare per sempre: 4xx = colpa tua (o della richiesta), 5xx = colpa del server. Solo questa distinzione ti fa capire subito, davanti a un errore, da che parte guardare.

Sezione 11.4I metodi: GET, POST e gli altri

Il metodo dice che tipo di azione vuoi fare sulla risorsa. Sono pochi e con significati precisi — e sono la base di come funzionano le API che programmerai.

I metodi HTTP principali il "verbo" della richiesta
GET — leggere
chiedi una risorsa senza modificarla: una pagina, un'immagine, dei dati. È «sicuro» (non cambia nulla sul server) e ripetibile. Il 99% della navigazione è fatto di GET.
POST — creare/inviare
mandi dati al server nel corpo per creare qualcosa o eseguire un'azione: inviare un modulo, pubblicare un commento, fare login. Non è ripetibile a cuor leggero (rifare una POST può creare un doppione — ecco l'avviso «non ricaricare la pagina» dopo un pagamento).
PUT / PATCH — aggiornare
modificano una risorsa esistente (PUT la sostituisce, PATCH la corregge in parte).
DELETE — cancellare
elimina una risorsa. Il nome dice tutto.
Questi «verbi» sono le fondamenta delle API REST, lo stile con cui web e app si scambiano dati: GET /utenti/5 (leggi l'utente 5), POST /utenti (creane uno nuovo), DELETE /utenti/5 (cancellalo). Quando costruirai un backend con FastAPI, scriverai esattamente funzioni per questi metodi. HTTP è il ponte diretto verso la programmazione web.

Sezione 11.5Header e cookie

Gli header sono le tante «etichette» che accompagnano richieste e risposte. Tra questi, uno risolve il problema che avevamo lasciato aperto: come fa un protocollo «senza memoria» a tenerti loggato?

📐 Il cookie: la memoria del web

HTTP è senza stato, ma il web ha bisogno di ricordare (chi sei, cosa hai nel carrello). Il trucco: al primo accesso il server ti manda un cookie (un header Set-Cookie con un codice), e il browser lo rimanda a ogni richiesta successiva. Così il server, pur non «ricordando» nulla di suo, ti riconosce perché gli riporti tu stesso il biglietto che ti ha dato. È come il numeretto del guardaroba: il guardarobiere non ti ricorda, ma il tuo numero identifica la tua giacca. I cookie sono ciò che tiene in piedi login, carrelli e preferenze — e, nel bene e nel male, anche il tracciamento pubblicitario.

💡 Header che incontrerai spesso

Content-Type (che formato è la risposta: HTML? JSON? immagine?), Content-Length (quanto è grande), Cache-Control (per quanto conservarla — Cap. 13), Location (dove reindirizzare, nei 3xx), Authorization (le credenziali). Gli header sono il modo in cui client e server si accordano sui dettagli senza sporcare i dati veri. Li vedrai tutti, dal vivo, nel laboratorio.

Sezione 11.6Laboratorio: parlare HTTP con curl

Basta guardare: adesso parli tu HTTP, a mano, con curl — lo strumento con cui si fanno richieste HTTP dal terminale. È il modo migliore per vedere richieste e risposte nude e crude.

Il comando curl -i mostrami header e risposta
Terminale
C:\> curl -i https://example.com
curl — un browser da riga di comando
fa una richiesta HTTP e stampa la risposta. Senza fronzoli: niente rendering, solo il testo grezzo del protocollo.
-i — includi gli header
mostra anche la riga di stato e gli header della risposta, non solo il contenuto. Perfetto per vedere il codice e i metadati. (curl -I maiuscola: solo gli header.)
Output
HTTP/1.1 200 OK              ← la riga di stato: codice 200
Content-Type: text/html         ← è HTML
Content-Length: 1256
Cache-Control: max-age=604800

<!doctype html>                 ← poi il corpo: la pagina vera
<html> …
Prove da fare: curl -I https://google.com (vedrai un 301 che reindirizza a www!); curl https://api.github.com (una API che risponde in JSON, non in HTML); curl -i https://example.com/nonesiste (un bel 404). Cambia l'URL e osserva i codici di stato cambiare: stai parlando la lingua nativa del web, quella che scriverai tu stesso lato server nei prossimi libri.
🔑 Quello che devi portarti dietro dal Capitolo 11
  1. HTTP è richiesta → risposta in testo, su TCP, ed è senza stato (§11.1).
  2. Richiesta = riga (metodo + percorso + versione) + header + (corpo) (§11.2). Risposta = codice di stato + header + corpo (§11.3).
  3. Codici: 2xx ok, 3xx redirect, 4xx colpa tua, 5xx colpa del server (§11.3). Metodi: GET leggi, POST crea/invia, PUT/PATCH aggiorna, DELETE cancella — la base delle API (§11.4).
  4. I cookie danno «memoria» a un protocollo senza stato (login, carrello) (§11.5). E ora parli HTTP a mano con curl (§11.6).

Sai come il web chiede e risponde — ma tutto, finora, viaggia in chiaro: chiunque sulla rete (il WiFi del bar, il tuo operatore, un malintenzionato) potrebbe leggere le tue richieste HTTP, password comprese. È inaccettabile. Serve una busta blindata attorno a HTTP: cifratura, autenticazione, garanzia di parlare davvero col sito giusto. Nel Capitolo 12 aggiungiamo la S di sicurezza — HTTPS e il protocollo TLS — e capiamo cosa c'è davvero dietro quel lucchetto nella barra degli indirizzi.

Sezione 11.7Esercizi

Esercizio 11.1

Per ciascuno, di' se è colpa del client o del server, e cosa significa: 404, 500, 301, 200, 403.

Esercizio 11.2

Quale metodo useresti per: (a) leggere il profilo dell'utente 7; (b) creare un nuovo commento; (c) cancellare una foto; (d) aprire la home di un sito?

Esercizio 11.3

HTTP è «senza stato»: come fa allora un sito a tenerti loggato mentre navighi tra le pagine? Spiega con l'analogia del guardaroba.

Esercizio 11.4

(Pratico) Esegui curl -I https://google.com: che codice ricevi e perché? Poi curl -i https://example.com/xyz: che codice, e di chi è la «colpa»?

Sezione 11.8Soluzioni

Soluzione 11.1

404 client (risorsa non esistente); 500 server (errore interno); 301 redirezione (risorsa spostata, il browser segue); 200 successo; 403 client (vietato, non hai i permessi). Regola: 4xx = lato client, 5xx = lato server.

Soluzione 11.2

(a) GET /utenti/7 (leggere); (b) POST /commenti (creare); (c) DELETE /foto/… (cancellare); (d) GET / (leggere la home).

Soluzione 11.3

Con i cookie: dopo il login il server manda un cookie (un codice) che il browser rispedisce a ogni richiesta. Il server non «ricorda» te, ma riconosce il cookie che gli riporti — come il guardarobiere che non ricorda te ma identifica la tua giacca dal numeretto che gli mostri.

Soluzione 11.4

curl -I https://google.com dà tipicamente un 301 (Google reindirizza alla versione www / canonica: la risorsa è «altrove»). curl -i https://example.com/xyz dà un 404: la pagina non esiste, ed è un errore 4xx → lato client/richiesta (hai chiesto qualcosa che non c'è).
